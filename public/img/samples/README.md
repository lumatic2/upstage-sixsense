# 예시 메뉴판 이미지 — 생성 기록

제보 페이지(`/contribute.html`)의 "예시로 해보기" 버튼이 쓰는 이미지다.
심사위원·팀원이 **사진 없이도 파이프라인 전체를 볼 수 있어야** 해서 둔다
(`docs/SITEMAP.md` §`/contribute.html`).

## 왜 실사진을 안 쓰나

`experiments/parse-poc/photos/` 는 남의 네이버 블로그에서 모은 것이고,
`photos/sources.md` 에 **"실측 실험 전용(레포 내 보존, 재배포 아님)"** 이라고 적어 두었다.
공개 사이트에 올리면 그 기록과 어긋나고 남의 저작물을 재배포하는 것이 된다.
그래서 **실재하지 않는 가게**의 메뉴판을 생성해 쓴다.

## ⚠ 상호는 실재 여부를 반드시 확인한다

처음 쓴 **"명륜손칼국수"** 는 지어낸 이름이었는데 **종로구 혜화로에 실재하는 가게**였다
(2026-07-22, Kakao 검색에서 주소까지 정확히 반환돼 발각). 그대로 뒀으면 실제 가게 이름에
우리가 만든 가격표를 붙여 공개하는 셈이었다.

현재 상호 **"한입국수당"** 은 Kakao 전국 검색 0건으로 확인했다(2026-07-22).
이름을 바꾸면 **같은 검사를 다시 한다**:

```
https://dapi.kakao.com/v2/local/search/keyword.json?query=<상호>
→ documents 가 비어 있어야 한다
```

## `sample-menu-pasta.jpg` — 사용자 제공본

`라이프치히하우스` 양식 메뉴판. **사용자가 직접 만들어 건넨 이미지**라 생성 프롬프트가 없다
(원본은 `scripts/make-sample-menu.mjs` 에 넣어 재인코딩만 했다).
상호는 Kakao 전국 검색 **0건** 확인(2026-07-22).

가격이 `17.0` 같은 **천원단위 표기**라 `extract-menu.js` 의 소수 변환 경로를 밟는 유일한 예시다
— 파싱 규칙을 건드리면 이 예시로 먼저 확인한다.

## 생성 방법 (Codex 내장 이미지 생성)

```bash
cd <임시 디렉터리>
codex exec --skip-git-repo-check -s workspace-write "<아래 프롬프트>"
```

### `gen-printed.png` — 인쇄 메뉴판

> 이미지 생성 도구로 이미지를 만들어 이 디렉터리에 gen-printed.png 로 저장해줘.
>
> 내용: 한국 국수집 벽에 걸린 인쇄 메뉴판을 정면에서 찍은 실사 사진.
> - 상단 큰 글씨 상호: 한입국수당
> - 메뉴와 가격(반드시 이 항목 그대로, 또렷하게 읽히게):
>   손칼국수 7,000원 / 바지락칼국수 8,500원 / 들깨칼국수 9,000원 / 콩국수 9,500원 /
>   비빔국수 7,500원 / 김치볶음밥 7,000원 / 제육덮밥 8,000원 / 만두국 8,000원 / 공기밥 1,000원
> - 실제 식당에서 스마트폰으로 찍은 사진 같은 질감(약간의 조명 반사, 자연스러운 벽 배경).
>   다만 글자는 흐리지 않고 선명해야 함.
> - 실존하는 브랜드 로고나 상표는 넣지 말 것.

### `gen-board.png` — 손글씨 화이트보드

> 이미지 생성 도구로 이미지를 만들어 이 디렉터리에 gen-board.png 로 저장해줘.
>
> 내용: 한국 백반집 입구에 세워둔 손글씨 화이트보드를 정면에서 찍은 실사 사진.
> - 맨 위 제목: 오늘의 백반
> - 손글씨(마커펜) 느낌으로 아래 항목 그대로:
>   제육볶음 8,000 / 고등어구이 9,000 / 된장찌개 7,000 / 순두부찌개 7,500 / 계란말이 3,000
> - 맨 아래 작게: 11:30~14:00 학생 500원 할인
> - 실제로 찍은 사진 같은 질감(주변 벽·바닥 살짝 보임, 자연광). 손글씨지만 글자는 또렷하게.
> - 실존 브랜드 로고·상표 넣지 말 것.

### 웹용 재인코딩

생성물은 1536×1024 PNG 약 2.3MB 라 그대로 올리면 썸네일 두 장에 4.6MB 가 나간다.

```bash
node scripts/make-sample-menu.mjs <gen-*.png 가 있는 디렉터리>
```

긴 변 1400px JPEG(q82)로 줄여 `sample-menu-printed.jpg` · `sample-menu-board.jpg` 를 만든다
(각 약 160KB).

## 파싱 실측 (2026-07-22, 배포본 `/api/parse-menu`)

| 파일 | 정확 추출 | 누락 | 오탐 | 상호 추출 |
|---|---|---|---|---|
| `sample-menu-pasta.jpg` | **15/16** | 해산물 토마토 리조또 1건 | 0 (`item_01` 은 아래 참조) | `라이프치히하우스` (Solar) |
| `sample-menu-printed.jpg` | **9/9** | 0 | 0 | `한입국수당` (Solar) |
| `sample-menu-board.jpg` | **4/5** | 순두부찌개 1건 | `학생 500`(하단 안내문) 1건 | 빈칸 — 보드에 상호가 없으므로 정상 |

> `item_01`: Document Parse 가 양식 메뉴판의 SALAD 구역을 **차트로 오인**해
> `Chart Type: bar … item_01 12.5` 를 함께 뱉었고, 그 자리표시자가 화면에
> "item_01 12,500원" 으로 나갔다. `extract-menu.js` 에서 `item_숫자` 이름을 버리도록 고쳤다
> (2026-07-22). 우리 파싱 버그가 아니라 업스트림 산출물의 부산물이다.

손글씨 보드가 완벽하지 않은 것은 **그대로 둔다.** 실측 정확도 76.4%(`experiments/parse-poc/accuracy.md`)
와 결이 같고, 사람 검수 단계가 왜 있는지를 시연에서 그대로 보여준다. 100% 만 나오는 예시만 두면
"AI 가 다 한다"는 인상을 주는데 그건 이 서비스의 주장이 아니다.

예시를 바꾸면 이 표도 **다시 재서** 갱신한다(추정으로 적지 않는다 — 이 표는 한 번
측정 전에 "5/5"라고 적어 뒀다가 실측에서 틀린 적이 있다).
