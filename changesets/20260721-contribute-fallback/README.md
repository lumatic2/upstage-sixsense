# 20260721-contribute-fallback

- Target: DR11 step-5 — SITEMAP 이 요구했으나 미구현이던 두 가지(파싱 0행 폴백 · 정확도 정직 표기).
- Scope: `public/contribute.html`.

## 무엇이 문제였나

파싱이 0행이면 `throw` 후 **1단계로 되돌려 보냈다.** 사진을 다시 고르라는 말인데, 손글씨·조명·각도
때문에 안 읽힌 사진은 다시 올려도 안 읽힌다. 즉 **"AI 가 못 읽었으니 당신도 못 한다"** 로 끝났다 —
사실은 사람이 적으면 되는 일이다.

SITEMAP `:96` 이 "Parse 실패 시 수기 입력 폴백 / 0행 추출 시 빈 행 1개"를, `:91,94` 가 "정확도를
100%인 척하지 않는다"를 이미 설계로 못박고 있었다. 둘 다 화면에 없었다.

## 고친 것

**① 수기 입력 폴백.** 0행·파싱 실패면 2단계에 머문 채 빈 입력 행 1개를 준다(`행 추가` 로 늘림).
사진은 이미 서버로 보낼 준비가 끝나 있어(`PHOTO`) 그대로 함께 접수된다 — 사용자가 아무것도 못 적어도
운영진이 원본 사진을 보고 채울 수 있다. 수기 입력과 파싱 결과를 서버는 구분하지 않는다
(어차피 검수는 사람이 한다).

**② 정확도 표기.** 파싱 성공 화면에 실측치를 적었다 — "메뉴판 사진 6장·72개 항목 중 55개(76.4%)를
정확히 읽었습니다. 나머지를 사람이 잡는 단계가 있어서 이 숫자로도 서비스가 됩니다."
수치 출처는 `experiments/parse-poc/accuracy.md`(정본)에서 읽었다. 폴백 화면에도 같은 숫자로
"흔한 일입니다"를 설명한다 — 실패를 사용자 탓으로 돌리지 않기 위해서다.

## Verification (실브라우저, 배포 URL)

메뉴판이 아닌 사진(캔버스로 만든 단색 JPEG)을 실제로 올려 0행 경로를 탔다.

- [x] 1단계로 튕기지 않고 **2단계 유지** · 수기 입력 행 1개 제공 · 파싱 표 숨김 · 폴백 안내문 노출
- [x] 수집 정확성: `김치찌개 / "8,000원"` → `{name:"김치찌개", price:8000}` (쉼표·단위 제거 확인),
      2행 모두 정상 수집
- [x] Failure probe: 폴백 신설로 기존 거부 경로가 깨지지 않음 —
      `a.txt` → "JPEG · PNG · WebP 만 올릴 수 있습니다." / 9MB → "8MB 이하 사진만 올릴 수 있습니다."
- [x] 파싱 성공 시 폴백 상태가 자동 해제되는지(`exitManual`) 코드 경로 확인
- [x] **시트에 제출하지 않았다** — 접수 버튼 직전까지만 확인(테스트 제보 금지 규약)
