# 20260721-personalization-in-chat

- Target: 사용자 지시 2026-07-21 — 개인화 추천을 어디에 표시하고 어떻게 관리할 것인가. 선택지 3안 제시 후 **"에이전트가 대화로 관리"** 확정(수집 범위는 현행 유지).
- Scope: `public/app.html` 만. 개인화 해명·관리 턴 신설(`profileTurn`·`catSummary`·`clearCats`), 중복 섹션 노트 제거, `loadPicks(wait, {record})` 옵션 추가.
- 문제 정의:
  - 개인화 신호가 **성격이 다른 둘**인데 한 덩어리로 관리되고 있었다 — `last`(사용자가 직접 검색한 조건, 명시적) vs `cats`(카드 클릭에서 **추론한** 분류 관심, 사용자가 준 적 없음). 끄는 수단이 "기록 지우기" 하나뿐이라 "분류 가점만 끄기" 가 불가능했다.
  - 표시가 네 군데로 흩어져 있었고 그중 둘(카드 배지 / 섹션 노트)은 같은 사실의 중복이었다. 제어(로그인 다이얼로그)는 효과가 보이는 자리에서 가장 멀었다.
- Contract:
  - 해명 턴은 **순서가 실제로 바뀐 경우에만** 낸다 — 서버 `personalized` 플래그(가점을 줬어도 순서 불변이면 false)를 트리거로 쓴다. 안 바뀌었는데 "개인화했다" 고 말하면 그것도 거짓말이다.
  - `반영 끄기` 는 `cats` 만 지우고 `last` 는 보존한다. 두 신호를 갈라 관리한다.
  - 컨트롤을 누르면 **화면이 실제로 바뀌어야 한다** — 말만 하고 순서가 그대로면 검증할 근거가 없다. 즉시 재추천을 돌린다.
  - 카드 배지("전에 본 곳과 같은 분류")는 그대로 둔다 — 왜 이 카드인지의 설명은 카드가 계속 진다.
- 검증 중 적발해 고친 버그:
  - **"전부 잊기" 가 거짓 보고를 했다.** `localStorage.removeItem` 직후 `loadPicks()` 안의 `noteConditions()` 가 현재 조건을 도로 써넣어, "전부 잊었습니다" 를 말한 시점에 프로필이 이미 되살아나 있었다(`last:{budget:10000}` 실측). 개인화 정직성이 이 서비스의 핵심 주장인데 그 주장을 담당하는 버튼이 거짓말을 하고 있었다. → `loadPicks(null, {record:false})` 로 해당 렌더만 기록을 건너뛰고, 문구도 사실에 맞췄다("전부 지웠습니다. 다음 검색부터 다시 쌓입니다").
