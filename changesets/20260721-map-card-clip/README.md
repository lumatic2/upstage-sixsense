# 20260721-map-card-clip

- Target: DR11 step-4 — 사용자 실관측 "PC에서 일부 식당의 카드가 잘려서 나온다".
- Scope: `public/app.html`(진짜 원인) · `public/index.html`(랜딩 데모 방어).

## 계획의 전제가 틀렸다

계획 감사는 이렇게 판정했다: *랜딩 데모에는 보정이 없어 항상 잘리고, `/app.html` 은 보정이 있어
정상이다.* **둘 다 틀렸다.** 실측으로 뒤집혔다.

| | 감사 판정 | 실측 (1440×900, 배포 URL) |
|---|---|---|
| 랜딩 데모 | 가장자리 마커에서 **항상** 잘림 | 마커 22개 전수 클릭 — **잘림 0**, 보정 발동 0회. 카드가 최소 110px 안쪽 (초기 `setBounds` 여백이 마커를 가장자리에서 떼어놓는다) |
| `/app.html` | 보정이 동작하므로 정상 | **상단 270.5px 넘침** — 카드 높이(251px)보다 크다. 즉 통째로 안 보인다 |

## 진짜 원인 — 보정이 "느려서" 실패한다

`/app.html` 의 `fitCardIntoView()` 는 카드를 그린 뒤 넘친 만큼 `MAP.panBy()` 로 지도를 민다.
**카카오의 pan 은 애니메이션이다.** 그래서 결과적으로는 카드가 들어오지만, 그 전까지
**1초 가까이 잘린 채로 보인다.** 사용자가 본 것은 그 1초다.

측정이 이걸 드러냈다 — 같은 카드가 클릭 320ms 뒤엔 `+270.5` 넘침, 몇 초 뒤 다시 재니 `-12`(안쪽).
"보정이 있다"는 코드 존재만 보고 정상이라 판정하면 절대 안 잡히는 종류의 결함이다.

## 고친 방식 — 미는 대신 **자리를 고른다**

카드가 설 자리를 **그리기 전에** 정한다(`flipIfNoRoomAbove`). 위에 자리가 없으면 마커 아래로
뒤집고, 조금 모자란 정도(≤60px)면 그만큼만 내린다. 지도를 안 움직이니 처음부터 안 잘린다.

**손잡이 선택에서 두 번 실패했다:**
1. `yAnchor` — CustomOverlay 생성 시점에 고정이라, 높이를 알기 전에는 정할 수 없다.
2. CSS 클래스 `.mcard-wrap.below{transform:…}` — **먹지 않았다.** 배포 후 computed transform 이
   기본값 그대로였다(`matrix(.96,0,0,.96,0,6)`). `.in` 전환과 충돌한 것으로 보인다.
3. → **카카오가 오버레이 컨테이너에 박는 inline `margin-top`** 이 유일하게 확실한 손잡이였다.
   이게 카카오가 앵커를 구현하는 실제 수단이라, 같은 축을 그대로 쓴다.

랜딩에는 잘림이 없었지만 `fitDemoCard()`(app 과 같은 계산)를 이식해 뒀다 — 마커를 늘리거나
지도를 드래그·줌하면 언제든 같은 조건이 생긴다. SITEMAP §2 에 "보정 없이 마커를 추가하지 마라"로 못박았다.

## Verification (실브라우저 1440×900, 배포 URL, 클릭 후 **320ms** 시점 = 사용자가 보는 순간)

- [x] `/app.html` 마커 14개 전수 클릭 — **잘린 것 0**. 최악 케이스 `백미향마라탕 명륜점`
      (마커가 상단에서 8px): **+270.5 → -0.7**
- [x] Failure probe(애니메이션 비의존): 클릭 **25ms** 후에 이미 카드가 경계 안(-86) —
      `transitionend`·200ms 타이머 어느 쪽에도 의존하지 않는다
- [x] 랜딩 데모 마커 22개 전수 — 잘림 0 (회귀 없음)
- [x] `.mcard-wrap` inline script 문법 검사 PASS · 콘솔 red 0

## 남긴 것

`fitCardIntoView`/`fitDemoCard` 의 `panBy` 보정은 **지우지 않았다.** 위아래 양쪽 다 자리가 없는
극단(줌 아웃·아주 긴 카드)에서는 여전히 마지막 방어선이다. 이제는 상시 경로가 아니라 예외 경로다.
