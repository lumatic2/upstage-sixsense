## 판정: **70~90 밴드 — "사용자 수정 전제로 진행 가능"** (README 기준)

## 과정 기록 — 정확도를 끌어올린 것들 (파이프라인 깊이 서사 재료)

1. **1차 실행 62.5% 미만 (3장 0건)**: Document Parse 실패가 아니라 후처리 결함. OCR 텍스트가 `<figure><img alt="...">` 속성으로 오는데 초기 추출기가 태그를 통째로 버림 → **alt 승격** 후 전 장 텍스트 확보. *(같은 OCR 응답 재사용, API 재호출 없이 후처리만 2회 반복 개선: 0건→62.5%→76.4%)*
2. **한국 메뉴판 표기 변형 처리**: 천원단위 소수(`3.0`=3,000 / `1.5`=1,500 — 성대권 백반집 실표기), OCR 공백 천단위(`4 000`, `21 ,000`, `4. 000`), 3자리+원(`500원`), 한 줄 다항목(세로 메뉴 3열 병합), 비한글 UI 토큰 프리픽스(`OUT`, `ALL MENU`) 제거.
3. **옵션 실험 (판정 규약 이행)**: `output_formats` markdown/text 재호출(3장) — OCR 내용 동일, 유의미한 개선 없음(`*.mdtext.json` 보존). 남은 오차는 원본 해상도(웹 축소본) 기인.

## 잔여 실패 모드 (M3 설계 입력)

- **가격 오독** (7.0→17.0): 저해상에서 발생. → 프리필 UX 필수 + 이상치 하이라이트 후보(같은 카테고리 대비 2배↑).
- **이름/가격 줄 분리 누락** (라면/3,500원이 별도 figure): 좌표(data-coord) 기반 인접 병합으로 개선 여지 — M3에서는 프리필 후 사용자 추가로 흡수.
- **카테고리 프리픽스** (볶음밥 김치햄볶음밥): 시각 레이아웃 정보 필요 — 동일하게 사용자 수정으로 흡수.
- **손글씨 첨삭**(500원→700원 인상 덧글)은 원리상 한계 — 검수 단계 존재 이유.
