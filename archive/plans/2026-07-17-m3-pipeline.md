## 스캐폴딩 결정
- 작업 유형: product-service
- frontend: hanipmap 바닐라 JS 유지(ADR-0001, Next.js 전환 없음). 이 레포에는 E2E 검증용 최소 테스트 페이지 `public/test.html` 1장만(사진 업로드→파싱 프리필 폼 + 자연어 검색 입력) — 본 UI 는 팀원 이식 시 hanipmap 제보 폼에 연결.
- backend: Vercel serverless functions(Node 18+, REST). `api/parse-menu.js`(Document Parse 프록시, multipart 수신, 자동 저장 금지 — 후보 반환만) · `api/parse-query.js`(Solar 구조화, 2초 타임아웃 → 정규식 폴백). 계약은 trd-requests ① 5.9/5.10 그대로.
- data: Supabase(팀원 소유 프로젝트) — 학식은 b안 `cafeteria_menus` 별도 테이블. 스키마를 `db/schema.sql` 로 작성해두고, 팀원 권한 도착 전까지 DB write 는 로컬 fixture(`db/fixtures/*.json`)로 대체 검증. 마이그레이션 도구 없음 — SQL 수동 적용(대회 규모).
- design: 해당 없음 — 기존 hanipmap 데모 UI 유지(ADR-0001 결정), 신규 화면 없음(test.html 은 무스타일 검증 도구).
- deploy: 이 레포를 사용자 Vercel 계정에 스테이징 배포(Vercel REST API — `~/.claude/memory/deploy.md` 관례, CLI 는 한글 사용자명 이슈). env: `UPSTAGE_API_KEY` 1개(Vercel 환경변수, 서버 전용). 본배포는 팀원 hanipmap Vercel 로 이식(범위 밖).
- 관측: Vercel function 기본 로그 + 모든 API 실패를 구조화 JSON(`{error, code}`)으로 응답 — 별도 수집 도구 없음(대회 데모 규모, 에러는 데모 리허설에서 로그로 확인).

## 범위와 중단점
- 이번 milestone 이 닫는 것: ① 사진→파싱→후보 반환(검수 프리필) ② 학식 크롤러→구조화 ③ Solar 질의 구조화(+폴백), 스테이징 배포 URL 에서 E2E 관측.
- 범위 밖: hanipmap 본레포 이식(팀원 권한·회신 도착 후 별도 작업) · Supabase 실 insert(권한 전 — fixture 로 대체, 도착 시 `db/schema.sql` 적용만 남김) · 자연캠 학식.
- 중단점: M2 판정 <70% 이면 step-1 진입 전 decision_required 정지 · Upstage API 장애(blocked) · 스테이징 배포용 Vercel 토큰 부재 시 secret_required 정지.
