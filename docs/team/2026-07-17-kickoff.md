## 1. 최종 아웃풋 (조립 완성 상태)

팀 배포 주소 한 곳에서, 사용자 경험 순서로:

1. **처음 온 사람**: 예산을 고르면 → 학식 + 주변 식당이 지도에 표시 + 하단 카드 리스트. 자연어 검색("과선배가 사는 날 2만원 둘이")도 같은 필터에 꽂힘(Solar 구조화).
2. **식당 선택**: 상세 정보(메뉴·가격·위치) + 네이버 길찾기 링크.
3. **다시 온 사람**: 켜자마자 과거 선택 기록 기반 개인화 추천이 첫 화면에 뜸 (예산 조절·직접 선택 여전히 가능). 추천 이유는 Solar가 문장으로 생성.
4. **데이터 공급**: 메뉴판 **사진 1장**으로 제보(Document Parse 프리필→수정→pending) — 지도에 뜨는 메뉴·가격은 이 파이프라인 + 팀 시드 수집 + 학식 크롤러가 채운다.
5. **발표**: 루브릭 6항목 대응 PPT + 위 시나리오 시연.

**전 조각의 동작 레퍼런스**: https://upstage-sixsense-staging.vercel.app/test.html (이 레포 `public/test.html` — 코드가 곧 스펙)

### 개인화 추천 설계 (파트 D의 스펙, 2026-07-17 리서치 확정)

- **계정**: Supabase `signInAnonymously()` — 로그인 화면 없이 앱 켜는 순간 계정 생성, 추후 `linkIdentity()`로 소셜 승격 가능 (https://supabase.com/docs/guides/auth/auth-anonymous, 접근 2026-07-17).
- **기록**: `user_events` 테이블 (user_id, place_id, action: 노출/선택/상세/길찾기, ts) + RLS 본인 한정.
- **추천 점수** = w₁·예산적합도 + w₂·태그 선호도(과거 선택 태그 유사도) + w₃·frecency(빈도+최근성 지수감쇠, Firefox 랭킹 검증 패턴) + w₄·인기도. 신규 유저는 기록이 없어 w₂·w₃ 항이 자연히 0 → cold start 분기 로직 불필요.
- **오픈소스 추천 엔진(Gorse 등)은 비채택** — 상시 서버 필요(serverless 부적합) + 이 규모에선 collaborative filtering이 통계적으로 무의미. PPT "확장 계획"에서 이벤트 스키마가 Gorse 호환임을 언급.
- 구현은 Postgres 함수 또는 serverless JS — 별도 인프라 0.

## 2. 파트 분배 (1인 1파트 × 고정 인터페이스)

> 표의 "인터페이스"는 **동결**이다 — 바꾸고 싶으면 구현 전에 테크리드에게. 각 파트는 서로를 기다리지 않는다.

> 담당은 전원 ☐ — 7/18 미팅에서 배정한다 (결정 문서 §4). 배정 참고: A·D는 서버·DB를 다뤄 난도가 높고 B·C는 화면 작업. D는 A와 맞물리는 부분이 있어 통합 담당과 가까운 사람이 유리.

| 파트 | 담당 | 범위 | 고정 인터페이스 (입력→출력) | 인수 기준 (Verify) |
|---|---|---|---|---|
| **A. 서버·데이터 기반** | ☐ | 이 레포 `api/*.js` 복사(무수정) + `db/schema.sql` 적용(user_events 포함) + 학식 수집·장소 일괄 등록 스크립트 | `POST /api/parse-menu`·`POST /api/parse-query` 규격 그대로 이식 | 팀 배포에서 두 API 200 + `cafeteria_menus`·`user_events` 테이블 존재 |
| **B. 메뉴 제보** | ☐ | 제보 폼에 사진 업로드 연결 + 오류 처리 + 파싱 정확도 재측정 취합(`experiments/parse-poc/accuracy.md`) | `POST /api/parse-menu` `{image: base64, filename}` → `{items:[{name,price}]}` (에러: `{error,code}`) | 실사진 업로드→프리필→항목 수정→pending 제출. 실패 사진(비이미지)에서 에러 메시지 표시. 재측정 표 갱신 |
| **C. 탐색 화면** | ☐ | 검색창을 parse-query 호출로 교체 + 첫 화면 예산 슬라이더 + 해석 결과(source) 표시 + 학식 카드 노출 | `POST /api/parse-query` `{query}` → `{budget, walkMax, tags, reason, source}` (항상 200 — 폴백 내장) · 학식: `cafeteria_menus` SELECT 또는 임시 `/cafeteria-sample.json` | 질의 3종 + 슬라이더 조작이 필터·지도에 반영. `source` 화면 표시. 오늘 날짜 학식이 지도/리스트에 보임 |
| **D. 개인화 서비스** | ☐ | 익명 auth + events API + 추천 스코어링 + 재방문 추천 홈 + 식당 상세·길찾기 화면 | `POST /api/events` `{place_id, action}` / `GET /api/recommend` → `[{place, score, reason}]` (reason은 Solar 생성) | 이벤트 3종 기록→recommend 점수 반영을 SQL로 증명 + 신규 유저(기록 0)도 200 + 카드→상세→길찾기 동선 동작 |
| **공통. 발표·데이터 수집** | 전원 | PPT 조립(담당 ☐ 미팅 지정, 원고: `docs/presentation/deck.md`) + 명륜동 실촬영 각자 5~8곳을 제보 기능으로 직접 입력(파이프라인 dogfood) | — | 루브릭 6항목 슬라이드 + 식당 ≥20곳 시드 입력 + 재측정 표본 ≥3장 + 7/24 전체 리허설 1회 |

- **events 호출 심기**(한 줄짜리)는 파트 D 소유가 아니라 각 화면 담당이 자기 파트에서 — B(제보 제출 시), C(검색 시), D(선택·상세·길찾기 시). 리더 몰빵 방지선.
- 테크리드 운영 업무(PR 리뷰·머지, 크롤러 스케줄, 스테이징 유지, 인터페이스 분쟁 조정, 7/23 통합 E2E 주관)는 파트가 아니라 상시 역할.

## 3. 바이브코딩 착수 프롬프트 (각자 AI에 이 문단 + 레퍼런스 파일을 주고 시작)

- **파트 C**: "검색 입력이 지금 정규식 파싱을 쓰는데, 첨부한 test.html §②처럼 POST /api/parse-query를 호출해서 {budget, walkMax, tags}를 받아 기존 필터 함수에 넘기게 바꿔줘. 응답의 source 필드를 검색 결과 옆에 작게 표시해줘. 첫 화면에 예산 슬라이더를 추가해서 같은 필터 함수를 직접 호출하게 해줘. 그리고 test.html §③처럼 오늘 학식을 카드로 노출해줘."

## 5. 리스크·인터페이스 분쟁 규칙
