# 확정 결정 사항

> 팀 sixsense · 작성 2026-07-18 · 결정 주체: 전유성 (테크리드 결정 후 통보 항목)
> 미팅에서 함께 정할 안건(저장소·배포, 결정권, 발표자, 통합 담당)은 이 문서가 아니라 미팅에서 다룬다.

## 1. 제품 범위

**형태는 모바일 웹서비스다.** 설치 없이 링크(QR)로 접속한다. 심사위원이 본인 폰으로 바로 열어볼 수 있고, 남은 기간에 앱은 불가능하다. 화면은 폰 기준으로 만들고 데스크톱은 깨지지 않는 수준이면 된다.

**지역은 성균관대 명륜캠퍼스와 도보권(명륜동·혜화 일대)만 다룬다.** 자연캠퍼스는 제외한다. 데이터를 직접 채워야 하는 서비스라 지역을 좁혀야 지도에 밀도가 생긴다. 다른 캠퍼스 복제 가능성은 발표의 확장성 항목으로 돌린다.

**타깃은 한 끼 예산이 정해진 성균관대 재학생이다.** "점심 뭐 먹지"가 아니라 "이 돈으로 뭐 먹지"가 우리가 푸는 문제다. 심사기준의 "문제 정의의 구체성 및 타깃 명확성"(5점)에 대응한다.

**기능은 시연 5장면이 전부다. 이 밖에는 만들지 않는다.**

| # | 장면 | 기능 |
|---|---|---|
| 1 | 예산 선택 → 학식+주변 식당이 지도·카드에 표시 | 예산 필터 + 지도 |
| 2 | "과선배가 사는 날 2만원 둘이" → 조건 해석 | 자연어 검색 (Solar) |
| 3 | 식당 선택 → 예산 안 메뉴 조합 + 길찾기 | 상세 화면 |
| 4 | 재방문자에게 과거 기록 기반 추천 | 개인화 |
| 5 | 메뉴판 사진 1장 → 제보 완료, 데이터가 쌓임 | 사진 제보 (Document Parse) |

**만들지 않는 것(명시적 제외):**

- 리뷰·평점·커뮤니티 (에브리타임과 정면 충돌 지대, 우리 차별점 아님)
- 회원가입·로그인 화면 (개인화는 익명 계정으로 처리)
- 영수증 인식·잔여 예산 관리 (기간 밖, 발표 "다음 단계" 슬라이드로)
- 예약·주문·결제, 영업시간 실시간 확인, 앱 버전

개발 중에 기능 추가 의견이 나오면 논쟁하지 않고 이 목록을 기준으로 판단한다. 목록을 바꾸려면 미팅 안건으로 올린다.

**품질 기준: UI는 동작하고 깨지지 않는 수준이면 충분하다.** 심사 배점에 UI 완성도가 없다. 남는 시간은 데이터 수집과 파이프라인 정교화에 쓴다.

## 2. 기술

**화면은 프레임워크 없이 순수 HTML + CSS + JavaScript로 만든다.**
빌드 과정이 없어 파일을 고치면 바로 화면에 반영되고, AI 코딩 도구로 작업할 때 오류가 나는 지점이 크게 줄어든다. 7일짜리 프로젝트에서 프레임워크 학습 비용은 순손실이다.

**지도는 네이버 지도 API를 쓴다.**
캠퍼스 주변 국내 장소 데이터가 가장 정확하고, 상세 화면의 길찾기는 네이버 지도 링크 하나로 해결된다(별도 개발 없음).

**DB와 계정은 Supabase를 쓴다.**
데이터베이스와 익명 로그인이 한 서비스에 들어 있다. 익명 로그인은 사용자가 가입 화면을 보지 않아도 접속 순간 계정이 생기는 방식이라, 회원가입 기능을 만들지 않고 개인화(시연 장면 4)를 구현할 수 있다.

**서버와 배포는 Vercel을 쓴다.**
git에 push하면 자동으로 배포된다. **Upstage API 키는 반드시 서버 함수 안에서만 사용한다.** 화면 코드(브라우저)에 키를 넣으면 누구나 열람해 훔칠 수 있다. 이 규칙은 전원 예외 없음.

**Upstage 제품은 쓰는 곳과 안 쓰는 곳을 구분한다.**

| 대상 | 처리 | 이유 |
|---|---|---|
| 메뉴판 사진 | Document Parse | 사진 속에 갇힌 비정형 정보의 구조화. 제품이 설계된 용도 그대로 |
| 검색 문장, 추천 이유 | Solar | 자연어 해석과 문장 생성 |
| 학식 식단 페이지 | 자동 수집 스크립트 (Upstage 안 씀) | 이미 HTML 표로 구조화된 데이터. 억지 적용은 "목적에 적합한 제품 적용"(10점) 감점 사유 |

학식 자동 수집은 파트 A(서버·데이터 기반) 소유다. 하루 한 번 학식 페이지를 열어 식단표를 읽고 DB에 저장하는 수십 줄짜리 스크립트로, 화면 작업과 무관하다. Document Parse를 학식에 안 쓰는 것은 못 쓰는 게 아니라 안 쓰는 판단이며, 이 구분 자체가 발표의 "적재적소" 근거가 된다.

**파트 사이의 연결부(API 계약) 4개를 지금 확정하고 개발 중에 바꾸지 않는다.**

> 쉬운 비유: 식당의 주문서 양식. 홀 직원과 주방이 따로 일할 수 있는 이유는 둘 사이에 주문서 양식이 정해져 있기 때문이다. 홀은 양식대로 적어 넘기고 주방은 양식대로 읽는다. 홀은 주방이 어떻게 요리하는지 몰라도 되고, 주방은 홀이 손님을 어떻게 응대하는지 몰라도 된다. 양식만 안 바뀌면 둘은 서로를 기다리지 않는다. 우리 프로젝트의 주방은 서버(파트 A·D), 홀은 화면(파트 B·C·D)이고, API 계약이 그 주문서 양식이다.

| 계약 | 화면이 보내는 것 → 서버가 돌려주는 것 | 쓰는 파트 |
|---|---|---|
| 사진 파싱 | 메뉴판 사진 1장 → 메뉴·가격 목록 | B (제보) |
| 검색 해석 | 문장 → 예산·거리·태그 | C (탐색) |
| 행동 기록 | 장소 + 행동(선택/상세/길찾기) → 저장 완료 | B·C·D 각 화면에서 한 줄 호출 |
| 추천 | (자동으로 본인 기록 참조) → 추천 목록 + 추천 이유 | D (개인화 홈) |

계약이 확정되어 있으면 화면 담당은 서버가 없어도 가짜 응답(목 데이터)으로 오늘 개발을 시작할 수 있고, 서버 담당은 화면 없이 규격 충족만 확인하면 된다. 반대로 개발 중에 양식을 바꾸면 서버와 화면이 동시에 고쳐야 하는 대기 관계가 생긴다. 그래서 규격 변경은 통합 담당 승인 없이는 금지다. 테이블은 4개(장소, 메뉴, 학식 식단, 사용자 행동 기록)이며 상세 필드 명세는 개발 문서에서 관리한다.

## 3. 데이터

**초기 데이터는 출처를 둘로 나눈다. 장소 정보는 지도 API에서, 메뉴·가격은 직접 촬영으로.**

| 데이터 | 출처 | 방법 |
|---|---|---|
| 식당 이름·좌표·주소·카테고리 | 네이버/카카오 Local API (정식 제공 데이터) | 스크립트로 일괄 등록 |
| 메뉴·가격 | 팀 실촬영 메뉴판 사진, 전원 각 5~8곳 (합계 20곳 이상) | 우리 제보 기능으로 직접 입력 |
| 학식 식단 | 학교 페이지 자동 수집 | 파트 A 스크립트 |

네이버 플레이스 화면에 보이는 메뉴를 베껴오는 것은 금지한다. 이유는 세 가지다. 지도 서비스 약관이 화면 데이터 수집을 금지하고(심사 Q&A에서 출처 질문에 답할 수 없게 된다), "이 데이터는 어디에도 구조화되어 있지 않아 우리가 사진으로 생산한다"는 발표 서사가 거짓이 되며, 방치된 옛 메뉴가 많아 "검증된 현재 가격"이라는 약속과도 어긋난다. 메뉴·가격 API가 존재하지 않는다는 사실은 직접 확인했고, 바로 그 공백이 이 서비스의 존재 이유다.

실촬영은 네이버 베끼기와 법적 성격이 다르다. 메뉴 이름과 가격은 사실 정보라 저작권 보호 대상이 아니고(저작권법상 창작성 요건 미충족), 공개된 장소에 게시된 가격을 옮겨 적는 행위와 같다. 크라우드소싱 가격 서비스들이 모두 이 원리로 운영된다. 다만 안전선 하나를 긋는다. **메뉴판 사진 원본은 서비스에 게시하지 않는다.** 메뉴판에 일러스트 등 창작 요소가 있을 수 있으므로 사진은 파싱 입력으로만 쓰고, 추출된 텍스트 데이터만 지도에 올린다.

팀 촬영 입력은 우리 제보 파이프라인을 우리가 첫 사용자로 검증하는 과정을 겸한다. 버그가 있으면 데모 전에 우리가 먼저 만난다. 구역 배분은 미팅에서 정한다.

**파싱 결과는 바로 지도에 올리지 않고 사람이 확인한 뒤 반영한다.**

> 쉬운 비유: 위키백과가 아니라 신문사 방식. 기자가 쓴 기사도 데스크가 한 번 보고 내보낸다. AI는 가격 숫자를 틀릴 수 있고, 틀린 가격이 지도에 뜨면 예산 기반 추천이라는 서비스의 신뢰가 무너진다.

흐름: 사진 제보 → AI 자동 추출 → 제보자가 그 자리에서 수정 → 승인 대기 → 확인 후 지도 반영. 자동화에 사람 검증이 결합된 파이프라인은 심사기준 "정보 처리의 깊이·정교성"(15점)에 직접 대응한다.

**파싱 정확도를 숫자로 만들고 7/23에 동결한다.**

수집한 메뉴판 사진 중 일부를 채점용으로 지정해 정답(실제 메뉴·가격)을 적어두고, AI 추출 결과와 대조해 "항목 몇 개 중 몇 개"를 센다. 발표에서 "사진을 올리면 잘 추출된다"는 주장이지만 "실촬영 N장, 항목 M개 기준 X%"는 증거다. 숫자는 7/23에 동결하고 이후 재측정하지 않는다. 정확도가 100%가 아닌 것은 문제가 아니다. 검수 단계를 둔 이유가 그것이고, 정직한 실측치에 보완 장치를 더한 쪽이 부풀린 숫자보다 강하다.

## 4. 팀 운영

**저장소는 전유성 계정에 신규 생성하고 미팅 전에 준비를 마친다.** 폴더 구조, 파트별 작업 파일, 각자의 작업 브랜치까지 만들어 둔다. 팀원이 알아야 할 것은 "본인 브랜치에서 본인 파일만 수정한다" 하나다. 파트별로 만지는 파일이 분리되어 있어 충돌이 원천적으로 나지 않는다.

**git 조작은 AI 코딩 도구에 맡긴다.** 커밋과 푸시는 도구에 "작업 저장하고 올려줘"라고 지시하면 되고, PR 생성은 GitHub 웹 화면의 버튼 조작이라 스크린샷 가이드 한 장으로 안내한다. 이슈·브랜치 전략 같은 개념 교육은 하지 않는다.

**머지와 배포는 전유성이 단독 관리한다.** PR 판단·머지, Vercel 배포, 도메인, API 키 전부. 배포 권한이 한 곳에 있으면 서비스가 예기치 않게 죽는 경로가 줄어든다. 통합·게이트 역할이 한 사람에게 있는 것은 개발 노동이 한 사람에게 몰리는 것과 다르다. 각 파트가 독립적이면 게이트 통과는 가볍고 병목이 되지 않는다.

**진행 규약:**

- 각자 본인 브랜치에서 작업하고 PR을 올린다. 남의 파일은 수정하지 않는다.
- 작업이 막히면 화면과 오류 메시지를 공유하고 다른 작업으로 전환한다. 한 문제를 혼자 3시간 이상 붙잡지 않는다.
- 완료 판정은 본인 선언이 아니라 파트별 완료 기준 충족과 통합 담당 확인으로 한다.

**미팅에서 함께 정할 것:** 결정권 규칙(제안: 제품 방향은 다수결, 기술은 테크리드, 데이터 구조는 파트 A 담당), 발표자 지정, 데이터 수집 구역 배분, 파트 배정 수락.

## 5. 발표

발표 스토리 골격은 지금 정하지 않는다. 기능과 데이터가 형태를 갖춘 뒤(7/22 전후) 실물 기준으로 짠다. 지금 확정하는 것은 일정 원칙 두 개다.

- **7/23 숫자 동결**: 파싱 정확도, 데이터 개수 등 발표에 들어가는 모든 숫자는 7/23 이후 바꾸지 않는다.
- **7/24 전체 리허설 1회**: 시연 시나리오 5장면을 실제 배포 주소에서 처음부터 끝까지. 발표는 지정된 발표자가 하되 리허설은 전원 참석.
