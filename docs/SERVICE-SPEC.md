# 한입지도 — 서비스 스펙 (발표자료 제작용)

> **누구에게**: 발표자료를 만드는 팀원에게.
> **무엇을**: 이 서비스가 무엇이고, 어떻게 만들었고, 어떤 숫자를 댈 수 있는지.
> **언제 기준**: 2026-07-23 실측. 숫자는 전부 재서 적었고 §7에 측정 방법을 남겼다.
> 문서 끝(§13)에 7분 발표용 슬라이드 구조안이 있다.

- 라이브: <https://sixsense.askewly.com>
- 코드: <https://github.com/lumatic2/upstage-sixsense>
- 서비스 설명 페이지(심사위원이 바로 볼 수 있는 증거): <https://sixsense.askewly.com/about.html>

## 1. 한 줄 요약

**성균관대 명륜캠퍼스 주변에서, 오늘 가진 예산으로 먹을 수 있는 곳을 알려주는 서비스.**
가격 정보가 웹에 없어서 범용 챗봇이 못 하는 일을, 메뉴판 사진을 직접 읽어 데이터로 만들어서 한다.

---

## 2. 문제 정의와 타깃

### 타깃
성균관대 명륜캠퍼스 학생. 점심·저녁마다 "얼마 있는데 뭐 먹지"를 반복하는 사람.

### 문제
**동네 식당의 실제 메뉴와 가격은 웹에 없다.** 블로그에는 몇 년 전 가격이 있고, 지도 앱에는 상호와 위치만 있다. 가격은 대부분 **가게에 붙은 메뉴판 사진에만** 존재한다.

그래서 ChatGPT에 "명륜동에서 8천원으로 뭐 먹지"라고 물으면 이렇게 끝난다.

> "정확한 가격은 매장에 문의하세요."

이건 모델이 부족해서가 아니라 **읽을 데이터가 없어서**다. 문제는 모델이 아니라 데이터 쪽에 있다.

### 우리 답
데이터를 **직접 만든다.** 메뉴판 사진을 모아 Upstage Document Parse로 읽고, 사람이 검수한 것만 서비스에 올린다. 그 위에서 예산·도보 시간으로 검색한다.

### "그냥 ChatGPT 쓰면 되지 않나?"에 대한 답
| | 범용 LLM | 한입지도 |
|---|---|---|
| 가격 출처 | 학습 데이터(시점 불명) | 메뉴판 사진 + 촬영일 + 검수자 |
| 없는 정보 | 그럴듯하게 지어냄 | 빈칸으로 둠 |
| 검증 | 없음 | 다른 모델이 근거 판정 |
| 예산 검색 | 문장으로만 | 실제 가격 데이터로 필터 |

## 3. 화면 구성

| 화면 | 경로 | 하는 일 |
|---|---|---|
| 랜딩 | `/` | 서비스 소개. 예산 칩을 눌러 결과를 바로 맛본다 |
| 메뉴 찾기 | `/app.html` | 본 서비스. 지도 + 대화 + 예산·도보 필터 |
| 사진 제보 | `/contribute.html` | 메뉴판 사진 올리기 → 파싱 결과 확인 → 접수 |
| 데이터 파이프라인 | `/about.html` | 만드는 과정 전체 설명 + **읽는 자리에서 바로 실행되는 증거 4개** |
| 검수 | `/review.html` | 운영자용. 파싱 결과를 원본 사진과 대조해 확인/제외 |

**로그인이 없다.** 계정 없이 전부 쓴다. 지난 방문의 관심어는 브라우저 안에만 남고, 그 기억이 **추천 순서를 바꾸지는 않는다**(근거 → `docs/adr/0004-personalization.md`).

### `/about.html`의 증거 블록 4개 (발표에서 강조할 만한 부분)
설명 옆에 **그 자리에서 실제로 도는 것**을 붙였다. 스크린샷이 아니라 진짜 API 호출이다.

1. 오늘의 학식 (열면 자동으로 채워짐)
2. 검수 통과 데이터 목록 (열면 자동으로 채워짐)
3. **예시 메뉴판을 눌러 Document Parse 실행** → 후보 16개가 그 자리에 나옴
4. **문장을 넣으면 Solar가 조건으로 변환** → 원본 응답 그대로 표시

## 4. 데이터 파이프라인

```
들어오는 것                처리                        모이는 곳        나가는 곳
─────────────            ──────────────              ──────────      ──────────
메뉴판 사진      →   Upstage Document Parse   ┐
학식 웹페이지    →   매일 자동 수집(크롤링)    ├→   사람 검수   →   한입지도
가게 주소        →   주소 → 지도 좌표          ┘
```

**핵심 주장: 입력은 세 갈래인데 나가는 문은 하나다.** 어느 경로로 들어왔든 사람이 확인한 것만 화면에 나간다.

### 단계별

**① 수집** — 명륜동·혜화 식당을 직접 돌며 메뉴판을 촬영. 출처와 촬영일을 함께 저장.

**② 파싱** — Upstage Document Parse에 `ocr=force`, `coordinates=true`로 요청. 메뉴명과 가격을 뽑아낸다.

**③ 검수** — 사람이 원본 사진과 대조하며 한 줄씩 확인. **`검수=확인`인 행만 서비스에 노출된다.** 이 게이트는 API 레벨에 있어서 우회 경로가 없다.

**④ 보강** — 주소로 좌표를 채우고(도보 시간 계산에 필요), 학식은 매일 06:10에 크롤링(`vercel.json` cron).

**⑤ 서비스** — 예산·도보 시간·태그로 필터 → Solar가 추천 이유 생성 → 다른 모델이 근거 판정.

### 학식은 왜 파싱하지 않나
학교 식단 페이지는 **이미 표로 정리된 HTML**이다. 여기에 사진 파싱을 얹으면 비용은 늘고 정확도는 떨어진다. 쓸 수 있다고 다 쓰지 않았다는 것이 이 프로젝트의 판단이다.

| 입력 | 형태 | 처리 |
|---|---|---|
| 메뉴판 | 사진(비구조) | Document Parse → 검수 |
| 학식 | HTML(구조화) | 자동 수집 → 형식 맞추기 |
| 위치 | 주소 텍스트 | 좌표 변환 |

## 5. 데이터 구조

**정본은 구글 스프레드시트 하나다.** 별도 DB를 쓰지 않는다(`docs/adr/0005-stack-record-correction.md`). 팀원 누구나 열어서 고칠 수 있고, 검수도 시트에서 한다.

### 시트 `식당` (11열)
`식당ID · 식당이름 · 카테고리 · 주소 · 위도 · 경도 · 태그 · 메뉴판 사진 링크 · 수집자 · 촬영일 · 상태`

### 시트 `메뉴` (7열)
`식당ID · 식당이름 · 메뉴명 · 가격(원) · 출처 · 검수 · 비고`

- **출처**: `사진파싱` / `제보` / `직접입력` — 값이 틀렸을 때 어디를 고쳐야 하는지 알려준다
- **검수**: `확인` / `제외` / `대기` — 화면에 나갈지 말지를 가르는 스위치
- **비고**: `사이드` / `한끼` — 곁들임 판정을 사람이 뒤집는 칸

### 시트 `학식`
크롤러가 매일 채운다. 마지막으로 들어온 날짜를 함께 내보내서, **"오늘 휴무"와 "정보 없음"을 구별**한다.

## 6. Upstage 적용

**네 자리에서 쓴다. 데이터가 들어올 때 두 곳, 사용자에게 나갈 때 두 곳.**

| 제품 | 무슨 일을 시켰나 | 왜 이 모델인가 |
|---|---|---|
| **Document Parse** | 메뉴판 사진에서 메뉴명과 가격을 읽음 (`ocr=force`, `coordinates=true`) | 가격은 웹 어디에도 없고 메뉴판 사진에만 있다 |
| **Solar `solar-pro2`** | 제보받은 사진에서 가게 이름을 찾음 | 사람이 타이핑할 칸을 줄인다. 이름이 없으면 빈칸을 돌려주게 못 박았다 |
| **Solar `solar-pro2`** | 사용자 문장에서 예산·도보 시간·태그를 뽑고, 답과 추천 이유를 씀 (한 번의 왕복, 구조화 출력) | "둘이서 2만원"이면 1인 1만원으로 나눠야 하는데 `solar-mini`는 이 나눗셈을 틀렸다 |
| **Solar `solar-mini`** | `solar-pro2`가 쓴 추천 이유를 데이터와 대조해 채점 | 쓴 모델이 스스로 채점하면 검증이 아니라서 일부러 다른 모델에 맡겼다 |

**안 쓴 자리: 학식 식단.** 이미 표로 정리된 HTML이라 사진 파싱을 얹으면 비용만 늘고 정확도는 떨어진다.

### 구조화 출력 예시
입력: `"8천원 이하 10분 거리 이내 혼밥"`
출력:
```
{ "budget": 8000, "walkMax": 10, "tags": ["혼밥"],
  "reason": "8천원 이하 예산, 10분 이내 거리, 혼밥 태그 조건 명시됨", "source": "solar" }
```

## 7. 환각을 막는 장치 (차별성의 핵심)

이 서비스의 주장은 **"모델이 말한 것을 데이터가 검증한다"**이다. 네 겹으로 걸었다.

### ① 검수 게이트 — AI는 후보를 만들고 사람이 통과시킨다
`검수=확인`이 아닌 행은 API가 내보내지 않는다. 제보로 들어온 것도 예외 없다.

### ② 근거 판정 — 생성과 검증에 다른 모델
`solar-pro2`가 추천 이유를 쓰면, `solar-mini`가 그 문장을 실제 메뉴·가격·도보 시간과 대조해 `grounded / notGrounded / notSure`로 판정한다. 근거가 없으면 문장을 버리고 데이터 사실 문장으로 바꾼다.
화면에는 **통과하지 못해 바뀐 카드에만** "근거 미달 → 데이터 사실로 교체"가 붙는다. 통과한 카드에 도장을 찍으면 정작 눈에 띄어야 할 표시가 묻히기 때문이다.

### ③ 기억 검증 — 사용자가 말한 낱말을 데이터와 대조
"마라탕버거"를 검색해도 기억하지 않는다. **우리 데이터 안에 실제로 있는 낱말만** 남긴다. 꺼낼 때도 다시 대조한다.

### ④ 추천 칩 검증 — 제안하기 전에 결과가 있는지 확인
"도보 3분 안" 같은 후속 제안을 띄우기 전에, 그 조건으로 실제 결과가 나오는지 먼저 센다. 빈 화면으로 보내는 제안은 띄우지 않는다.

## 8. 실측 수치 (2026-07-23)

### 규모
| 항목 | 값 |
|---|---|
| 수집한 식당 | **43곳** (좌표 확보 42곳) |
| 파싱이 만든 메뉴 후보 | **579개** |
| 검수 결과 | 확인 **487** · 제외 **30** · 대기 **62** |
| 화면에 나가는 메뉴 | **486개** (본메뉴 361 · 곁들임 125) |
| 본메뉴 가격 | 중앙값 11,000원 (2,500 ~ 75,000원) |
| 조회하는 학식당 | **4곳** (패컬티·은행골·법고을·금잔디) |
| 카테고리 | 한식·일식·중식·양식·분식·세계음식·고기/구이·술집·카페/디저트 |

제외한 30개는 영업시간, 가게 연혁, 옵션 가격처럼 **메뉴가 아닌 것**들이다.

### 파싱 정확도
사진 5장·72개 메뉴를 사람이 채점한 결과 **55개 정확 = 76.4%**.

중요한 건 정확도 자체보다 **무엇이 정확도를 흔드는지 알아낸 것**이다. 사진 상태별로 나눠 보면 지배 변수는 하나였다.

| 사진 상태 | 정확도 |
|---|---|
| 인쇄체 · 해상도 충분 (2장) | **100%** (37/37) |
| 표기 변형 · 손글씨 (2장) | **64.0%** (16/25) |
| 저해상 흐림 (1장) | **20.0%** (2/10) |

**반사와 거리는 문제가 아니었다.** 유리창에 붙은 메뉴를 멀리서 반사까지 얹어 찍은 사진이 25개 항목 전부 맞았다. 무너지는 건 해상도가 부족할 때뿐이었다.
→ 그래서 "잘 찍어 달라"가 아니라 **"해상도만 확보하면 된다"**가 수집 지침이 됐다.

### 비용과 재현성
정확도 측정에 Document Parse **8회 호출, 약 $0.08**. 그때 받은 **응답 원본 5건을 파일로 저장**해 두었기 때문에, 후처리 규칙을 고칠 때마다 API를 다시 부르지 않고 같은 응답으로 재채점할 수 있다.
실제로 이 방식으로 **API 재호출 없이 규칙만 두 번 고쳐 0% → 62.5% → 76.4%**로 올렸다.

> 1차에 3장이 0건이었던 것은 Document Parse 실패가 아니라 우리 후처리 결함이었다. OCR 텍스트가 `<figure><img alt="...">` 속성으로 오는데 초기 추출기가 태그를 통째로 버렸다. **alt 승격** 후 전 장의 텍스트를 확보했다.

### 측정 방법 (숫자를 다시 재야 할 때)
- 규모: `GET /api/data`
- 검수 3상태: `GET /api/review?status=확인|제외|대기` (헤더 `x-review-token`)
- 정확도: `experiments/parse-poc/accuracy.md`

## 9. 시행착오 (Solution Depth 재료)

발표에서 "깊이"를 보여줄 때 쓸 다섯 가지. 전부 실제로 겪고 고친 것이다.

| 걸린 것 | 왜 문제인가 | 어떻게 했나 |
|---|---|---|
| **곁들임 메뉴** | "8천원 이하"에 공기밥 1,000원이 걸리면 그 가게가 조건을 통과한다 | 메뉴마다 본메뉴/곁들임을 판정해 **예산 판정과 개수 세기 모두 본메뉴로만** 한다. 규칙은 63개 케이스로 고정 |
| **천원단위 소수 표기** | 일부 식당이 가격을 `7.0`·`1.5`로 적는다 | 1,000배로 환산. OCR이 만든 공백 천단위(`4 000`, `21 ,000`)도 같은 자리에서 흡수 |
| **메뉴판을 그래프로 읽음** | Document Parse가 메뉴 구역을 그래프로 오인해 `Chart Type: bar … item_01 12.5`를 뱉었고, 그 빈칸 이름이 화면에 **"item_01 12,500원"**으로 나갔다 | 기계가 붙인 빈칸 이름은 버린다. 우리 버그가 아니라 **읽어 온 결과에 섞여 들어온 것**이라 받는 자리에서 막는 게 맞다 |
| **LLM의 기억 오염** | "마라탕버거"(없는 메뉴)를 검색하면 그걸 관심어로 저장해 다음 추천에 썼다 | 기억할 낱말을 **실제 메뉴·식당 이름과 대조**해 통과한 것만 남긴다. 꺼낼 때도 다시 대조 |
| **추천 이유의 근거** | 이유 문장은 Solar가 쓴다. 데이터에 없는 말이 섞여도 문장만 봐서는 티가 안 난다 | 실제 데이터와 한 줄씩 대조. 대조는 **다른 모델**이 맡고, 근거가 없으면 데이터 사실로 바꾼다 |

### 곁들임 판정의 설계 원칙 (디테일을 물으면)
규칙은 **어느 가게에나 통하는 것만** 담고, 한 가게 사정은 **시트 `비고` 열로 뒤집는다.**
- 규칙에 넣은 것: 고량주·이과두주 같은 술, 만두류·꽃빵, `100g` 단가 표기, `반접시`
- 시트로 뒤집은 것: `모듬튀김 1인`(같은 집에 온전한 것이 따로 있음), `Green Salad`(같은 집 `Steak Salad`는 한 끼)

"튀김 1인"이나 "샐러드"를 규칙에 넣었으면 다른 가게 본메뉴까지 끌려갔다.

## 10. 장애 대응

| 멈추는 곳 | 그냥 두면 | 대응 |
|---|---|---|
| 학식 크롤러 | 새 식단이 안 들어오면 "오늘 휴무"를 내보낸다. 데이터가 없는 것과 가게가 쉬는 것이 구별되지 않는다 | 마지막으로 들어온 날짜를 함께 보내, 오늘보다 이르면 **"정보 없음"**으로 표시 |
| Document Parse | 제보 화면에서 OCR이 실패하면 사용자가 바로 나간다 | 한 줄도 못 읽거나 실패하면 **직접 입력으로 흐름을 이어간다** |
| Solar | 이유 생성이 실패하면 추천 자체가 죽는다 | 데이터 기반 템플릿 이유로 폴백. **추천 루프는 죽지 않는다** |
| 기능 하나 | 하나가 죽으면 서비스 전체가 불능 | 기능을 따로 떼어 두어 하나가 멈춰도 나머지는 돈다. 추천이 안 되어도 지도와 학식은 보인다 |

## 11. 기술 스택

| 층 | 선택 | 이유 |
|---|---|---|
| 프론트 | **바닐라 JS** (빌드 스텝 없음) | 화면이 5개고 상태가 얕다. 프레임워크 도입 비용이 회수되지 않는다 |
| 지도 | **Kakao Maps** | `docs/adr/0002-kakao-maps-over-naver.md` |
| 데이터 | **구글 스프레드시트 단일** | 팀원이 바로 열어 고친다. 검수도 여기서 한다 |
| 서버 | **Vercel serverless functions** | API 키를 서버에만 둔다. 클라이언트에 노출 없음 |
| AI | **Upstage Document Parse + Solar** | §6 |

**API 목록**: `/api/data` `/api/chat` `/api/recommend` `/api/parse-query` `/api/parse-menu` `/api/identify` `/api/contribute` `/api/review` `/api/config` + cron `/api/cron/crawl-cafeteria`

**도보 시간 기준점은 경영관 고정.** 브라우저 위치 권한을 받지 않으므로 "도보 N분"은 사용자 현위치가 아니라 캠퍼스 안 고정 지점에서 잰 값이고, 화면 문구도 그렇게 말한다.

### 검증 (매 배포마다 돈다)
| 테스트 | 케이스 |
|---|---|
| 곁들임 판정 | 63 |
| 대화에서 조건 뽑기 | 35 |
| 추천 칩 검증 | 18 |
| 배포된 사이트를 실제 브라우저로 훑기 | 16 |

규칙을 고치다 예전에 맞던 것이 깨지면 여기서 걸린다.

## 12. 한계 (물어보면 정직하게 답할 것)

- **정확도 76.4%.** 저해상 사진에서 크게 떨어진다. 그래서 사람 검수를 필수 단계로 뒀다.
- **손글씨 첨삭은 못 읽는다.** 5,000원을 지우고 7,000원을 덧쓴 메뉴판은 실패했다.
- **범위가 명륜캠퍼스 주변 43곳.** 확장은 같은 파이프라인을 다른 동네에 돌리면 되지만 아직 안 했다.
- **로그인이 없다.** 기기를 바꾸면 기억이 이어지지 않는다. 의도된 선택이다(ADR-0004).
- **가격은 촬영 시점의 값.** 그래서 촬영일을 함께 저장하고 오래된 것부터 다시 확인한다.

## 13. 슬라이드 구조안 (발표 7분 + 질의응답 5분)

> 배점이 **데이터 파이프라인 20 + 처리 깊이 15 = 35점**, Upstage 활용 20점이다.
> **슬라이드 절반 이상을 파이프라인과 깊이에 써야 한다.** 서비스 소개는 짧게.

| # | 슬라이드 | 시간 | 핵심 메시지 | 근거 자료 |
|---|---|---|---|---|
| 1 | 표지 | 10초 | 서비스명 · 팀 · 한 줄 정의 | — |
| 2 | **문제** | 40초 | ChatGPT에 물었더니 "매장에 문의하세요". **모델이 부족한 게 아니라 읽을 데이터가 없다** | §2. 실제 챗봇 답변 캡처 1장 |
| 3 | **타깃** | 20초 | 명륜캠 학생, 하루 두 번 반복되는 결정 | §2 |
| 4 | **해법 한 장** | 30초 | 데이터를 직접 만든다 → 그 위에서 예산으로 검색 | §1 |
| 5 | **파이프라인 전체도** ★ | 60초 | 입력 3갈래 → 각기 다른 처리 → **사람 검수 한 문** → 서비스 | §4 흐름도. `/about.html` 상단 SVG를 그대로 쓰면 됨 |
| 6 | **단계별 상세** ★ | 60초 | 수집 → 파싱 → 검수 → 보강. 각 단계의 산출물과 게이트 | §4 |
| 7 | **데이터 구조** ★ | 40초 | 모든 행이 출처·검수·촬영일을 들고 다닌다. **왜 그 칸이 필요한가** | §5 |
| 8 | **Upstage 적용 4자리** ★ | 50초 | 표 한 장 + **"학식에는 안 썼다"** | §6 |
| 9 | **환각 방지 4겹** ★ | 60초 | 검수 게이트 · 생성/검증 모델 분리 · 기억 검증 · 칩 검증 | §7. 이 서비스의 차별성 |
| 10 | **정확도와 그 해석** ★ | 50초 | 76.4%, 그리고 **지배 변수는 해상도 하나**. 반사·거리는 문제가 아니었다 | §8 구간별 표 |
| 11 | **재현성** | 30초 | 응답 원본을 박제해 API 재호출 없이 재채점. 0% → 62.5% → 76.4% | §8 |
| 12 | **시행착오** ★ | 40초 | 5개 중 2~3개만 골라서. `item_01 12,500원` 사고가 가장 잘 먹힌다 | §9 |
| 13 | 규모 실측 | 20초 | 43곳 · 후보 579 · 화면 486 · 학식 4곳 | §8 |
| 14 | **시연 영상** | 60~90초 | 예산 넣기 → 추천 → 근거 배지 → 제보 → 파싱 | 별도 녹화 |
| 15 | 한계와 다음 | 20초 | 정직하게. 확장 계획 한 줄 | §12 |
| 16 | 마무리 | 10초 | 링크 2개(라이브·깃허브) | — |

★ = 배점이 걸린 슬라이드. 여기를 줄이지 말 것.

### 발표 원칙
- **숫자는 전부 §8에서 가져온다.** 추정치를 슬라이드에 올리지 않는다.
- **"안 한 것"을 말한다.** 학식에 파싱을 안 쓴 판단, 순서 개인화를 걷어낸 판단. 도구를 다 쓰는 것보다 가려 쓴 것이 심사에서 세다.
- **`/about.html`을 열어 두면 백업이 된다.** 시연이 막히면 그 페이지의 실행 블록이 진짜 API를 그 자리에서 돌린다.

### 예상 질문과 답
| 질문 | 답 |
|---|---|
| "그냥 ChatGPT랑 뭐가 다르죠?" | 가격 데이터를 우리가 만들었다. 출처·촬영일·검수자가 행마다 붙어 있고, 모델이 한 말은 다른 모델이 데이터와 대조한다. §2 표 |
| "정확도가 76%면 낮지 않나요?" | 그래서 사람 검수를 필수로 뒀다. 그리고 76%는 평균이고, 해상도만 확보되면 100%다. 무엇이 정확도를 흔드는지 알아낸 것이 결과다. §8 |
| "데이터가 43곳뿐인데 확장은요?" | 파이프라인은 지역에 의존하지 않는다. 사진과 검수 인력만 있으면 같은 흐름이 돈다. 자연과학캠퍼스가 다음 후보다 |
| "왜 DB를 안 쓰나요?" | 검수가 사람 일이고, 시트가 그 사람의 작업 화면이다. DB를 두면 검수 UI를 따로 만들어야 한다. ADR-0005 |
| "Solar 대신 GPT를 쓰면?" | 구조화 출력이 한 번의 왕복으로 답과 조건을 같이 준다. 그리고 검증용으로 `solar-mini`를 따로 쓸 수 있어 생성/검증 모델 분리가 같은 제품군 안에서 된다 |
