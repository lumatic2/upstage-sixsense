# 예상 Q&A 확정 답변 (M4 step-2)

> PRD "차별점" 확정 문구(2026-07-15 팀 합의) 기반 + 이후 실측으로 보강.

## Q1. 그냥 ChatGPT/Claude에 물어보면 되지 않나? ★승부처

범용 챗봇은 ① 오늘자 학식 메뉴를 모르고 ② 주변 식당의 현재 가격을 모르며 ③ 숫자를 환각한다.
우리는 **1차 소스만** 답에 쓴다 — 학식은 오늘 크롤링한 HTML, 식당 가격은 파싱 후 **사람이 검수·승인한** 데이터.
"가격"은 환각이 허용되지 않는 도메인이라는 것이 서비스 논리의 출발점이다.

## Q2. 거지맵이랑 뭐가 다른가? (정면 대응 — 숨기지 않는다)

표면은 닮았다(예산 필터+대학가 지도). 차이는 **데이터 생산 방식**:
- 거지맵 = 수동 크라우드소싱(사용자가 타이핑) → 우리 = **사진 1장 → Document Parse → 프리필** ($0.01/장, 실측 76.4%).
- 학식 자동 크롤링 통합, 상황 자연어 검색("과선배가 사는 날")도 그쪽엔 없다.
- 리스크로 인지하고 있으며(ADR-0001 수용 리스크 (a)), 발표에서 선제 공개한다.

## Q3. 파싱이 틀리면? 데이터 오염은?

실측으로 답한다: 저해상·손글씨에서 가격 오독 실사례(7.0→17.0)를 확보했다.
그래서 **전량 자동 반영을 하지 않는다** — 파싱은 프리필까지만, 사용자 수정 → pending → 운영자 승인 3단.
"어디까지 자동인가"의 경계를 감이 아니라 **실측 데이터로** 그었다는 것이 이 프로젝트의 처리 깊이다.

## Q4. 왜 학식에는 Document Parse를 안 썼나?

학식은 이미 구조화된 HTML이다(실사 확인). 거기에 문서 파서를 쓰는 건 요건을 채우려는 억지 적용이고,
심사 기준 "목적에 적합한 제품 적용"에 정확히 반하는 일이다. **비정형(사진)=Document Parse, 정형(HTML)=크롤러** — 이 구분 자체가 우리의 Upstage 활용 설계다.

## Q5. Solar가 느리거나 죽으면?

실측 0.7~1.3초, 2.5초 타임아웃 → 기존 정규식 폴백. 검색은 외부 장애와 무관하게 항상 동작한다(`source` 필드로 경로 투명화).
모델 선정도 실측이다: solar-mini는 예산 환각·인원 나눗셈 실패를 보여 solar-pro2를 채택했다.

## Q6. 확장 가능한가?

구조가 이식 단위다: 학식 크롤러(대학별 URL만 교체 — 서울대 동일 구조 확인) + 메뉴판 파싱(지역 무관) + 예산 질의(공통).
대학가 단위 복제가 로드맵이며, 명륜캠은 검증 1호다.
