16장 초안에서 점수 안 나는 칸(소개·규모·마무리)만 병합해 11장으로 줄였다 — ★는 그대로 다 살아 있다.

> **시연 슬롯 주의**: 전체 클릭스루는 약 5분이라 그대로 넣으면 발표 시간을 넘긴다. 발표에서는
> **★ 화면(추천 요약줄 · about 4지점) 위주로 75초 안팎으로 트림**하거나 라이브로 빠르게 지나간다.
> 5분짜리 전체 녹화는 **제출용 시연 영상**으로 따로 낸다(제출물 요건).

## 발표 원칙

- **숫자는 전부 SPEC §8에서 가져온다.** 추정치를 슬라이드에 올리지 않는다. 화면(`/about.html`·랜딩)의 숫자와 슬라이드가 어긋나면 안 된다 — 화면은 자동 갱신이라 항상 최신이다.
- **"안 한 것"을 말한다.** 학식에 파싱을 안 쓴 판단, 순서 개인화를 걷어낸 판단. 도구를 다 쓰는 것보다 **가려 쓴 것**이 심사에서 세다.
- **`/about.html`을 열어 두면 백업이 된다.** 시연이 막히면 그 페이지의 실행 블록이 진짜 API를 그 자리에서 돌린다.
- **한 줄 방어선**: "그냥 ChatGPT에 물어보면 되지 않나?" → *"가격이 웹에 없어서 못 한다. 우리는 메뉴판 사진을 직접 읽어 데이터로 만들었다."*

## 예상 질문과 답 (질의응답 5분)

| 질문 | 답 |
|---|---|
| "그냥 ChatGPT랑 뭐가 다르죠?" | 가격 데이터를 우리가 만들었다. 출처·촬영일·검수자가 행마다 붙어 있고, 모델이 한 말은 다른 모델이 데이터와 대조한다. (SPEC §2 표) |
| "정확도가 76%면 낮지 않나요?" | 그래서 사람 검수를 필수로 뒀다. 76%는 평균이고, 해상도만 확보되면 100%다. 무엇이 정확도를 흔드는지 알아낸 것이 결과다. (§8) |
| "데이터가 50곳뿐인데 확장은요?" | 파이프라인은 지역에 의존하지 않는다. 사진과 검수 인력만 있으면 같은 흐름이 돈다. 자연과학캠퍼스가 다음 후보다 |
| "왜 DB를 안 쓰나요?" | 검수가 사람 일이고, 시트가 그 사람의 작업 화면이다. DB를 두면 검수 UI를 따로 만들어야 한다. (ADR-0005) |
| "Solar 대신 GPT를 쓰면?" | 구조화 출력이 한 번의 왕복으로 답과 조건을 같이 준다. 검증용으로 `solar-mini`를 따로 쓸 수 있어 생성/검증 모델 분리가 같은 제품군 안에서 된다 |
| "근거 검증은 실제로 뭘 걸러냈나요?" | 생성 프롬프트가 근거 없는 표현을 애초에 막아서 대부분 통과한다 — 검증은 **그 통과를 증명하는** 장치다. 프롬프트를 풀면 solar-mini가 맛·평판 문장을 잡아내 데이터 사실로 바꾼다(약식 시연 가능) |
