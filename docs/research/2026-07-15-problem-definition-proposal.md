## 결론 먼저

1. **기존 문제 정의("특정금액 이하 필터링으로 취향 음식점 찾기 어려움")는 폐기를 제안한다.** 거지맵(2026-03 출시, 누적 131만 이용자)이 예산 필터+학식+대학가 지도로 정확히 같은 문제를 이미 전국 단위에서 풀었고, 학식 정보는 에브리타임 "오늘의 학식"이 표준 채널이다.
2. 증언의 무게중심은 "찾기 어려움"이 아니라 **"예산 총량 관리 실패"**였다 — 월말에 끼니를 거르고(E3), 삼각김밥으로 때우고(E5), 벌크 닭가슴살로 회피한다(E3).
3. 내 추천 문제 정의는 **후보 C2 (예산 상태 기반 식사 계획)**. 이유와 반대 증거는 아래.

## 1. 왜 기존 정의를 폐기하나 (충돌 게이트 결과)

| 킬러 | 확인된 사실 |
|---|---|
| 거지맵 | 예산 범위 필터(1,000~8,000원) + 대학 구내식당 포함 + 대학가 지도. 131만 이용자로 시장 검증 완료 |
| 에브리타임 | "오늘의 학식" 메뉴+후기 — 대학생 표준 채널 |
| 학생 사이드프로젝트 | 학식+지도 조합이 서울대·숭실대·연세대·경희대에서 반복 제작됨 — 진입장벽 없음 = 차별화 불가 신호 |

루브릭의 승부처가 "그냥 ChatGPT에 물어보면 되지 않나"인데, 기존 정의는 그 전 단계인 "그냥 거지맵 쓰면 되지 않나"에서 막힌다.

## 2. 리서치가 가리키는 진짜 pain (증거 요약)

- **클러스터 A — 식비 총량 부족** (최강, 9+건): "월말이 다가오면 끼니를 줄이는 식으로 버틴다"(S-N02, E3, 2026-04), 하루 5~6천원 목표 벌크구매(S-N04), 천원 조식 오픈런. 실제 행동 변화 다수.
- **클러스터 B — 학식 품질 회피** (5+건): "맛없어서 삼각김밥으로 떼움"(S-K03).
- 클러스터 C — 탐색 난이도 (3건, 얇음): 기존 가설 지지 증언은 존재하나 소수.
- E1 배경: 서울 외식 대표메뉴 1만원 내외(한국소비자원 2026-03), 외식물가 +2.6%(2026-06, 둔화 국면 — "런치플레이션 심화" 프레임 금지).
- **정량의 한계: "찾기 어렵다"를 지지하는 E1~E2는 0건. "식비가 부담이다"까지만 증명됨.**

## 3. 데이터 실사가 가른 것 (Upstage 사용처)

- 성대 학식 식단은 **이미 깔끔한 HTML 표+가격 텍스트** (직접 클릭 확인) → **학식 파싱에 Document AI를 쓰면 억지**. 크롤링이면 충분.
- 주변 식당 가격은 반대로 **구조화 데이터가 사실상 없음** — 네이버 Local API에 가격 필드 자체가 없음(실행 확인). 가격은 메뉴판 사진 속에 갇혀 있을 가능성(미검증 — 오늘 밤 검증 가능).
- 영수증은 대회 공지가 명시한 예시 문서 유형이고, Document Parse는 이미지·PDF·HWP를 네이티브 지원.

## 5. 추천: C2 — "예산 상태 기반 식사 계획"

**문제 정의 문안(초안):** 식비 예산이 정해진 대학생은 ① 지출은 영수증·결제내역에, ② 오늘의 선택지는 학식 페이지·메뉴판 사진·커뮤니티에 흩어져 있어, "남은 예산으로 이번 주를 어떻게 먹을지"를 계산해주는 도구가 없다. 그 결과 월말에 끼니를 거르거나 편의점으로 때우는 예산 실패가 반복된다.

**왜 이걸 추천하나:**
- **증거가 가장 강한 클러스터 위에 서 있다** (A: 총량 부족, 행동 변화 증거 다수).
- **거지맵과 정면 충돌하지 않는다**: 거지맵은 "싼 곳이 어디냐"(탐색·무상태), C2는 "내 잔여 예산으로 뭘 먹을 수 있냐"(개인 상태·누적 최적화). 관점이 다르다.
- **"ChatGPT에 물어보면 되지 않나"에 구조적으로 답한다**: LLM은 상태가 없다 — 내 지출 누적, 오늘자 학식·가격을 모른다. 이건 프롬프트로 못 메우는 격차다.
- **Upstage 사용이 자연스럽다**: 영수증(대회 공지 명시 예시) 파싱 = Document Parse, 예산 계획·추천 설명 = Solar. 루브릭 "목적에 적합한 제품 적용 10점"에 정합.
- **파이프라인 서사(35점 구간)가 선다**: 영수증 사진 → 파싱 → 지출 상태 / 학식 크롤링 + 메뉴판 파싱 → 선택지 → 잔여 예산 제약 하 추천. "어떤 정보를 어디서 왜 가져와 어떻게 가공하는가"가 명확.

**반대 증거·리스크 (합의 때 반드시 논의):**
1. **가계부 앱(뱅크샐러드·토스 등)이 지출 추적을 이미 한다** — 단 "식사 선택지와 연결한 계획"은 안 하는 것으로 보이나, 이 충돌 검사는 아직 안 했다(오늘 밤 30분 검증 필요).
2. **행동 비용**: 학생이 영수증을 매번 찍을까? dogfood 전까지 미검증. (완화: 하루 예산만 입력하는 라이트 모드 병행)
3. "찾기 어려움" 증언(S-N01)을 버리는 결정 — C 클러스터가 얇다는 판단이 틀렸다면 회귀 필요.
4. 메뉴판 사진 파싱 가설(C3 구간) 미검증 — 실패하면 주변 식당 선택지는 착한가격업소 CSV+학식만으로 축소 운영.
