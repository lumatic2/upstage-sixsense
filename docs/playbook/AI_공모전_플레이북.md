# AI 공모전 플레이북 — 범용 실행 루프 (접수→제출→회고)

> 작성일: 2026-07-12 · 최종 개정 2026-07-17 · 상태: **v2.20** · 최근 변경: §2-F 용어집 신설(v2.16 이월 finding — 플래닝/스캐폴딩/아키텍처 등 10개 용어) + assets 템플릿 2종 고유명사 익명화. **전체 이력 → 문서 끝 §변경 이력.**
> 목적: "어떤 AI 공모전이든" 접수→제출→회고까지 도는 재사용 절차. 실전에서 작동한 것 / 부족했던 것 / 차기에 바꿀 것을 한 문서로 고정한다.
> 유래: 실전 1개 대회(멀티 슬롯 예선, 3개 제출)의 작업 로그·자체 채점·필드 표본에서 귀납. 대회 특화 세부는 익명화해 "한 실전 사례"로 본문에 유지하고, 고유명사는 싣지 않되 재검증용 상대경로는 §5에 남긴다.
> 이식: 본문의 개인 경로·제품명(`INVENTORY.md`·askewly design system·`ncli`·`shelf`·`last30days`·`insane-search`·Codex 등)은 **예시 인스턴스**다 — `§2-D 도구·자산 바인딩 표`로 네 환경에 매핑한다(자산이 없으면 표의 "최소 스키마"로 대체 생성). 3자 공유용 파생본은 이 표 기준으로 개인 예시를 치환해 뽑는다(평행 복사본을 손으로 유지하지 않는다).

**무게중심 원칙 (garbage in, garbage out).** 구현(Phase 5)은 병목이 아니다. 승부는 ① 전반부의 리서치·스캐폴딩(Phase 1~4 — 잘못 정의된 문제는 뒤에서 못 고친다)과 ② 최종부의 사람 직접 QC/QA(Phase 6~7 — 마지막에 잘 하는 것)에서 갈린다. 시간 배분도 이 곡선을 따른다: 계획·기준 수립에 노력의 70~80%(알렉스 방법론), 구현은 위임·병렬화로 압축.

## 실행 순서 (한눈에)

> **Phase 번호 = 진행 순서.** 슬롯이 여럿이면 [슬롯 루프]를 슬롯마다 반복한다. 오른쪽 **[머리]는 그 Phase를 어떤 관점으로 접근하나**(문제 / 제작 / 쓰는 사람 / 패키징 / 심사자) — 관점이 바뀌면 세션을 나누지만 "1 Phase = 1 세션"은 강제가 아니다. 규모별 압축은 §2의 S/M/L 라우팅. (실제로 세션을 어떻게 여닫는지는 아래 §실전 운용법.)

```text
0  Phase 0   참가 결정 — 어느 대회에 나갈까 (최대 레버)
1  Phase 1   심사 역설계·루브릭 확정 (착수 전, 슬롯 무관 1회)

── 슬롯 루프 (슬롯마다 반복) ──
2  Phase 2   문제 발굴·확정            [머리: 문제]
3  Phase 3   해법 결정·범위 고정         [머리: 문제]
4  Phase 4   작업 설계·킥오프           [머리: 문제→제작 전환]
5  Phase 5   구현·검증 루프            [머리: 제작]
6  Phase 6   사람 직접 QC·dogfood      [머리: 쓰는 사람]
7  Phase 7   패키징·정합              [머리: 패키징]
8  Phase 8   제출 전 리허설 채점         [머리: 심사자]
── 슬롯 루프 끝 ──

9  Phase 9   제출 후 회고 → 플레이북 역주입
```

부록: §2-A 적대검증 · §2-B 소스 인벤토리 · §2-C 팀전 · §2-D 도구·자산 바인딩 · §2-E 게이트 골격 · §2-F 용어집 · §3 체크리스트 · §4 회고 틀 · §5 근거

## 실전 운용법 (이 문서를 대회에서 어떻게 굴리나)

> 플레이북 = 방법론(왜·무엇을). 실전에선 아래 순서로 쓴다. 핵심은 **플레이북을 로그 세션에 붙여넣는 게 아니라, 플레이북으로 짠 계약을 붙여넣는 것**.

**로그가 채점 표면인 대회** (작업 로그가 원본 그대로 제출됨) — **3층 로그 전략** (이 패러다임은 실전 2개 중 1개에만 적용됐다 — 산출물만 심사하는 대회면 이 블록 전체를 건너뛴다):
- **①층 사전 자산화(평시)**: 게이트 골격(§2-E)·`kickoff-template`·적대검증(§2-A)·체크리스트를 대회 무관 평시에 정비. 대회 중 로그엔 "자산을 이 대회에 인스턴스화·특화"하는 발화만 남고, 이건 감점이 아니라 준비된 하네스 엔지니어링의 증거로 읽힌다.
- **②층 스캐폴딩을 작업 세션 안에서 발화로**: 요구·게이트 설계·평가 축·범위 확정을 작업 세션 첫 구간에서 수행 = 계획 발화 자체가 채점 신호(노력의 70~80%가 여기 들어가고, 그게 로그에 보여야 한다). 구 방식(로그 밖 외부 세션에서 길게 설계)은 ① 가장 공들인 사고가 심사에 안 보이고 ② 작업 세션이 "지시받은 대로만 움직인 것"처럼 읽혔다 — 실전 감점 사유(한 슬롯 자체 심사 84/100).
- **③층 외부 채널은 위생 필터로만**: 로그에 남으면 안 되는 발화(심사 최적화·확률 추정·타 대회 언급)만 외부에서 — 외부 채널은 "전략을 짜는 곳"이 아니라 "로그에 못 넣는 것을 거르는 곳"이다. ⚠ **플레이북을 로그 세션에 통째로 붙여넣지 않는다** — 심사 역설계·편향 내용이 로그에 박히면 "심사 게이밍"으로 읽히는 자책골 + 노이즈.
- **운용 순서**: 전략 세션 산출물 = 채운 `kickoff-template`(중립 계약) + 게이트 계획(§2-E) + 빈 `flow-map` → 로그(작업) 세션 첫 메시지엔 **kickoff만** 붙여넣고 그 프레임 아래 Phase 2~7 실행, `flow-map`은 매 세션 종료 시 갱신.
- **개입 문장 레퍼토리** (자연스러운 상황에서만 — 사람 개입이 로그에 보이게): 근거 실물 확인("그 JD 확인한거야? 웹으로 띄워줘")·E2E 완료 기준·AI 제안 기각+이유·독립 검증자 투입·평가 기준 업데이트 지시·빌더/리뷰어 역할 분리·산으로 가기 차단.

**로그가 채점 대상이 아닌 대회** (산출물만 심사 — 대다수): 위 3층 블록 전체 생략. 플레이북을 그대로 참조해 진행하고, 규모가 작으면 §2의 S/M/L로 압축한다.

**자산 (평시 정비, 로그 대회 전용)**: `assets/kickoff-template.md`(붙여넣기용 계약) · `assets/flow-map-template.md`(9섹션 상태 장부) · 게이트 골격은 §2-E. 산출물만 심사하는 대회에선 이 템플릿 2종을 쓰지 않아도 되고, 그때 유효한 건 S/M/L 라우팅과 세션 분리 선택뿐이다.

## 2. 절차

각 단계에 [결정론적]=체크만 하면 되는 것 / [판단]=사람 소유 결정을 표시한다. **[판단]은 두 종류**: 대개 **HITL**(에이전트가 초안 → 사람이 고르고 승인; 예: Phase 2-3 문제 발산)이고, 일부는 **사람 전결**(위임 불가, 사람이 직접 결단) — 전결 노드 = **Phase 0 go/no-go · Phase 1-9 차별화 축 · Phase 3-4 scope contract · Phase 6 최종 taste/QC**. 나머지 [판단]은 HITL 기본.

**규모/시간 라우팅 (S/M/L)** — 이 축은 "무슨 대회냐"(Phase 0 유형)와 **직교**하는 "며칠이냐"다. 전 루프는 1주+ 기준이니 짧으면 압축한다 (gstack priority hierarchy 차용):

| 트랙 | 상황 | 압축 방식 |
|---|---|---|
| **L (1주+)** | 제출형·기획·사업계획 | 본편 전체 |
| **M (48h 전후)** | 주말 해커톤 | Phase 2 렌즈 3→2·발산 7→5 / Phase 4-6 4층→커버리지+프리모템만 / Phase 7·Phase 8 유지 |
| **S (무박 24h)** | 현장 해커톤 | Phase 1·Phase 2 **대회 전 사전수행** / Phase 2 렌즈 1회·발산 즉석 / **Phase 4-6 드라이런 생략(커버리지만)** / Phase 5 직행 / Phase 7→발표 리허설 / Phase 8 demo gods 필수 |

**어떤 트랙이든 절대 스킵 불가**: Phase 0 go/no-go · Phase 1-1 규정 대조 · Phase 3 Measurement Plan · Phase 7 클린 재현 · §2-A IRON LAW. **먼저 버릴 것**: 시각화 파생물 · 문서 윤문 · 중간 게이트 이중 리뷰(적대만, 독립 생략).

### Phase 0. 대회 유형 분류·참가 결정 (대회 후보 발견 시 — Phase 1보다 먼저) ★최대 레버

> 가장 큰 승부 레버는 대회 안의 실행이 아니라 **어느 대회에 나가느냐**다. 이길 수 없는 대회에 풀타임을 태우는 것이 최대 비용.

1. [결정론적] **유형 분류** — 유형이 승부처와 플레이북 변형을 결정한다:

   | 유형 | 승부처 | 이 플레이북의 변형 |
   |---|---|---|
   | 제출형 해커톤 (산출물+문항, AI/사람 심사) | 문제 정의 + 증거 구조 | **본편 그대로** (이 플레이북의 원형 유형) |
   | 오프라인 해커톤 (시간 제한·현장 발표) | 범위 축소 + 발표·데모 | Phase 2 압축(사전 리서치), Phase 7→발표 리허설로 대체, 팀전이면 §2-C |
   | 데이터 경진 (DACON/Kaggle 리더보드) | 정량 점수 + 일반화 | Phase 1 심사 역설계 **스킵**(리더보드가 심사), 루브릭→metric·CV 전략, Phase 8→제출 앙상블/오버핏 점검. 단 보고서 심사 병행 대회면 본편 부분 적용 |
   | 아이디어·기획 공모전 (문서·발표, 사람 심사) | 서사 + 심사위원 공감 | Phase 1-3 심사자 모델링 = **심사위원 프로필·과거 심사평 리서치**, Phase 5 구현→목업·스토리보드, 시각 자료 비중↑ |
   | 사업계획 공모전 (PSST·시장성) | 시장 근거 + 실현 가능성 | Phase 2에 TAM/SAM/SOM·BM 검증 추가(market-researcher 에이전트), 재무 추정은 E1~E2 근거. **`/business-docs` 스킬 활성화**(BMC·PSST 1:1:1·운영기관별 평가지표 매칭·자가채점+AI 비평가 = Phase 1 루브릭·Phase 8 리허설의 기성품) + `office-hours` 포싱 질문으로 수요 검증 |
   | 콘텐츠 공모전 (영상·디자인·글) | 심미·감정 + 주제 적합 | 루브릭 역설계 약화(심미는 역설계 한계), 수상작 스타일 분석 + 대중 투표면 배포 전략 추가 |

2. [판단] **go/no-go 게이트** — 기대값 vs 투입 비용:
   - **기대값** = (상금 + 커리어·포트폴리오 가치 + **자산화 가치**(이 대회로 만들 재사용 엔진·데이터)) × 추정 승률
   - **승률 프록시** 3개: 자기 역량 3축 매칭(Phase 1-7 뷰) / 예상 경쟁 밀도(참가 규모·문턱 낮을수록 밀도↑) / 유형 적합성(내 강점 유형인가 — 위 표)
   - **비용** = 마감까지 실투입 시간 + 기회비용(동시 진행 중인 것)
   - **하드 no-go**: 역량 매칭 0 / 저작권 독소조항(출품작 권리 전부 주최 귀속 등) / 규정상 내 방식(AI 활용 등) 금지 / demo·증거를 만들 수 없는 유형
   - 판정과 근거를 한 줄이라도 기록 — 발표 후 회고(Phase 9)에서 go/no-go 판단 자체를 채점한다.
   - ※ 연동: 대회 수집 자동화(openclaw cron, 일정 자동 추가)가 이 게이트 필드(유형·마감·상금·참가 자격·저작권 조항)를 수집 시점에 미리 채우도록 수정 — 사람은 판정만.

### Phase 1. 심사 역설계 + 루브릭 확정 (착수 전, 슬롯 무관 1회) ★비용 곡선의 지렛대

> 심사 기준을 후반(제출 직전 자기 심사)에야 세우면 수정 비용이 최대인 시점에 감점이 발견된다. **루브릭을 Phase 1에서 확정하고 이후 모든 게이트가 그 루브릭 대비 채점**하면 같은 발견이 수정 비용 최소 시점으로 당겨진다. (한 실전 사례: 심사 시뮬레이션을 제출 후에 해서, "루프를 제출 전으로 옮겼으면 50점 근접 가능"이 회고 결론이었다.)

1. [결정론적] 공고·안내·FAQ·공지 원문을 전부 수집하고 제출물 형식·증거 규정·실격 조건·타임라인을 표로 고정한다. 규정의 각 조항에 "이것이 금지하는 것 / 요구하는 것 / 침묵하는 것"을 단다(침묵 구간이 전략 여지).
2. [판단] **심사 루브릭 작성**: 공개 제출 요구(문항, 필수 산출물)를 문항×배점 루브릭으로 번역하고, 항목마다 **"규정 내에서 도달 가능한 10점"을 정의**한다. 구조 옵션: 문항×만점 대신 **가중 감점형**(카테고리 100점 시작, 심각도별 감점 Critical -25 / High -15 / Medium -8 / Low -3, gstack qa) — "왜 이 점수인가"의 설명력이 좋아 자기 채점·리허설에 유리. 규정상 불가능한 기준(예: 공개정보만 유효한 대회에서 현직자 인터뷰·실측)을 천장으로 잡는 오류 금지 — 실전에서 실제로 범했다가 철회한 오류다. 각 항목의 예상 감점 유형을 "제출물 고유(고칠 수 있음) vs 전 참가자 공통 천장(무시)"으로 사전 분류한다.
   - **루브릭 캘리브레이션 게이트** ★자체 루브릭을 순위·확률에 쓰기 전 필수 (실전 반증 — 근거 40): 루브릭을 **진단**("무엇이 약한가")에 쓰는 것과 **순위·확률 추정**("어느 슬롯이 더 유리한가", "통과 확률 N%")에 쓰는 것을 구분한다. 진단은 검증 없이도 유용하지만, 순위·확률에는 **분리력(discrimination) 검증**이 필요하다. 절차 — **Phase 1-8①의 수집 대상을 라벨된 표본으로 격상해** 재사용한다: ①은 "이전 기수 수상작·공개 제출물 분석(어떤 방향이 이겼나)"까지만 요구하므로 **탈락작·라벨 정합은 보장하지 않는다** — 이 게이트가 ①에 추가하는 것이 곧 **라벨 요구**(각 표본에 수상/탈락 또는 진출/비진출 라벨을 붙여 수집)다. 그렇게 만든 라벨 표본에 이 루브릭을 그대로 먹여 수상 그룹이 탈락 그룹보다 실제로 높게 나오는지 확인한다(각 그룹 3건 이상 목표·표본 편향 명시). 추가 비용은 "①을 어차피 하는 김에 라벨을 함께 기록"하는 수준이지 새 리서치가 아니다 — 단 **탈락작이 공개되지 않는 대회가 많다**는 점이 이 게이트의 실제 제약이며, 그때는 아래 "라벨 없음" 경로로 닫는다. **분리하지 못하면 그 루브릭은 순위 판단·확률 추정에 사용 금지** — 진단 체크리스트로만 쓰고 Phase 9-1의 확률 추정도 만들지 않는다. 라벨 데이터가 없는 신설 대회면 "캘리브레이션 불가"를 명시하고 확률 추정을 **생략**한다(추정을 안 하는 것이 틀린 추정보다 낫다 — 틀린 확률은 자원 배분을 실제로 왜곡한다) — **단 완전 무신호로 두지 말고 §2-A "프로바이더 독립" 패널을 프록시 외부 신호로 1회 돌린다**(공식 라벨은 없어도 자기평가보다 실제 결과에 가깝다, 근거 42). 확률 숫자를 만들진 않되 "독립 패널 대비 우리 상대 위치"는 남길 수 있다. 이 게이트는 Phase 9-1의 *"발표 결과와 대조해 루브릭 자체를 채점"* 을 **착수 전으로 당긴 것**이다 — 이 Phase가 루브릭에 적용하는 shift-left 논리를 루브릭 자신에게도 적용한다. **실전 실패**: 이 게이트 없이 만든 자체 루브릭이 최상위 두 슬롯을 못 갈랐고(동점), 서로 다른 축으로 만든 두 번째 자체 루브릭도 같은 슬롯을 같은 방향으로 오판했다 — 실제로 가장 선전한 슬롯을 두 장치 모두 과소평가(단일 루브릭 노이즈가 아니라 계통 오차, 근거 40).
3. [판단] **심사자 능력·성향 모델링**: 루브릭이 "무엇을 채점하나"라면 이것은 "심사자가 무엇을 *할 수 있고 무엇에 반응하나*"의 가설 표다. 심사 주체가 AI냐 사람이냐로 갈린다(하이브리드: 1차 AI·본선 사람이면 단계별로 둘 다).

   **(A) AI 심사자 — 능력 가설 표**: 최소 이 4행 — 스크립트 재실행을 하는가 / 어떤 소스에 접근 가능한가(네이버·로그인 벽) / zip을 얼마나 깊이 여는가 / 컨텍스트 한계로 긴 문서를 다 읽는가. 각 가설에 **참일 때·거짓일 때 유불리 + 양쪽에 통하는 헤지**를 단다(예: 재실행을 하든 안 하든 유리하려면 → 재실행 생존 + 서술로도 읽히는 examples). 발표 후 실측으로 표를 채워 차기 대회 입력으로.

   **(B) 사람 심사자 — 성향 모델링**: 사람은 루브릭이 아니라 **편향으로** 채점한다(루브릭 방어는 기본기일 뿐). 두 갈래:
   - **B-1. 알려진 심사위원 역설계 (기록 있으면)** — 발표 준비의 0단계: ① **공개 루브릭부터 확인**(주최가 이미 공개했을 수 있다 — MLH·TAIKAI 등 GitHub/블로그) ② 심사위원 배경·성향 리서치(LinkedIn·과거 강연·소속사 가치관·과거 심사평, 가능하면 사전 대화) ③ 같은 대회 과거 수상작 분석(무엇이 이겼나 — 단 **솔루션 재활용 제출은 감점 신호**) ④ 심사위원 도메인에 피치 이원화(기술 심사=아키텍처 난이도, 투자·임팩트 심사=시장·서사).
   - **B-2. 기록 없으면 = 일반 방법론 (근거 기반, 근거 32~38)**:
     - **준비도 > 열정**: 열정은 "준비돼 보임"을 통해서만 점수화된다(Chen 2009) — 리허설이 감정 표현에 우선.
     - **문제 프레이밍 → 기술 순서 고정**: 심사자가 좌절을 공유해야 이후 설명이 먹힌다. 기술 스택 나열로 시작 금지.
     - **하나의 완결 "와우" 플로우 > 여러 반쯤 기능**: peak-end 법칙상 절정을 **끝 직전**에 배치.
     - **첫 5~15초 + 마지막 문장에 리소스 불균형 투입**: thin-slicing — 초 단위 첫인상이 최종 평가와 강상관(Ambady).
     - **새로움은 "설명 가능한 새로움"으로**: 루브릭은 창의성에 가점하나 사람은 불확실성 회피로 낯선 걸 벌점(Mueller 2012) — 비유·기존 카테고리 연결로 포장.
     - **데모 실패 안전장치 필수**: 라이브 데모 붕괴는 halo를 역방향으로 전이(Phase 8-3 demo gods와 정합).
   - **B-3. 편향 활용/방어** (통제 밖이나 대응 가능): 발표 **순서** = 마지막이 유리·중간 회피(recency 실증, Bruine de Bruin 2005), 못 고르면 첫/끝에 집중 / **심사 피로** = 늦은 슬롯이면 더 짧고 명료하게(판사 결정 피로 유추 — 출처 약함) / **halo·유사성** = 첫인상 신호 하나(팀 소개·데모 안도감)에 투자 + 심사위원과의 접점(도메인·관심사)을 초반 노출.
   - 발표·사람 심사 대회면 (B)를 **Phase 8 리허설 스크립트 + Phase 6 taste 체크의 기준**으로 쓴다. AI 심사 대회는 (A)만.
4. [결정론적] **scoring surface 설계**: 심사자(AI든 사람이든)가 실제로 읽을 표면(답변 문항, README, manifest, examples, 검증 matrix, logs)을 나열하고 각 표면이 줄 신호를 루브릭 항목에 매핑한다. 핵심 원칙: **중요 fact는 한 곳에 두지 않고 README 요약→답변 재료→증거 경로→matrix 행으로 반복 연결**. "로그에만 있는 근거"는 심사 표면에서 없는 것과 같다. **AI 심사 친화 서식**(Phase 1-3 "긴 문서 다 못 읽음" 가설의 헤지): 각 표면 최상단에 **TL;DR**(컨텍스트 잘림 방어) · 핵심 주장 앞에 명시 마커(`[핵심 성과]`·`[해결 문제]`) · 긴 산문보다 **표·불릿** 우선. 단 키워드 마커는 요건이 아니라 전술 — 사람 심사자에겐 과하지 않게 조절.
   - **surface > depth 프레이밍 주의** ★가설(심사 피드백 0줄 — 검증 필요): 심사AI가 재실행·claim 대조를 **깊이 하지 않고 표면만 읽을** 가능성이 있다. 그렇다면 "60/60 성공" 같은 표면상 완결된 주장이 "40/60인데 우리가 잡아냄 — 정직한 실패"를 **이긴다** — 우리 플레이북의 정직성·재실행 생존 차별점(§2-A·Phase 5-6)이 *심사가 재실행한다는 전제*에 걸려 있음을 자각한다. **대응(정직성 라인은 지킨다)**: 거짓 60/60을 주장하지 않는다(심사가 진짜 재실행하면 자멸 — 필드 최다 고장 모드). 대신 **정직한 결과를 표면 읽기에도 이기게 프레이밍** — 성과를 맨 앞 TL;DR·peak-end에 배치, 정직한 한계는 "고백"이 아니라 **"이만큼 엄밀히 봤다"는 rigor의 증거**로 배치, 있는 걸 파묻지 않기. **테스트**: §2-A "프로바이더 독립" 패널도 표면을 읽으니, 정직 프레이밍이 패널에서 점수가 깎이면 그게 재프레이밍 신호다.
5. [결정론적] **claim graph 계약**: 모든 핵심 주장은 `공개 URL → source_id → 문제 후보 → fixture → 산출물 경로 → 검증 행 → 답변 claim` 경로를 타야 하며, 못 타는 claim은 supported / internal_only / unsupported / out_of_scope / stale 로 분류해 unsupported는 답변에서 제거한다.
6. [판단] 출제자 의도: 주최측 영상·인터뷰·공지의 강조 역량을 슬롯별로 정리한다. 인터뷰에서 직접 언급된 문제는 본선용/미끼일 수 있다 — **인터뷰에 없는 독자 문제로 독창성 확보**를 항상 검토.
7. [결정론적] **자기 역량·자산 인벤토리 대조**: 정본(`~/projects/cover-letter/docs/me/INVENTORY.md` — 도메인 지식·스킬·재사용 가능 프로젝트 자산이 STAR 단위로 관리됨)에서 이 대회 관점의 3축 뷰를 뽑는다 — ① 도메인 지식(예: 회계·세무·재무) ② 도구·모델 접근성(오케스트레이터·하네스·MCP) ③ 재사용 자산(기존 엔진·벤치마크·인프라). 목적: "풀어야 할 문제"가 아니라 **"내가 이길 수 있는 문제"**를 고르는 입력.
8. [판단] **슬롯 선택·경쟁 환경 분석** (멀티 슬롯 대회면 승부의 절반): ① 이전 기수 수상작·공개 제출물 분석(어떤 방향이 이겼나) ② 참가 규모 → 슬롯당 경쟁 밀도 추정(범용 스택 슬롯 = 격전지, 도메인 장벽 슬롯 = 밀도 낮음) ③ **도메인 장벽 높은 슬롯 × Phase 1-7(자기 역량 인벤토리)의 내 전문성 매칭 = problem-owner 카드** — 범용 스택 슬롯은 표본이 밀집한 격전지라는 것은 필드 관측으로 유지된다(한 실전 사례: 공개 표본 9건 중 5건이 그 슬롯에 몰림). **단 "도메인 장벽 슬롯에서 마진이 최대"라는 실전 근거는 철회됐다**(v2.15, 근거 39): 그 "마진"은 자기심사로 상향된 점수에서 나온 값이었고, 실제 결과는 **반대로** 났다 — 도메인 장벽 슬롯이 탈락하고 격전지로 분류한 슬롯이 유일하게 컷에 닿았다. 원칙(내 전문성과 매칭되는 슬롯을 고른다)은 a priori 근거로 유지하되, **"장벽 슬롯이 곧 안전 마진"이라는 추론은 쓰지 않는다** — 경쟁 밀도가 낮다는 것과 내가 그 슬롯에서 이긴다는 것은 별개이고, 밀도 낮은 슬롯은 심사 정원도 같이 낮을 수 있다. ④ 멀티 제출 시 방법론 다양화 헤지(§4-6)도 여기서 결정. ⑤ **실무경험·리소스 레버리지 축** ★유망(1대회 실제 심사위원 구두 피드백[강] + 1대회 결과 정합[자체 해석이라 교차 근거로 약함] — 근거 41a·KG `early-stage-judging-rewards-demand-evidence`): 얼리스테이지 심사는 기술 완성도·체계성보다 **수요 증거의 양 + 문제 정의의 실무 접지 + 보유 리소스(인맥·경험) 활용**을 보상한다. 공개자료 5방향 리서치로 재구성한 문제에는 **상한**이 있다(현직자만 아는 병목·인맥으로만 얻는 검증은 리서치로 못 넘는다 — 한 실전 사례는 심사위원이 직접 진술했고, 다른 사례는 결과·Q1 갭이 같은 벽을 지목했으나 후자는 자체 해석이다). → 문제 후보 발산 시 **팀·개인이 실제로 겪은 불편을 1순위 후보로**, 그리고 **설문을 발표 재료가 아니라 문제 선택의 입력으로 앞당긴다**(수상팀 공통 관측).
9. [판단] **차별화 축 명시 결정** ★우승과 통과의 분기점: 루브릭 방어(감점 없음)는 기본기일 뿐 — 상위권은 동점 클러스터를 이룬다(실전 관측: 상위권 점수 37~42 밀집). 심사자 기억에 남을 "한 방"을 어느 축에 걸지 **여기서 결정하고 이후 전 단계(Phase 2 문제 선택 → Phase 5 output 설계 → Phase 7 표면 배치)가 그 축에 투자**한다. 축 메뉴: 문제 선택의 독창성(니치) / 검증 깊이(재실행 생존·독립 라벨) / **비주얼·인터랙티브 데모**(만져지는 output은 서술을 이긴다) / 도메인 전문성(problem-owner) / 완성도·속도. **기본 전략(사용자 자산 기반): 니치 문제 독창성 × 인터랙티브 데모의 이중 축** — 전자는 Phase 2 발산-수렴에서, 후자는 askewly design system(직접 설계한 프론트엔드 자산)으로 Phase 5에서 실현. 단일 축은 심사 성향에 취약하므로 이중 축이 헤지도 겸한다.
10. [결정론적] 도구·자산 점검: 리서치 도구 가용성(Phase 2 표), 로그 수집기가 전 과정(서브에이전트 포함)을 실제로 잡는지 검증, 사전 자산(킥오프 템플릿·게이트 skeleton) 인스턴스화. 새 도메인이면 `shelf recall`로 기존 도구 셸프 조회부터.

### Phase 2. 문제 발굴·확정 (슬롯당, 작업 세션 안에서)

> 구명 "리서치·문제 정의". 개명 이유(v2.17): 리서치는 **활동**이라 Phase 3~5에서도 계속된다 — Phase 이름은 활동이 아니라 **출구에 남는 산출물**(확정된 문제 정의)로 짓는다.

> ⚠ 실전 최대 사후 발견: **최초 문제 발견·적대검증이 작업 로그 밖(전략 채널)에서 이뤄지면 심사가 "AI-Native 자율성 낮음"으로 읽는다** (한 슬롯 자체 심사 84/100의 감점 사유였고, 이후 설계를 수정했다). 리서치는 반드시 로그가 남는 작업 세션 안에서 한다.

1. [결정론적] 공개 소스맵 작성. **채널별 도구를 의도적으로 고른다** — 남들과 같은 검색은 같은 근거를 낳는다. 특히 **"고품질인데 일반 AI 검색이 못 긷는" 소스**(§2-B 인벤토리)를 우선 훑는다 — 무료 공식 API가 있는 것부터(Open DART·KOSIS·ECOS·law.go.kr·OpenAlex·KIPRIS·열린국회):

   | 용도 | 도구 | 비고 |
   |---|---|---|
   | 한국 커뮤니티·블로그·지식iN (일반 에이전트가 접근 못 하는 데이터) | `ncli` (Naver Open API: blog/news/kin/local/shop) | 차별적 근거 소스이자 양날 — 심사AI도 접근 못 하면 검증불가 처리될 수 있으므로 **반드시 접근성 A급 출처와 짝**(아래 7) |
   | 기업 재무·공시 정량 근거 | Open DART API (`OpenDartReader`) | 경쟁자는 웹 UI를 눈으로 읽는다 — API 시계열은 차별 폭 최대 (§2-B top7 #1) |
   | 거시·산업 공식 통계 | KOSIS API · ECOS API · data.go.kr | "카더라"를 공식 수치로 교체 |
   | 법령·판례·규제 제약 | law.go.kr API (`law-mcp` 연결됨) | 법적 실현가능성 근거 — 대부분의 팀이 스킵 |
   | 학술·기술 선행연구 | OpenAlex/Semantic Scholar API + arXiv | Google Scholar 봇 차단의 완전 무료 대체 |
   | 특허·상표 선점 확인 | KIPRIS Plus API | 저비용 고효과 차별성 검증 |
   | 최신 멀티플랫폼 동향·반응 | `last30days` 스킬 | Reddit/X/HN/YouTube 등을 engagement로 랭킹한 단일 브리프 |
   | 접근이 까다로운 공개 페이지 | `insane-search` 플러그인 | feeds·플랫폼 API·TLS 우회 단계적 판독 |
   | 일반 웹·공식 자료 | WebSearch/WebFetch, `gh` CLI(공개 레포) | 기본기 |

1-b. [판단] 수집한 지식을 **3층으로 구분** (gstack ETHOS "Search Before Building"): **L1** 검증된 관행(그대로 쓴다, 재발명 금지) / **L2** 신흥 트렌드(과열 주의 — 입력일 뿐 결론 아님) / **L3** 제1원리 발견(**"관행이 왜 여기서는 틀렸는가"** — 가장 가치 있음). L3가 나오면 명명하고 기록 — 이것이 차별화 축(Phase 1-9)의 원재료다.
2. [결정론적] **evidence level 태깅**: 각 소스에 source_id + 접근일 + **발행일** + **E-level** + **심사자 접근성(A/B)** 을 단다.

   | 등급 | 정의 | 예 | 허용 용도 |
   |---|---|---|---|
   | **E1** | 감사·규제·법정 데이터 | 감사받은 공시(DART/EDGAR), 규제기관 통계(KOSIS/ECOS), 법령·판례 | 모든 claim. **규모·재무·법적 claim은 E1~E2만 인정** |
   | **E2** | 피어리뷰·공식 1차 발표 | 학술 논문(KCI/OpenAlex), 정부·국책연 보고서, 회사 공식 발표·IR·채용공고 원문, 특허 | 사실·기술 타당성 claim |
   | **E3** | 검증 가능한 2차 | 복수 매체 교차 확인된 언론 보도, 증권사·컨설팅 리포트 | 해석·전망 claim (단독 매체는 E4로 강등) |
   | **E4** | 전문가 개인 | 전문가 블로그, 컨퍼런스 발표, 인터뷰 영상 | 맥락·출제 의도 파악. 답변 단독 인용 금지 |
   | **E5** | 익명·커뮤니티 | 지식iN, 앱리뷰, 커뮤니티, 개인 블로그 | **문제 존재 증명 전용** — 규모 주장 금지, E1~E2 짝 필수 |

   운용 규칙: ① claim 유형이 요구하는 최소 등급을 넘는 근거만 답변에 인용(정량→E1~E2, 존재→E5 가능하되 짝, 기술→E2 이상, 법적→E1) ② E-level과 심사자 접근성은 **독립 축** — E1이라도 심사AI가 접근 못 하는 형태(예: HWP 첨부)면 접근 가능한 표면으로 옮겨 담는다. **접근성 B 게이트** ★가설(근거 44 — 심사 피드백 0줄): 심사AI 채점 대회에서 접근성 B 출처(로그인 벽·비공개 API 수집물 등 심사자가 재확인 못 하는 것)는 **핵심 claim의 단독 근거 금지** — A급 짝이 없으면 그 claim을 낮은 등급으로 축소하거나 제거한다. "독창적 소스"라는 이유로 이 게이트를 오버라이드하지 않는다(차별화 가치보다 검증불가 처리 리스크가 클 수 있다) ③ **stale과 결합**: 발행일이 접근일 기준 1년 초과면 등급 무관 stale 태그 → 최신 자료로 유효성 재확인 전 핵심 근거 금지, 조직 내부 사실(제품·조직·채용공고)은 6개월로 강화. stale 근거로 세운 문제 정의는 문제 자체가 소멸했을 수 있다.
3. [판단] **문제 후보 발산 (수렴과 별도 pass — 평가 금지)**: 발산 중 후보를 평가하지 않는다 — 평가가 개입하는 순간 니치 후보가 목록에 오르기 전에 죽는다(defer judgment). 원시 후보 **≥7**을 목표로 채굴 경로 체크리스트를 전부 돈다:
   - 리뷰·커뮤니티 불만 마이닝 (`ncli`·앱리뷰 — 불만 1건이 나오면 그 배후 워크플로우를 추적해 문제로 재구성)
   - 채용공고 — 무엇을 뽑나 = 무엇이 아픈가 (실전에서 실증된 채굴 경로)
   - IR·공시의 리스크 섹션, 실적발표 Q&A (Open DART)
   - 규제 변화 — 새 법령·시행령이 만드는 신규 의무 (law.go.kr)
   - 공식 FAQ·고객센터 문서 — 반복 질문 = 반복 마찰
   - 업무 프로세스 맵을 그려 병목 추정 (실무자 렌즈 선행 적용)
   - 유추 이식 — 타 산업에서 이미 풀린 문제가 이 도메인에선 미해결인 것
4. [판단] **스코어카드 수렴 (게이트 → 가중합 → tie-breaker)**: ① **게이트 축 먼저** — 규정 적합·기존 제품 무충돌·demo 가능성은 가중합에 섞지 않고 통과/탈락로 거른다(가중합에 넣으면 "독창성 만점이 규정 위반을 상쇄"하는 왜곡). ② 생존 후보만 **가중합** — 평가 축은 Phase 1 루브릭에서 도출, 가중치는 루브릭 배점 비중 + 차별화 축(Phase 1-9)에 가중해 킥오프에서 선언. ③ **동점 tie-breaker 서열 고정**: 차별화 축 점수 → 자기 역량 매칭(Phase 1-7). 각 셀에 source_id 필수(`추정` 셀은 종료 조건 위반). ④ **게이트는 여기서 끝이 아니다** — 렌즈 반증(Phase 2-5)·기존 제품 충돌(Phase 2-6, ①의 상세 실행)·삼각측량 천장(Phase 2-7)도 후보를 죽이거나 상한을 씌우는 게이트다. 이들이 4 *뒤* 스텝에 오므로 **4의 가중합 순위는 5~7 게이트 통과 전까지 잠정**(게이트가 후보를 죽이면 재가중). Phase 2-6은 ①과 중복이 아니라 ①의 문서화 단계다.
5. [판단] **다관점 렌즈 회전 (반증 포함)**: 리서치가 한 관점에 매몰되면 문제의 한 면만 본다. 살아남은 각 후보를 최소 3개 렌즈로 돌려본다 — 수요자 / 공급자 / 실무자(그 순간 손을 움직이는 사람) / 책임자(결재·리스크 소유자) / 회사(사업·경쟁) / 산업·밸류체인(전후 단계) / 국가·규제. 렌즈마다 두 질문: ① 이 관점에서 이 문제는 어떻게 보이는가(다르게 정의되는가) ② **이 관점에서 반증이 있는가** — "이미 해결됐다" 또는 "실은 문제가 아니다"는 증거를 *의도적으로* 찾는다(확증 편향 상쇄). 산출물 = 렌즈 매트릭스(렌즈 × 후보: 지지/반증/무관심 + 근거 source_id). 반증을 찾다 실패한 기록 자체가 답변의 "탈락 근거·차별성" 소재가 된다. 어느 렌즈에서 반증이 성립하면 후보 기각 또는 문제 재정의.
6. [결정론적] 기존 제품 충돌 검사: 그 조직의 기존 제품·본선 문제·범용 도구와 겹치면 탈락 사유를 문서로 남기고 제외한다. (5의 "회사 렌즈 반증"의 특수 케이스 — 렌즈 매트릭스와 교차 확인.)
7. [결정론적] **출처 삼각측량 = E-level 사다리 짝짓기**: E4~E5(존재 증언)에는 E1~E2(규모·사실)를 짝지운다. 증언만 있으면 8점 천장. 실전 감점의 실체도 "출처 종류"가 아니라 "삼각측량 부재"였다.
8. [판단] **좁은 문제를 끝까지** — 넓은 플랫폼보다 좁은 문제의 완결(입력→판단→산출물→다음 행동)이 강하다. 중단 기준(고유 pain point 없음 / demo 불가 / 문제정의가 범용적 / 비공개 의존)을 미리 정의한다.
9. [결정론적/판단] **리서치 종료 게이트 (품질 조건 ∨ 마감 hard-stop — 먼저 닿는 쪽으로 종료)**: 순수 품질 게이트는 마감 없는 니치 문제에서 **종료 불가**에 빠진다(E1~E2가 애초에 없는 문제, 마지막 소스가 계속 새 사실을 더하는 경우) — 그래서 품질 조건과 함께 마감 역산 hard-stop·미충족 탈출로를 둔다. **먼저 닿는 조건으로 닫되, 무엇으로 닫았는지 기록**한다.

   **품질 조건** (전부 충족 = 정상 종료):
   - [ ] 문제에 대한 **외부 정의 ≥5** 수집 (자기 정의 확정은 그 후 — 레포 Judge 규약)
   - [ ] 선택 후보에 **E1~E2 근거 ≥2** + E4~E5 증언마다 짝 완료
   - [ ] **렌즈 ≥3개 검토 + 렌즈별 반증 탐색 수행** 기록 존재
   - [ ] 스코어카드의 전 평가 축이 근거 있는 값으로 채워짐 (`추정` 셀 0개)
   - [ ] **saturation**: 마지막 신규 소스 3개가 새로운 사실을 더하지 않음 — 과다 리서치 방지는 시간이 아니라 이 신호로
   - [ ] **독립 적대 검증 1회 통과** (§2-A): clean 컨텍스트에 문제 정의+근거만 주고 *"이 문제는 실재하지 않는다 / 이미 해결됐다 / 규모가 과장됐다를 논증하라."* 문제 정의는 최고 레버(garbage in)인데 이 게이트 전까지 유일한 반증은 후보를 만든 같은 컨텍스트의 self-critique(Phase 2-5 렌즈)뿐 — §2-A가 신뢰 못 한다고 명시한 형태다. 완료 판정을 생성자와 분리(§2-A IRON LAW의 리서치 버전).
   - [ ] **가시성**: 리서치 산출물(소스맵·렌즈 매트릭스·삼각측량·스코어카드·반증 실패 기록)이 Phase 1-4 scoring surface에 남고 Phase 1-5 claim graph 노드에 연결됨 — "로그 밖 리서치 = 안 한 것"(인트로 경고)의 종료 버전.
   - [ ] **문제 정의 성공 신호 선언**: "이 정의가 맞았다면 나중에 무엇으로 확인되나"를 1줄 — 예: Phase 6 dogfood에서 실사용자가 pain 인정 / Phase 9 실측에서 문제 실재. Phase 9-2 판정표 입력이 된다.

   **마감 hard-stop과 탈출로** (품질 미충족인데 시간 소진 or 조건이 구조적으로 불가할 때):
   - **마감 역산 hard-stop**: 슬롯 시간 예산의 리서치 몫을 소진하면 미충족이어도 멈춘다. 미충족 항목을 Open Risks(Phase 4-3 장부)에 명시하고, 그 리스크를 Phase 5 output·답변에서 어떻게 헤지할지 결정한다(모르는 걸 아는 척 금지).
   - **증거 부재 → downgrade/기각**: "증거를 못 찾음"과 "증거가 없음"을 구분한다. E1~E2가 *존재하지 않아* "근거 ≥2"가 원천 불가면 그 후보를 (a) E-level 요구가 낮은 claim으로 **범위 축소**하거나 (b) **기각**하고 차순위 후보로 회귀. 못 찾은 것이면 계속.
   - **saturation 미달 → 좁히거나 승인**: 계속 새 사실이 나와 saturation이 안 오면(=문제가 아직 안 좁혀짐) 무한 리서치 대신 **Phase 3 scope로 회귀해 문제를 더 좁히거나**, 의사결정권자(사용자) 1줄 승인으로 현재 정의를 잠정 확정한다.

### Phase 3. 해법 결정·범위 고정 (문제 확정 직후, 같은 작업 세션 안)

> 구명 "솔루션 선택·범위 계약". 출구 = **무엇을 만들지 + 어디까지만 만들지**가 문서로 고정된 상태(아래 scope contract).

> **문제/솔루션 분리 게이트**: Phase 2이 닫히기 전(문제 미확정) 솔루션 논의 금지. 실전 로그 실측 — 문제 후보가 솔루션 정의로 서술돼 사용자가 직접 재작업을 지시했다. 섞이면 재작업이 실제로 발생한다.

0. [판단] 문제 자체에 **포싱 질문 3종**(gstack office-hours 차용)을 먼저 적용: ① 수요 실재 — "이게 사라지면 진짜로 곤란해지는 사람이 있는가"(관심≠수요) ② status quo — "지금은 이걸 어떻게 때우고 있고 그 비용은"(진짜 경쟁자는 기존 워크어라운드; 아무도 아무것도 안 하고 있으면 안 아픈 문제) ③ desperate specificity — "가장 아쉬운 실무자의 직함·승진/문책 조건"(카테고리는 사람이 아니다). 답이 뭉개지면 Phase 2 렌즈 매트릭스로 회귀.
1. [판단] **솔루션 후보 발산 (문제당 ≥3)** — 두 축의 메뉴를 돌린다:
   - **형태**: 자동화 도구 / 판단 보조(추천+근거 제시) / 검증기(사람 산출물을 검사) / 파이프라인 / 가시화·대시보드
   - **해결 깊이**: 정보 제공 → 판단 보조 → 판단 대행 → 실행 연결. 어디 서느냐가 임팩트와 리스크를 동시에 정한다 — 한 실전 사례의 감점("산출물이 reviewer-handoff에서 멈춤, 다음 행동 완결까지 안 닿음")이 깊이 선택의 실패 사례다.
2. [결정론적] **Measurement Plan 선행 — 측정할 수 없는 솔루션은 기각**: 각 후보에 대해 솔루션 확정 *전에* 정의한다 — 성공 기준(숫자) / 골든셋·fixture를 만들 수 있는가 / negative 케이스가 정의되는가 / 재현 경로가 서는가. 하나라도 불가면 그 후보는 평가 없이 탈락 (Cherny: "구현 전에 스스로 검증할 방법부터" / 심사 역설계: "검증 가능성 자체가 어필"). **모호어 금지** (gstack spec anti-pattern): 성공 기준에 "잘 작동한다"·"정확하다"류 주관어 금지 — 테스트 가능한 pass/fail 문장만.
3. [판단] **솔루션 평가** — Phase 2-4와 같은 게이트→가중합 구조 재사용:
   - 게이트: Measurement Plan 성립(위 2) / 필요 데이터가 공개·접근 가능(비공개 의존=즉사) / 마감 내 E2E 완결 가능
   - 가중합: 임팩트(심사 핵심 질문 = "실제로 일을 얼마나 줄이나") / 내 역량·자산 매칭(기존 엔진·인프라 재사용 — Phase 1-7 뷰) / **차별화 축 적합**(인터랙티브 데모가 나오는 형태인가 — Phase 1-9 관통) / 구현 비용·리스크(역가중)
4. [판단] **범위 계약(scope contract)으로 마감** — 문서로 고정:
   - **scope 모드 명시 선언** (gstack plan-ceo-review): EXPANSION / SELECTIVE / HOLD / REDUCTION 중 하나 — 선언 후 몰래 바꾸지 않는다(silent drift 금지). 예선 = 보통 HOLD, 본선 시간압박 = REDUCTION.
   - 클러치가 되는 핵심 기능 **1개** (제한 시간 안에 가장 가치 있는 하나 — 산으로 가기 차단)
   - **anti-scope**: 만들지 않을 것 목록 — "이 게이트에서 하지 않을 일" 미명문화가 실전 재작업의 원인이었다
   - 완결 경계: 입력→판단→산출물→**다음 행동**의 어디까지 닿는지 명시 결정
   - 알렉스 반박 1회 의무 적용: "문제 선정은 좋은데 이 솔루션은 임팩트가 약하지 않은가 — 데모 가능한 비주얼 output으로 재설계하면?"
5. [결정론적] **적대 검증 1회 (§2-A 호출)** — 솔루션 결정 직후가 뒤집기 가장 싼 시점 (실전에서도 scope 확정 직후에 적대 검증을 배치했다).

산출물: 문제 정의 + 솔루션 형태·깊이 + Measurement Plan + scope contract(핵심 1기능·anti-scope·완결 경계) — 이 묶음이 Phase 4 킥오프 계약의 입력이다.

### Phase 4. 작업 설계·킥오프 (슬롯당 첫 세션)

> 구명 "스캐폴딩·플래닝·킥오프 계약". 출구 = 실행 준비 완료(게이트 순서·산출물 skeleton·킥오프 계약이 선 상태).
>
> [조건부] **로그가 채점 표면인 대회면** "전략을 로그 안/밖 어디서 하느냐"가 추가 변수가 된다(계획 발화가 로그에 보여야 채점됨) → **§실전 운용법 "3층 로그 전략" 참조.** 이 Phase 본문은 패러다임 무관 범용 뼈대만 둔다.

1. [결정론적] **킥오프 프롬프트에 계약을 전부 싣는다**: 게이트 순서 + 각 게이트 산출물 skeleton + 평가 축(Phase 1 루브릭 유래) + Phase 3 산출물(솔루션 형태·anti-scope·Measurement Plan) + "각 산출물은 내 승인 후 다음 단계" + 로그 원본성 규칙 + 위임 원칙("원자료는 에이전트 로그, 판단은 메인 로그"). — 실전 로그 실측: 계약 없는 킥오프는 즉흥 재작업을 유발했고, 머리·세션 계약+플로우맵까지 실은 킥오프가 가장 안정적이었다.
2. [결정론적] 머리 분리 = 세션 분리: 문제 머리(A)→제작 머리(B)→사용자 머리(C)→패키징 머리(P)를 별도 세션으로. **킥오프는 첫 세션을 여는 계약이다**: Phase 4 킥오프 프롬프트로 슬롯의 첫 작업 세션을 열고, 그 프레임 아래에서 Phase 2·Phase 3(문제·솔루션 확정)를 **문제 머리(A)의 내용으로 실행**한다(Phase 4-1이 요구하는 Phase 3 산출물은 A→B 경계에서 확정·재확인). 세션 경계=머리 경계, resume 금지, 경계마다 handoff 문서(다음 세션이 읽을 파일 ≤5 + 재오픈 금지 목록 + 다음 첫 행동).
3. [결정론적] 공용 상태 장부(flow-map, 자산: `flow-map-template.md` — Sequence Status / Evidence Chain / Decision Log / Open Risks / Completion Record)를 두고 매 세션(머리) 종료 시 갱신한다.
4. [판단] 로그가 채점 표면인 대회면 **개입 문장 레퍼토리**로 사람 개입을 로그에 남긴다(목록 → §실전 운용법).
5. [결정론적] **유형별 머리·세션 변형** — 원리: 머리(=세션)를 나누는 이유는 작업량이 아니라 **다른 관점이 필요해서**다(문제를 정하는 머리 A / 만드는 머리 B / 쓰는 사람의 머리 C / 패키징 머리 P — 세션 분리는 관점 전환의 컨텍스트 오염 방지 장치). 세션 경계 판단(유형 무관): ① 다음 작업에 다른 관점이 필요한가 + ② 이전 산출물이 동결 가능한가(재오픈 금지 선언 가능) — 둘 다 yes면 경계.

   | 유형 | 머리 흐름 | 세션 경계 |
   |---|---|---|
   | 제출형 해커톤 | A→B→C→P 원형 | 머리 = 세션 (1:1) |
   | 데이터 경진 | A(EDA·metric 분석) → B(베이스라인→**실험 사이클 반복**) → C(validation 전략·리더보드 프로빙 통제) → P(최종 제출 선택·재현 코드) | B는 단일 세션이 아니라 사이클 — 경계는 실험 배치 단위 |
   | 아이디어 공모전 | A 비대(승부의 대부분) → B(문서·목업) → C(**제3자 첫 독해 테스트**) → P | C가 dogfood 대신 "처음 읽는 사람" 테스트 |
   | 오프라인 해커톤 | A·Phase 3를 **대회 전 사전 수행** → 현장은 B·C만 | 세션 분리 불가 → 게이트를 시간 체크포인트로 변환 |
   | 사업계획 | A+시장 검증(`/business-docs`·office-hours 포싱 질문) → B(계획서) → C(심사위원 관점 독해·기관 평가지표 매칭) → P(발표 리허설) | 원형 유사 |

6. **Phase 4 출구 게이트: 스캐폴딩 검증 (4층, 싼 것부터)** ★스캐폴딩 결함은 하류 전체를 오염시키고 회귀 비용이 가장 크다 — 최고 품질의 스캐폴딩이 최고 품질의 결과를 만든다.
   - **6-0. 스캐폴딩 구조 발산-수렴** [판단] (검증 전에 먼저): 첫 떠오른 게이트 구조 하나를 바로 검증하지 말고 — 대안 스캐폴딩 구조 **2~3개를 발산**(예: 머리·세션 경계를 다르게 그은 안, 게이트를 더 잘게/굵게 쪼갠 안)한 뒤 Phase 1 루브릭 커버리지·회귀 비용 기준으로 **수렴**한다. Phase 2/Phase 3가 문제·솔루션에 강제하는 발산-수렴을 스캐폴딩에도 적용 — "검증은 잘했지만 애초에 구조 선택지를 안 본" 국소최적을 막는다.
   - **6-a. 커버리지 검사** [결정론적]: **Phase 1 루브릭 전 항목 × 게이트 산출물 매핑표** 작성 — 빈 셀 = 설계 구멍("이 항목의 점수는 어디서 만들어지나"에 답 없음). Phase 3 scope contract·Measurement Plan·차별화 축도 어느 게이트에서 실현되는지 명시.
   - **6-b. 계획 적대 검증** [§2-A를 계획에]: clean 검증자에게 킥오프 계약+scope contract**만** 주고 — "이 계획대로 실행하면 어디서 막히는가 / 어떤 산출물이 계약에 없어 마지막에 급조되는가 / 게이트 소관이 겹치거나 비는 곳은". (실전 재작업 원인 "게이트 소관 분리 미명문화"가 이 검증이 잡았을 결함.)
   - **6-c. 프리모템** [판단]: "마감일이다. 이 제출물이 **떨어졌다**. 왜 떨어졌나?" — 실패를 기정사실로 놓고 원인을 서술(Klein premortem; 리스크 나열보다 구체적 결함이 나온다). 상위 실패 시나리오 3개 → 각각 예방 항목을 게이트/체크리스트에 역주입.
   - **6-d. 드라이런** [가장 강한 검증]: **가장 리스크 높은 게이트 하나**(파급력×발생확률 최상위 — "첫 게이트"가 아니라 실패 시 하류를 가장 크게 오염시킬 게이트, Phase 5-2 risk-based와 정합; 리스크가 균등하면 최상류 게이트)를 **소규모로 실제 실행** — 계약이 실행 가능한지, skeleton이 실제로 채워지는지, 프롬프트가 의도대로 작동하는지 관측 (알렉스 "스캐폴딩 테스트"의 실행형 = thin slice를 계획 자체에 적용).
   - **6-e. 계획의 Measurement Plan 선언** [결정론적] (Phase 3-2를 계획 자체에): 스캐폴딩이 성공했는지 나중에 판정할 **positive 성공 신호를 지금 선언**하되, 각 신호에 **계산식·데이터 원천·기록 시점·판정자**를 붙여 측정 가능하게 한다(선언만 하면 이 항목도 "모호어 금지" 위반 — 재작업이 나도 완료 체크를 통과해버린다). 예: "문제 머리(A) 재작업 = 사용자가 재작업을 지시한 턴 수(메인 로그 grep), 목표 0 / 계약 외 급조 산출물 = final/에 있으나 킥오프 계약에 없던 파일 수, 목표 0 / 게이트 소관 충돌 = 적대검증이 지적한 소관 중복 수, 목표 0". **판정은 Phase 9-2 판정표에서**(선언값 vs 관측값 vs verdict). 신호가 나쁘면(재작업>0) 스캐폴딩 실패로 회고에 근본원인 기록.
   - **수정·회귀 규칙**: 발견 결함은 킥오프 계약을 고치고 버전 표시(계약 v1→v2). 실행 중 스캐폴딩 결함 발견 시 전체 재시작이 아니라 **결함 게이트만 지목해 회귀** — 그 게이트만 재계약하고 하류 중 영향받는 산출물만 무효화(회귀 비용 국소화).
   - 이 검증 전 과정을 작업 세션 안에서 수행하면 그 자체가 최상급 하네스 엔지니어링 로그 증거(§실전 운용법 ②층과 정합).

### Phase 5. 구현·검증 루프 (제작 머리)

> 구현은 병목이 아니지만 **검증은 병목이다** — 에이전트 산출량이 사람 검증 속도를 초과하므로 검증을 구조화해야 한다. 지배 원칙: **precision > recall** (OpenAI 코드 검증) — 전부 검증하려는 순간 아무것도 검증 못 한다. 신호가 강한 소수 항목만 확실히 검증하고, 그 소수는 리스크 순서로 고른다.

1. [결정론적] **구현 루프 계약**: plugin/제품 contract 먼저 → 첫 thin slice → 구현·증거 루프. 에이전트에게 맡길 때의 표준 — **실패하는 테스트 먼저 작성 → 실패 확인 → 커밋(안전망) → 통과까지 구현, 테스트 수정 금지**(에이전트는 방치하면 구현부터 짠다). **완료 판정은 구현자와 분리** — 구현한 에이전트가 자기 완료를 인증하지 않는다(같은 컨텍스트는 모호한 결과를 관대하게 해석 — §2-A IRON LAW의 구현 루프 버전). 타입체크·린트는 상시 켜두는 결정론적 센서 — LLM 생성 코드 3대 빈발 결함(Missing Corner Case / Wrong Input Type / Hallucinated Object, arXiv 2403.08937)이 정적 타입체커+속성기반 테스트로 저비용 커버된다. 구현 노동은 위임·병렬화 — 병목은 여기가 아니다.
   - **테스트 자체의 유효성 검증** (AI가 테스트도 짠다는 전제의 그림자): AI가 만든 테스트는 tautological(`assert True`류)이거나 구현을 베껴 항상 통과할 수 있다. 저비용 3중 방어 — ① red 관측 강제(위 실패-먼저 계약이 이미 tautological 테스트를 걸러낸다 — 애초에 red가 안 뜬다) ② **의도적 결함 주입 1회**(핵심 로직 모듈 한정): 구현에 버그를 심어 테스트가 실제 red가 되는지 확인(mutation의 수동 스팟체크). 안 깨지면 그 테스트는 허수 ③ **테스트 작성 에이전트 ≠ 구현 에이전트**: 같은 컨텍스트는 구현의 잘못된 가정을 테스트에 그대로 복사한다(§2-A 컨텍스트 오염의 테스트 버전).
2. [판단] **검증 예산 배분**: ① 검증 대상을 **파급력×발생확률**로 줄 세워 상위부터(risk-based, ISO 29119 — 시간이 끊겨도 고위험은 커버된 채 끝나게) ② 배분은 testing trophy — static 상시 → **integration 레벨에 최대 예산** → E2E는 핵심 사용자 여정 1~2개만. LLM 앱의 integration 경계 = **모델 API↔앱 로직 / 검색↔생성(RAG) / 프론트↔백 계약** — 버그가 몰리는 이 이음매를 우선 대상으로 지목한다(선언만 하지 말고 대상을 박는다) ③ 리듬 — 매 빌드 smoke(핵심 경로 1회), 변경분엔 sanity만, 전체 regression은 제출 직전 1회(상시 유지 비용 지불 안 함).
3. [결정론적] **fixture taxonomy**: normal / needs_more_facts / unsupported / out_of_scope / negative / **adversarial** / regression / human-readability — 실패·거절도 "정상 품질 출력"으로 설계한다. **negative(도메인상 잘못된 입력)와 adversarial(악의·오염 입력)은 다른 유형** — adversarial = malformed 입력 / 프롬프트 인젝션 시도 / PII·시크릿 포함 입력 / (콘텐츠 대회면) 유해 출력 유도이며, 각각 **"안전하게 거절·마스킹"이 정상 출력**이다. AI 앱 특유의 실패면이라 negative에 뭉뚱그리면 검증 대상에서 빠진다(대회 규정의 저작권·개인정보 고지와도 직결). 원칙: **"scope는 좁게, 커버리지는 완전하게"** (gstack "Boil the Ocean"의 대회 재해석) — 문제 범위는 Phase 3에서 좁혔으니, 좁힌 범위 *안*의 edge·negative·adversarial 케이스는 shortcut 없이 전부 닫는다. AI 시대에 커버리지의 한계비용은 거의 0이고, 심사는 구현된 만큼만 점수를 준다.
4. [결정론적] **오라클 없는 출력 검증** (LLM 응답·생성물·추천 등 "정답을 모르는" 산출물 — AI 대회의 핵심 검증 문제):
   - **선행: 비결정성 통제** (골든마스터·메타모픽의 성립 조건 — 이걸 안 하면 아래가 구현 단계에서 그냥 깨진다): ① 통제 가능한 비결정성부터 핀 — temperature=0(또는 최소)·seed 고정·모델/프롬프트 버전 핀·시간/랜덤 의존 제거. 골든마스터 바이트 대조는 이 고정 위에서만 성립 ② 본질적 잔여(샘플링) 비결정성은 정확 일치를 포기하고 **분포로 검증** — 같은 입력 k회(k=5~10) 실행해 핵심 필드 다수결 합의 + 분산 임계 이하 + 메타모픽 불변식이 k회 전부 성립. 골든마스터 판정을 "바이트 일치"에서 **"k회 중 과반이 golden과 일치 + 불변식 위반 0"**으로 완화. **이걸 Phase 8로 미루지 않는다** — 골든셋을 *만드는 순간*(구현 단계) 이미 필요하다(Phase 8-2의 k회 통계 통제는 이 항목의 제출 직전 재확인일 뿐).
   - **메타모픽 관계 ≥2 정의**: 단일 정답 없이도 "입력을 이렇게 변형하면 출력 간 이 관계가 성립해야 한다"로 검증(동의어 치환→분류 동일, 순서 셔플→합계 불변 등). 오라클 문제의 확립된 해법(arXiv 2212.06118·2402.14480) — 구현 비용은 변형 함수+assertion 수준.
   - **골든마스터 역전 적용**: 초기 출력을 사람이 1회 검증 → golden으로 동결 → 이후 프롬프트 튜닝·재생성·리팩터마다 자동 대조(회귀 감지). 초기 산출물의 골든셋이 심사관 재실행에서 바이트 단위로 재현된 실전 사례가 있다. **도구**: 프롬프트 회귀의 자동 대조는 §2-D의 LLM Eval(promptfoo·giskard) — 프롬프트를 고칠 때마다 골든셋과 자동 재채점해 Phase 5 속도를 올린다.
   - **LLM 평가 골든셋은 실패 사례 20~50개로 즉시 시작**(Anthropic: "완벽한 eval을 기다리지 마라 — 늦출수록 만들기 어려워진다"). LLM-as-judge는 4조건 전부 충족 시만 신뢰: 차원별 루브릭 분리 채점 / "모르겠음" 탈출구 제공 / 소수 사람 채점과 캘리브레이션 / 피평가 모델과 다른 계열 judge.
5. [결정론적] **인터랙티브 데모 게이트** (차별화 축 Phase 1-9의 실현 표면): 스택 선택은 플레이북 밖 — `shelf recall`(frontend·design-reference 카드: shadcn-ui·cmdk·cult-ui·3D 팩) + askewly design system. 플레이북이 소유하는 것은 검증 게이트다:
   - **critical-path smoke 1개 = 심사 시연 시나리오 그대로**(Playwright): 앱 로드→핵심 인터랙션→기대 결과 렌더. 테스트가 리허설 스크립트를 겸한다.
   - 핵심 화면 3~5개 스크린샷 baseline(내장 `toHaveScreenshot()` — 외부 서비스 불필요) + axe-core 1줄(WCAG A/AA만; 자동 검사는 접근성 이슈의 ~57%만 잡음(Deque 실측) — 나머지는 Phase 6 사람 QC 몫).
   - **Error boundary 2개**: 앱 전체 + WebGL 캔버스 분리 — three.js는 context lost 자동 복구가 안 되는 known limitation이므로 `webglcontextlost`에 preventDefault+수동 재시도 fallback 필수.
   - **성능·응답성 예산** (LLM 앱 데모가 현장에서 hang하는 주 원인 = latency·rate-limit — 구현 단계에서 정한다): 응답 시간 상한 선언 + 초과 시 **로딩 상태 UI**(빈 화면·멈춤 금지) + **타임아웃 fallback**(무한 대기 금지) + rate-limit/quota 소진 시 graceful degradation(캐시된 예시 응답으로 전환). **상한을 P95로 걸면 1회 실행으론 측정 불가** — 데모 규모에선 percentile 대신 **관측 가능한 기준으로 강등**한다: "동일 인터랙션 k회(k=5~10) 반복해 전부 상한 이내 + 연속 요청 3회에 체감 열화 없음". 부하 확인 입력 = 빈 입력 / 최대 길이 입력 / 연속 요청. (풀 부하 테스트 리그는 이 규모에서 과잉 — Phase 8-3 demo gods의 seeded 데이터가 리허설 버전.)
   - 리허설 시 devtools console 열고 전체 시나리오 1회 통과 — red error 0 육안 확인(이 규모에서 도구화는 과잉).
6. [결정론적] **검증 증거의 가시화** ★AI-native 심사 대회의 최대 실점 레버 — "검증을 했다"가 심사 표면에 없으면 안 한 것과 같다(Phase 2 경고문 "리서치가 로그 밖에 있으면 자율성 낮음으로 읽힘"의 검증 버전): 각 검증 활동(비결정성 통제·메타모픽·골든셋·적대 검증·smoke·클린 재현)은 심사자가 읽는 표면에 떨어지는 **산출물**을 남긴다. 매핑 강제 — `검증 종류 → 산출물 파일 → scoring surface(Phase 1-4) → 답변 참조`.
   - 표준 산출물: `verification/matrix.md`(command / expected / observed / evidence 행) + 재실행 로그 요약 + 골든셋 통과 요약 + 적대 검증 verdict 요약.
   - README "검증" 섹션이 이들을 요약·링크하고, 답변 claim이 matrix 행을 가리키게 한다(Phase 1-5 claim graph의 "검증 행" 노드가 실재하도록).
   - **재실행 증거는 원시 로그가 아니라 요약 표면에** — 심사 AI의 컨텍스트 한계·zip 깊이 가정(Phase 1-3)상, 깊은 로그에만 있는 증거는 없는 것과 같다.
7. [판단] 위임: 서브에이전트에 좁은 과업만, 위임 전 의도 선언 → 위임 후 채택/기각 판단을 메인 발화로. Delegation Ledger(위임 과업/입력/결과/메인 검증/채택 여부/이유)로 구조화.
8. [결정론적] 세션(머리) 종료 전 **적대 검증 게이트** — §2-A 프로토콜대로. 통과 후 **루브릭 미니 채점**: 이 세션 산출물이 Phase 1 루브릭의 어느 항목 점수를 만들었는지 매핑하고 gap을 다음 세션 입력으로.
9. [판단] 멈춤 신호: 새 수렴 블로커가 없고 남은 지적이 구조 본질 한계면 엔진 수정을 멈추고 답변/서사 축으로 전환한다(whack-a-mole 금지). **3-strike rule** (gstack investigate): 같은 문제에 가설·수정 3회 연속 실패하면 STOP — 계속 파지 말고 문제를 재구성하거나 상위 게이트(Phase 3 scope)로 회귀 검토. "수정할 때마다 다른 곳에서 새 문제"가 슬로우다운 신호.

**의도적으로 채택하지 않는 것**: mutation testing은 핵심 채점/검증 로직 1~2개 모듈의 스팟체크 옵션으로만(테스트 스위트가 허수인지 5~10분 확인 — 상시 운용은 단기 ROI 낮음) / 시각회귀 외부 서비스(Chromatic·Percy — Playwright 내장으로 충분) / 상시 전체 regression suite.

### Phase 6. 사람 직접 QC/QA·dogfood (쓰는 사람 머리) ★위임 불가

> 최종 품질 판정은 위임할 수 없다. 에이전트 검증이 아무리 겹쳐도 "실사용자로서 처음부터 끝까지 직접 써본 사람"의 발견을 대체하지 못한다 — 실전에서도 스킬 출력 문구가 만든 본인에게조차 불명확했던 것을 dogfood에서야 잡았다.

1. [판단] **Dogfood 실행 프로토콜**: 실사용자 시나리오로 처음부터 끝까지 직접 사용 → finding을 fix/reject/defer로 분류 → rerun → 문서 동기화. scope 확장 finding은 기각하고 이유를 남긴다.
2. [판단] **사람만 잡는 것 — 5대 체크** (Phase 5 자동화가 *구조적으로* 못 잡는 잔여. 이게 Phase 6의 핵심이자 사람 QC의 본체다):
   - **첫 독해 이해가능성**: 이 출력·README·답변을 처음 보는 사람이 ① 무엇인지 ② 다음 행동이 뭔지 아는가. 만든 본인에겐 자명한 게 처음 보는 사람에겐 불명확하다 — 실전 실측(출력 문구가 만든 본인에게조차 불명확했던 걸 dogfood에서야 잡음).
   - **taste·품질 바**: "통과"가 아니라 "좋은가" — 내가 심사자면 이걸 상위권으로 볼까 / 부끄럽지 않은가(embarrassment test). 통과선과 우승선은 다르다(Phase 1-9 차별화 축의 사람 판정).
   - **접근성·UX 잔여**: axe가 못 잡는 ~43% — 포커스 순서의 논리, alt-text의 의미성, 키보드 전용 조작 가능 여부, 실제 색 대비 가독성, 로딩·에러 상태의 체감(Phase 5-5 자동 게이트가 남긴 사람 몫).
   - **엣지 실사용**: 자동 fixture가 아니라 사람이 즉흥적으로 이상하게 써보기 — 빈 입력·말도 안 되는 입력·연타·뒤로가기·새로고침. "의도적 오사용"에서 나오는 finding은 시나리오 테스트가 못 만든다.
   - **감정·심미** (콘텐츠·데모형): 만졌을 때의 인상, 데모의 "한 방"이 실제로 임팩트 있는가, 첫 3초. 심미는 역설계가 안 되는 축이라 사람 눈이 유일한 판정자다.
   - ※ **의미적 정확성·크로스-아티팩트 정합**(답이 형식만이 아니라 내용도 맞는가, 답변↔README↔코드↔데모가 모순 없는가)은 Phase 6의 코어가 아니다 — Phase 2-7 삼각측량·Phase 7 claim lock·Phase 8 재실행이 주 담당하고, Phase 6에선 사람이 스팟 확인만 한다.
3. [판단] **심사자의 눈 통과** (judge's-eye pass): 제출물을 심사자처럼 — fresh 상태로, 표면만 보고, Phase 1 루브릭을 들고 — 직접 읽어 감점 포인트를 사람이 찾는다. Phase 8 독립 심사관 에이전트의 사람 버전을 더 이르게 1회.
4. [판단] **finding 처리 + 가시화**: 개입할 때마다 "왜 지금 멈췄는지" 한 줄 — taste의 증거이자 로그 신호. 그리고 이 dogfood 자체가 검증 산출물(finding 목록·처리 결과)로 남아 심사 표면에 연결된다 — "사람이 직접 QC했다"가 보여야 점수가 된다(Phase 5-6 검증 증거 가시화 정합).

### Phase 7. 패키징·정합 (패키징 머리, 작업 세션 완전 종료 후 별도 세션)

> 패키징은 새 산출물을 만드는 단계가 아니라 **이미 만든 것을 깨지지 않게 동결·검증·제출**하는 단계다. 여기서 새 기능을 붙이면 검증 안 된 코드가 제출본에 섞인다 — 발견은 기능 추가가 아니라 회귀(해당 게이트 재실행)로만 처리한다. 실전 최다 사후 결함이 전부 이 단계 누락(클린 재현·CRLF·정합)에서 나왔다.

1. [결정론적] **패키징 파이프라인**: freeze → package build → package audit → answer draft → final consistency. zip 포함 파일이 하나라도 바뀌면 루프 처음부터 재실행. 수동 zip 편집 금지. **원자적 승격** (gstack skillify Iron Contract): 스테이징 경로에서 작업하고, 검증 통과 + 명시 승인 **둘 다** 있어야 `final/`로 이동 — "거의 제출본" 같은 중간상태를 남기지 않는다(실패 시 스테이징 폐기).
   - **package audit가 실제로 보는 것**: 매니페스트 ↔ 실제 zip 내용 1:1 대조(빠진 파일·군더더기 파일 0) / 스테이징 밖 절대경로 참조 0 / `.DS_Store`·`__pycache__`·로컬 캐시·빈 폴더 미포함 / zip이 실제로 풀리는지 1회 해제.
2. [결정론적] **claim lock**: 최종 답변 문장을 ok/weaken/remove로 잠근다 — 답변이 산출물보다 크게 말하지 않게. remove/weaken로 걸린 문장은 근거 표면(matrix 행·공개 URL)이 없어서 걸린 것이니, 지우거나 산출물에 근거를 만들어 ok로 승격한다.
3. [결정론적] **노출 스캔** (제출 전 마지막 위생): 절대경로(`C:\Users`·`/home/`) / 실명·개인정보 / API 키·토큰·시크릿 / 타 대회·전략 채널 언급 / 심사 최적화 발화 / 내부 메모 — zip 내 전 텍스트 파일을 grep 패턴으로 훑는다.
4. [결정론적] **클린 환경 재현 검증을 패키징 게이트에 포함** — 제출 zip을 fresh 디렉터리(가능하면 새 계정·컨테이너)에 풀어 **README만 보고** 설치→실행 재현. 제작 환경의 전역 설치·환경변수에 몰래 의존하는지 이때 드러난다. 실전 실측: 1차 제출 후에야 "설치 절차 README 누락"을 발견해 별도 수정 세션이 필요했다 — 사후가 아니라 게이트로.
5. [결정론적] **크로스플랫폼 아티팩트 체크**: 골든 해시·테스트가 CRLF/LF·OS 차이에 견디는지 타 OS 클론 1회로 확인. 필드 표본 9건 중 4건이 "전수 통과 주장이 재실행에서 깨짐"(전부 CRLF/Windows 계열) — **필드 최다 고장 모드**. 멀티 슬롯에서 엔진을 재사용하면 **슬롯 간 회귀**도 확인 — 한 슬롯의 수정이 다른 슬롯의 골든셋을 깨는지 전 슬롯 골든셋 재실행 1회.
6. [결정론적] 답변 글자수 스크립트 실측, 수치는 최종 README/코드로 재확인(실전 사례: 47→60건 정정).
7. [결정론적] **크로스-아티팩트 정합 매트릭스** (Phase 6-2가 여기로 위임한 의미 정합의 결정론적 버전): 같은 수치·주장이 **답변 ↔ README ↔ 코드/산출물 ↔ 데모**에서 전부 일치하는지 행으로 대조. 한 곳만 고치고 다른 곳을 안 고친 drift가 최다 정합 결함 — 특히 **마지막에 바꾼 숫자**를 역추적한다.
8. [결정론적] **제출 요건 대조 + 업로드 드라이런**: Phase 1-1 규정 표의 제출 형식(파일명 규칙·용량 상한·확장자·파일 개수·PDF/ZIP 구조)과 실제 패키지를 1:1 대조. 가능하면 실제 제출 폼에 **마감 전 미리 한 번 올려본다** — 용량 초과·형식 거부·업로드 실패는 마감 직전에 터지는 대표 사고다. 재제출 허용 대회면 조기 제출 후 교체 전략도 여기서 결정.
9. [결정론적] **발표 산출물 파이프라인** (발표형·라이브 심사 대회 — 실전 갭에서 신설): 발표자료·스크립트를 매번 즉흥 제작하지 않는다. **표준을 미리 정해 킥오프(§2-C-5)에서 고정**한다 — ① **도구**: 하나를 표준으로(후보: HTML→캡처→PDF / typst / 슬라이드 이미지 생성 + 편집. 재사용 템플릿 1개를 자산화) ② **구조**: 심사 배점 순서대로 슬라이드 골격(문제정의·검증설계·증거에 최대 비중, 데모 시연 위치 고정) ③ **발표자·스크립트 담당 사전 지정** ④ **착수 시점 하드룰 = 발표 N시간 전이 아니라 Phase 7 진입 시**. **실전 실패**: 발표 준비를 발표 1시간 전에 착수 → 슬라이드 구성·발표자·스크립트를 그때부터 논의 → 시간 크게 부족. 발표는 배점 10%라도 **마지막 인상(peak-end, Phase 1-3B)** 이라 즉흥은 비싸다.

### Phase 8. 제출 전 최종 리허설 채점

> Phase 1 루브릭 + Phase 5-8 미니 채점이 제 역할을 했다면 여기서는 **확인**만 남아야 한다. 이 단계에서 대형 재작업이 발생하면 그 자체가 프로세스 실패 신호 — 원인을 회고에 기록한다.

1. [결정론적] 제출물 전체(zip+답변)를 **독립 심사관 에이전트**(상호 비공개, Phase 1 루브릭, claim-evidence 대조 + 스크립트 실제 재실행)로 풀 채점 1회.
2. [판단] 감점을 "제출물 고유(고침) vs 공통 천장(무시)"으로 분류해 고유 감점만 수정 → 재채점. **분류·기각 판정에 §2-A "이의·보정 심사(비대칭)" 규칙이 그대로 걸린다** — 감점을 "공통 천장이니 무시"로 분류하는 것은 **기각 판정**이고, 그건 곧 자기 점수를 지키는 판정이다. 제작자 쪽이 단독으로 그 분류를 내리면 self-bias가 작동하며, 독립 심사자가 없으면 그 감점은 무시하지 않고 **살려둔 채 원점수를 유지**한다. 시간이 허락하면 9~10점 갭 3기법 적용: ① 공개 1차 출처 삼각측량 ② k회 실행 분포 보고+다수결 합의(비결정성의 통계 통제 — **Phase 5-4에서 세운 비결정성 통제의 제출 직전 최종 재확인**) ③ clean agent 독립 라벨(golden 자가 라벨 순환성 절단).
   - **3기법 투자는 Phase 1-2 캘리브레이션 게이트 통과가 선행 조건이다** (실전 부분 철회 — 근거 40): ①·②는 *"이 루브릭에서 8점을 10점으로 올리는 법"* 이므로, 루브릭이 분리력 검증을 통과하지 못했다면 **잘못된 목적함수를 더 열심히 최적화하는 것**이다. 미검증 루브릭에서는 ①·②에 시간을 쓰지 말고 그 시간을 Phase 2(문제 정의)·Phase 6(사람 QC)로 돌린다. **③ clean agent 독립 라벨만 캘리브레이션과 무관하게 유지**한다 — ③은 점수를 올리는 기법이 아니라 §2-A 독립성 요구의 채점 버전(자기 라벨 순환 절단)이며, 그 정당성은 루브릭의 유효성에 의존하지 않는다.
   - **수정 시 freeze 무효화**: 여기서 고쳐 답변·README·코드·zip 중 하나라도 바뀌면 이전 Phase 7 검증은 무효 — **Phase 7-1~8 전체를 재실행하고 패키지를 재freeze(새 해시)한 뒤** 재채점한다(§2-A IRON LAW의 패키징 버전). "감점만 살짝 고쳤다"로 건너뛰면 고친 답변과 승인된 zip이 drift한다.
3. [결정론적] **라이브 데모 리허설 — demo gods 프로토콜** (오프라인 해커톤·발표형 필수, 제출형은 데모 영상 제작에 준용): 라이브 데모는 준비와 무관하게 실패할 수 있다는 전제에서 출발한다.
   - **백업 영상 2회 녹화**(전체 시나리오 풀 테이크, 하나는 예비) — 시연 시작 전 확보. 실패 시 "영상으로 대체"를 즉시 선언할 수 있게.
   - **seeded·하드코딩 데이터**로 외부 API·네트워크·랜덤 의존 제거, **로컬 실행 우선**(현장 Wi-Fi 불신), 시연 전 **코드 프리즈**.
   - 실패 가능 구간마다 **fallback 경로 사전 결정**(조건문 수준의 kill switch — 별도 인프라에 의존하지 않는 게 핵심: 장애 원인과 스위치가 같은 자원을 쓰면 무력화).
   - Phase 5-5의 critical-path smoke가 리허설 스크립트 겸용 — 시연 직전 1회 그린 확인.
   - 작은 실패는 웃어넘기고 회복하는 태도까지가 프로토콜(완벽함보다 회복력이 신뢰를 준다).

### Phase 9. 제출 후 회고 기준선 (발표 전)

1. [결정론적] 자체 채점 기록 + 공개된 타 참가자 제출물 수집·동일 루브릭 채점(표본 편향 명시) + 발표 후 검증할 가설 목록화. 발표 결과와 대조해 **루브릭 자체를 채점**한다.
   - **용어 교정 — 라벨 없는 필드 채점은 캘리브레이션이 아니다** (실전 반증 — 근거 40): 타 참가자를 같은 루브릭으로 채점하면 **점수 분포**(내가 어느 클러스터에 있나)는 알 수 있지만, **루브릭이 결과를 예측하는지**는 알 수 없다 — 진출/탈락 라벨이 없기 때문이다. 이걸 "캘리브레이션"이라 부르면 검증했다는 착각이 생긴다. 진짜 캘리브레이션은 라벨된 표본이 필요하며 그 자리는 **Phase 1-2 캘리브레이션 게이트**다.
   - **통과 확률 추정은 Phase 1-2 게이트 통과 시에만 만든다.** 미통과·캘리브레이션 불가면 확률표를 **작성하지 않고** "루브릭 미검증 — 확률 추정 생략"을 명시한다. 대신 남길 수 있는 것은 **관측 사실**(우리 점수·필드 분포·표본 수·편향) + **§2-A "프로바이더 독립" 패널의 상대 평가**(근거 42 — 라벨 없어도 자기평가보다 실제에 가까운 유일한 외부 신호)다.
   - **사전 등록 자산화** (이번 회고의 유일한 순이득): 발표 *전에* 매긴 필드 점수는 결과를 모르고 매긴 **사전 등록된(pre-registered) 예측**이다. 나중에 진출/탈락 라벨이 확보되면(명단 공개·참가자의 공개 언급·수상작 공개) 이 표본이 그대로 **홀드아웃 검증셋**이 되어 Phase 1-2 게이트를 소급 실행할 수 있다. 그러니 발표 전 필드 채점은 계속하되 — **채점 시각·표본 출처·라벨 부재를 명시해 사전 등록성을 보존**하고, 라벨이 안 잡히면 훅을 **실행하지 않는다**(부분 라벨로 억지 추정하는 것이 바로 이 절이 금지하는 결함의 재발).
2. [결정론적] **선언 신호 판정표 (planned / observed / verdict)** — Phase 2-9(문제 정의 성공 신호)·Phase 3-2(솔루션 Measurement Plan)·Phase 4-6e(스캐폴딩 성공 신호)에서 선언한 신호를 여기서 전부 실측 대조한다. 각 행 = `신호 / 선언값(목표) / 관측값(계산식으로 실측) / verdict(달성·미달) / 미달 시 근본원인`. 이 표가 없으면 "성공 신호 선언"은 자기충족 예언에 그친다(재작업이 나도 완료 체크 통과) — 선언과 판정을 분리해 닫는다.
3. [결정론적] 확률 갱신 원칙(Phase 9-1 게이트를 통과해 확률을 만들기로 한 경우에만): 전 참가자 공통 천장형 보정은 순위 불변→확률 불변, 제출물 고유 정정만 확률을 움직임, 단일 심사관 ±1은 노이즈(구간 상단을 기대값으로 읽지 말 것).
   - **선행 조건 — 보정 자체가 §2-A 비대칭 규칙을 통과했을 때만 확률에 반영한다** (실전 반증 — 근거 39). 이 원칙은 "제출물 고유 정정이 확률을 움직인다"고 허용하는데, **그 정정이 자기심사 상향이면 편향이 그대로 확률로 전파된다.** 실전에서 이 경로로 자기심사 +2~3이 "마진 최대"가 되고 그것이 확률의 명시적 근거가 되어 최대 미스를 낳았다. 독립 심사를 못 거친 상향 정정은 확률에 **반영하지 않는다**.
   - **동점을 억지로 가르지 않는다**: 자체 채점이 두 슬롯을 못 갈랐다면 "가를 수 없음(무정보)"이 정답일 수 있다. 무정보를 보정으로 깨서 순위를 만들어내면 **원점수보다 나쁜 예측**이 된다 — 실전에서 실제로 일어난 일이다. 50점 척도로 상위권을 가르려는 시도 자체가 척도 해상도를 넘을 수 있음을 기본값으로 둔다.
4. [결정론적] **회고 역주입 + 가지치기** (양방향 — 추가만 하는 절이 아니다): 회고 교훈을 이 플레이북 해당 단계에 역주입하고 버전을 올린다(사전 자산화 루프의 완성). **동시에 반드시 강등·삭제 후보를 판정한다** — 이 플레이북은 precision>recall·"먼저 버릴 것"을 본문에서 강제하면서 정작 v2→다수 개정까지 한 번도 뭘 버리지 않아 배지가 ~10KB로 비대해졌다(자기 원칙 위반의 물증, v2.18에서 적발·시정). 규율:
   - **① n=1 경화 금지 (단, 대상은 경험적 일반화만)**: 이 가드는 **외부 세계에 대한 예측 주장**("무엇이 이긴다 / 어떤 슬롯이 유리하다" 같은 일반화)에만 적용된다 — 그런 주장은 단일 대회·단일 팀 관측이면 §5 등급 규약대로 **가설 등급으로만** 기입하고 2회+ 독립 대회에서 재현돼야 확정 승격한다(근거 41a·43이 그 예 — n=1이라 유망·가설). **예외: 문서 내부 대조·논리로 성립하는 process 결함은 표본 수가 아니라 내적 정합으로 확정된다** — 근거 39(자기 반론을 자기가 심사=구조적 self-bias)·40(두 독립 자체 루브릭이 같은 방향 오판=계통 오차)은 단일 대회 유래여도 "결과 없이 문서 내부 대조만으로 성립"하므로 확정 게이트가 정당하다. 즉 경계선은 **표본 수가 아니라 "외부 세계 일반화냐 vs 내부 정합으로 증명되는 결함이냐"**다. 한 번의 실패에서 나온 *외부 처방*을 규칙으로 박으면 다음 대회를 그 한 대회에 과적합시킨다.
   - **② 강등·삭제 후보 스캔**: 여러 대회를 거치며 실제로 안 걸리거나 실행 안 된 규칙, 한 대회 특화 세부는 부록으로 강등하거나 삭제한다 — "죽은 규칙"도 읽기·붙여넣기 비용이다(문서는 사용자가 새 프로젝트에 통째로 붙여넣는다).
   - **③ 변경 이력에 3종 다 적기**: 문서 끝 §변경 이력에 **추가·강등·삭제를 함께** 기록한다. 무엇을 버렸는지 안 적으면 다음 개정이 또 쌓기만 한다.
5. [결정론적] **회고 델타표 + 학습 태깅** (gstack retro): 이전 대회 회고와의 델타를 표로 계산(무엇이 나아졌고 무엇이 재발했나 — 재발 항목이 진짜 교훈 실패). 학습 항목은 `type(pattern/pitfall/preference) × source(observed/user-stated/inferred) × confidence(1~10)`로 태깅 — 기록 기준은 "이게 다음 대회의 시간을 절약하는가". 감점 원인은 증상(점수)이 아니라 근본 원인(루브릭 설계·표본 편향·검증불가 처리)까지 추적한다.

## 2-A. 적대적 검증 프로토콜 (공용 서브루틴 — Phase 5·Phase 7·Phase 8에서 호출)

**정의**: 반증을 기본값으로 하는 독립 검증. "잘 됐는지 봐줘"(일반 리뷰)가 아니라 "이게 틀렸다는 가정에서 근거를 찾아라"로 프레이밍한 별도 컨텍스트의 검증이다.

**발동 규칙 (v2.17 — 상시 자동 아님, risk-based)**: 적대검증은 노이즈 비용이 있다(confidence 3~4 지적·허수아비 반박이 늘 섞인다). 플레이북이 다른 단계에 강제하는 risk-based 원칙을 이 프로토콜 자신에게 적용한다 — **틀렸을 때 비용이 큰 판정 지점에서만 기본 발동**: ① 완료·게이트 통과 주장 ② freeze 경계(산출물 동결 직전) ③ 비가역 지점(제출 직전·외부 공개 직전). 그 외 중간 산출물은 아래 리뷰 모드 메뉴에서 더 싼 모드를 쓴다. **에이전트는 판정 지점이 아닌 곳에서 적대검증이 필요해 보이면 실행하지 말고 제안한다**(사람이 발동 결정).

**리뷰 모드 메뉴 (목적이 다르면 프레이밍이 달라야 한다)**:

| 모드 | 프레이밍 | 용도 | 수용 규칙 |
|---|---|---|---|
| **기계 게이트** (lint·golden set·재실행) | 결정론적 | 가장 싼 층 — 가능하면 항상 먼저 | 노이즈 0, 전부 수용 |
| **Peer review** (개선 지향 독립 리뷰) | "더 낫게 만들 지점을 찾아라" | 중간 산출물 품질 향상. 반증 프레이밍은 결함 탐지에 최적화돼 개선 아이디어를 잘 못 낸다 — 목적이 개선이면 이 모드 | **합리성 심사 후 선별 수용**(전면 수용 금지 — 확장 제안은 기본 deny) |
| **적대검증** (이 프로토콜) | "틀렸다는 가정에서 근거를 찾아라" | 판정 지점(발동 규칙 ①~③) 전용 | 아래 이의·보정 비대칭 |
| **프로바이더 독립 패널** | 표면 채점 | 외부 신호 프록시(라벨 없을 때) | 방향만 신뢰, 미세 등수 불신 |
| **사람 QC** (Phase 6) | 실사용자 | 사람만 잡는 5대 체크 | 위임 불가 |

**왜 일반 리뷰보다 나은가 (외부 근거, 접근일 2026-07-12)**:
- 같은 모델·같은 세션의 자기평가는 신뢰도가 낮다 — 외부 피드백 없는 자기수정은 성능을 오히려 악화시킬 수 있고(arXiv 2310.01798), 자기 출력에 관대한 self-bias가 실증됐다(CRITIC, arXiv 2305.11738). **자기비판에 의존 금지, 별도 모델/세션 필수.**
- 제작 맥락이 검증자를 오염시킨다 — 보안 코드리뷰에서 커밋 메시지 등 메타데이터를 제거하면 놓친 취약점 탐지가 68.75% 회복, "메타데이터 무시" 지시 추가 시 94%까지 회복(arXiv 2603.18740). "제작 과정을 아는 검증자는 제작자의 가정을 물려받는다"의 정량 근거.
- Anthropic 공식 권장: fresh 서브에이전트 컨텍스트의 리뷰어는 "diff와 기준만 보고, 변경을 만든 추론은 보지 않으므로 결과를 그 자체로 평가한다" (anthropic.com/engineering/effective-harnesses-for-long-running-agents).
- LLM-as-judge 편향 3종을 설계로 상쇄해야 한다: self-enhancement(자기 출력 후대) / position bias(순서로 판정 뒤집힘 → 비교 순서 무작위화) / verbosity bias(길면 후대 → 근거 밀도로 채점 명시).

**프로토콜**:

| 항목 | 규칙 |
|---|---|
| 선행 판정 | **scope drift 먼저** (gstack review): 결함 찾기 전에 "요구된 것을 정확히 만들었나 — 과다(겸사겸사 확장)도 과소(요구 누락)도 아닌가"를 별도 축으로 판정. |
| 컨텍스트 | **clean/minimal**: 산출물 + 채점 기준만. 제작 의도·과정 설명·커밋 메시지는 제거하거나 "무시" 명시 지시. 히스토리 없는 새 세션/서브에이전트에서. |
| 프레이밍 | 반증 기본값("떨어뜨릴 이유를 찾아라") + steelman(최선의 반박을 요구 — 허수아비 지적과 아부 둘 다 배제) + 칭찬 금지. |
| 출력 계약 | `[심각도 blocker/major/minor] (confidence: N/10) 파일:줄 — 내용` + 각 지적에 재현 시나리오 또는 재실행 증거 필수. **근거 위치를 인용 못 하는 발견 = 미검증**(검증자 환각 오탐 차단, gstack review). confidence 3~4는 부록으로 강등. |
| claim 분류 | 검증 대상 claim을 4모드로 분류 (gstack review): **DIFF-VERIFIABLE**(산출물로 증명 가능) / CROSS-REPO / **EXTERNAL-STATE**(재실행·재확인 불가 → 자동 UNVERIFIABLE) / CONTENT-SHAPE. 애매하면 UNVERIFIABLE 선호. 심사자 관점 대응: 심사AI가 검증 못 하는 claim은 무점 처리된다고 가정하고 EXTERNAL-STATE claim은 검증 가능한 표면으로 옮기거나 제거. |
| IRON LAW | **신선한 재실행 증거 없이 완료 주장 금지** (gstack ship): 산출물이 바뀌면 이전 검증 결과는 무효. 4대 합리화 문구를 명시 차단 — "should work now" / "confident하다" / "이미 테스트했다" / "사소한 변경이다". 필드 최다 고장 모드("전수 통과 주장이 재실행에서 깨짐")의 제작자 측 방지책. |
| 구성 | **cascade**: 경량 모델 1차 스크린 → 의심 케이스만 상위 모델 정밀(비용 40~60% 절감 근거 + 자체 실측: 한 대회에서 2차가 1차 놓친 오라벨·누락을 발견). 다인 검증은 단순 다수결보다 **근거 병기 + 조정(adjudication)**, N 늘리기보다 검증자 이질성(다른 모델·다른 관점 lens: 도메인규정/기술/사업)이 우선. |
| 프로바이더 독립 (외부 신호) | **clean context를 프로바이더 수준까지 민다** (실전 방향성 근거 42 · KG `independent-context-llm-evaluation-panel`). fresh 서브에이전트도 같은 모델 계열·같은 프롬프트 프레이밍을 공유할 수 있다 — 로컬 프로젝트의 LLM은 우리 프레이밍을 물려받아 self-bias가 산다. **프로젝트 context가 완전히 독립된 다른 프로바이더의 public LLM**(웹 인터페이스)에 산출물+평가기준만 주면 물려받을 경로 자체가 없다. 여러 프로바이더 = **패널**. **용법**: 공식 라벨(과거 수상/탈락)이 없어 Phase 1-2 캘리브레이션 게이트가 "라벨 없음"으로 막힐 때 이 패널을 **프록시 외부 신호**로 쓴다(자기평가보다는 낫다). **한계**: 방향은 믿되 정밀도는 아직 — 표본 1개 대회에서 자기평가가 *거꾸로* 맞힌 순위 역전을 패널이 바로잡았으나(근거 42) 미세 등수는 미검증. 이 패널도 **표면을 읽으므로** 정직하지만 표면상 약한 산출물을 과소평가할 수 있다 → 역이용해 "내 정직한 결과가 표면 읽기에도 이기게 프레이밍됐나"를 테스트하는 도구로도 쓴다(Phase 1-4·Phase 7 surface 프레이밍). |
| 시점 | 세션(머리) 경계 기본(산출물 동결 직전 = 수정 비용 최소 시점) + 패키징 후 1회 + 제출 전 리허설. ※"경계 vs 상시"의 직접 비교 연구는 공백 — 자체 관행 근거임을 명시. |
| 수용 규칙 | 결과 전면 수용 금지 — 지적의 합리성을 재평가하고 수용/기각 이유를 메인 로그에 남긴다. 기능 확장 제안은 기본 deny(기존 구멍 메우기만 accept). 개선 라운드 후 전체 재검증 대신 해당 우회만 스팟 확인. **단 재평가의 주체·방향은 아래 행이 묶는다** — 안 묶으면 이 행이 self-bias의 합법 통로가 된다. |
| 이의·보정 심사 (**비대칭**) | ★실전 최대 반증(근거 39). 위 "재평가"를 **제작자 쪽이** 하면 근거 9·10의 self-bias가 그대로 작동한다 — 지적을 기각하는 판정은 곧 자기 점수를 지키는 판정이기 때문이다. **방향 비대칭 규칙**: 지적 **수용·자기 하향**은 자기심사로 닫아도 된다(불리한 방향이라 편향이 반대로 작동 = 안전). 지적 **기각·자기 상향**은 **이해관계 없는 독립 심사자를 통과해야** 한다. 독립 심사자가 없으면 그 보정은 **적용하지 않고 원점수를 유지**한다("판정 보류"가 기본값). **실전 실패 서사**: 반론 9건을 제출자 쪽이 심사 → 대부분 인정·철회 → 한 슬롯에만 +2~3 → **원점수 동점(=가를 수 없음, 무정보)이 "마진 최대"라는 역방향 확신으로 전환** → 그 값이 통과 확률 추정의 명시적 근거가 되어 **최대 미스**를 낳았다. 보정이 원점수보다 예측을 **악화**시킨 사례 — 반론 내용 자체는 다수가 타당했다는 점이 핵심이다(결함은 내용이 아니라 **누가 판정했나**와 **한 방향으로만 쏠렸나**). |
| 검증 불가 출처 | 검증자가 접근 못 하는 출처(네이버 등)는 별도 카테고리로 표시 — 감점/중립 처리가 심사자마다 갈리므로 접근 가능한 1차 출처 짝이 있는지부터 본다. |

## 2-C. 팀전 변형 모듈 (오프라인 해커톤·팀 단위 대회에서 Phase 4~6에 겹쳐 적용)

> 본편은 1인 참가 전제. 팀전은 축이 하나 더 붙는다 — **"작업을 어떤 단위로 쪼개 누구에게 주나".** 아래 1~3은 v2.16에서 첫 팀전 실측으로 대폭 보강했다. **등급 = 가설**(1팀 관측 — 다음 팀 대회에서 재현 시 확정 승격). 실패한 접근과 성공한 접근을 나란히 둔다.

1. [판단] **분배 단위 = 역할이 아니라 "인터페이스로 격리된 독립 작업"** ★팀전 최대 레버 (근거 43·KG `tech-lead-independent-work-unit-decomposition`). 실전 실패: 4인 팀을 **보리스 체르니 5원형(역할 기반 분담)**으로 나누려다 실패했다. 원형은 *역할 정체성*(누가 개발/디자인)을 나누지만, 바이브코딩 MVP의 병목은 정체성이 아니라 **"독립적으로 떼어낼 수 있는 작업 단위의 부재"**다. 역할로 나누면 컨텍스트(리서치·아키텍처)를 가장 많이 쥔 한 사람에게 후속 작업이 계속 붙는 **구조적 인력(引力)** — "이 작업의 context를 내가 제일 잘 안다"는 매번 합리적인 판단의 누적이 곧 몰빵이다(게으름 아님).
   - **테크리더 모델(해법)**: 테크리더는 ① 완성될 결과물의 **구조를 이해**하고 ② 거기서 **독립 개발 가능한 조각**을 떼어 분배하고 ③ 조각들의 **인터페이스만 조율**한다. 모두가 개발하고 각자 맡은 조각의 리서치도 각자 한다. 고정 직무 분담이 아니다.
   - **분배 가능 신호**: 작업이 **Linear/GitHub 이슈로 독립적으로 안 쪼개지면** 아직 분배 단위가 아니다(인터페이스 미확정). 데이터·UI 조각도 "엑셀 파일 넘겨줘" 식 느슨한 핸드오프가 아니라 **입출력 스키마가 고정된 인터페이스**로 넘겨야 분배가 성립한다(실전: 구글시트 스키마를 줬어도 스키마가 헐거워 데이터 담당이 이탈, 결국 리더가 흡수 — 몰빵의 데이터 버전).
2. [판단] **병렬 리서치→머지의 결정권 쏠림 주의** (근거 43): "각자 AI로 1시간 리서치 → 공유 → 합의" 방식은 편하지만 **근거의 양 = 발언권**이 되기 쉬워 결정이 최다 근거 제시자(대개 리더)로 쏠린다. 문제는 "근거를 많이 댄 사람"과 "옳은 문제를 고른 사람"이 다를 수 있다는 것. 헤지 = 문제 후보를 **제안자와 분리된 기준(스코어카드)**으로 채점(§2-A 생성자≠판정자의 팀 버전) + 최종 선택 전 §2-C-4 외부신호 1회.
3. [결정론적] **도구 스택 표준 + 무엇이 실제로 통했나**: GitHub Organization(코드 + PR 리뷰 = 적대 검증의 팀 버전, 커밋 이력 = 기여 증거) / Linear(태스크·우선순위 — **테크리더가 분해한 독립 작업 단위를 이슈로 = 분배 가시화**, §2-C-1의 "쪼개지나" 테스트 도구) / Notion(공유 브리프·결정 로그·리서치 소스맵) / Slack(커뮤니케이션·알림). 원칙: **결정은 Notion 결정 로그에, 코드 판단은 PR 스레드에** — Slack에만 있는 결정은 로그 밖 결정과 같다.
   - **실측으로 통한 팀 패턴 2개**: ① **비개발 팀원의 QA/QC를 md 파일로 받아 에이전트에 위임** — 리더가 일일이 클릭 검수하는 수고를 없애고 비개발자를 실제 기여로 전환(효과적). ② **GitHub write 권한 초대**로 팀원을 실제 코드 작업에 합류시켜 기여 이력 남김.
4. [판단] **외부신호로 자기평가 교정** (라이브 팀 대회 필수 — 근거 42·43): 팀은 "우리 게 낫다"는 **집단 직감**에 더 쉽게 빠진다(자기평가 편향의 팀 버전). 실전 3회 연속 이 직감이 실제 순위와 어긋났다. 다른 팀 발표를 보며 "우리가 낫다"를 채점 근거로 쓰지 말 것. 대신 **§2-A 독립 프로바이더 LLM 패널**(프로젝트 context 없는 다른 회사 웹 LLM에 산출물+평가기준만)로 우리·경쟁 산출물을 함께 채점해 상대 위치를 보정 + 다른 팀의 **설문 규모·검증 깊이를 관측**해 외부 기준으로 삼는다.
5. [결정론적] 킥오프 계약의 팀 버전: 평가 축·차별화 축·게이트 순서 + **결정권 소유자**(동률 시 누가 정하나) + **테크리더 지정** + **발표 파이프라인·발표자 사전 지정**(Phase 7-9)을 첫 미팅에서 문서로 합의. 로그·제출 규정(기여 로그 요구 등)이 있으면 팀원 전원의 수집 셋업을 Phase 1에서 검증.

## 2-E. 게이트 골격 템플릿 (Phase 4에서 참조 — 게이트 1개 정의용)

> 게이트 **1개**의 범용 골격. 새 게이트를 정의할 때마다 복사해 채운다. 빈칸은 비워두지 말고 `pending`·`gap`·`{{다음 머리}}에서 닫음` 중 하나로 상태를 적는다. (원형: 유래 대회 한 슬롯의 게이트 문서군에서 추출. 예전 `assets/gate-skeleton-template.md`를 이 절로 흡수했다.)

**1개 게이트 골격**:
- **목적**: 이 게이트가 닫아야 하는 질문 1개
- **입력**: 이전 게이트가 넘긴 산출물·결정 + 이 게이트에서 새로 확인할 공개 자료(URL·경로)
- **anti-scope**: 다음 게이트로 미루는 일 / 절대 다루지 않을 판단 영역(예: 최종 전문 판단 자동화)
- **산출물 skeleton**: 파일 경로 + 빈칸/필드 구조(못 채운 필드는 `pending`) + (사람이 훑을 view 필요 시 html)
- **완료 선언**: `Sequence / Current gate / Completed / Open / Do not reopen / Next first action / Flow map updated`
- **다음 게이트 입력**: 다음 게이트가 이어받는 파일·결정 목록

**머리별 게이트 목록 예시** (1대회 인스턴스 — 대회/슬롯마다 이름·개수 조정, `A0.5`·`Pfinal` 등은 머리 안의 게이트 ID이지 Phase 코드 아님): 4머리를 게이트로 펴면 대략 **A 문제**(요구·로그 → 소스맵 → 문제 후보·스코어카드 → scope 계약 → B brief) → **B 제작**(근거 재확인 → 구현 계약 → thin slice → fixture·검증 matrix → C brief) → **C 사용자**(dogfood → 결함 수정 → 최종 readiness) → **P 패키징**(로그 동결 → zip·정합 → 제출 smoke). 각 머리는 3~6개 게이트, "이전 산출물만 입력·앞 결정 재론 금지" 원칙 유지.

## 2-F. 용어집 (이 문서의 예약어 — 세션·팀원·3자가 같은 말을 같은 뜻으로)

> v2.16에서 이월된 finding. 아래 용어는 이 문서 전체에서 **한 가지 뜻으로만** 쓴다 — 특히 플래닝/스캐폴딩/아키텍처는 실전에서 서로 뭉개져 쓰이며 소통 비용을 만들었다. 새 용어를 발명하기 전에 이 표를 먼저 확인한다.

| 용어 | 이 문서에서의 뜻 | 혼동 주의 (무엇과 다른가) |
|---|---|---|
| **플래닝** | Phase 0~4 전반부 활동 전체 — 참가 결정→심사 역설계→문제·해법 확정→작업 설계. 노력의 70~80%가 들어가는 구간 | 스캐폴딩은 플래닝의 산출물 중 "작업 실행 구조물"만 가리킨다 — 플래닝 ⊃ 스캐폴딩 |
| **스캐폴딩** | 작업을 실행 가능하게 만드는 구조물 — 게이트 순서·산출물 skeleton·킥오프 계약·머리/세션 경계 (Phase 4 산출물, 검증은 Phase 4-6) | **작업**의 구조다. 제출물 내부 구조(아키텍처)가 아니다 — 스캐폴딩 결함은 작업이 산으로 가고, 아키텍처 결함은 제품이 망가진다 |
| **아키텍처** | 제출물(솔루션) 자체의 내부 구조 — 컴포넌트·데이터 흐름·기술 선택 (Phase 3 해법·Phase 5 구현 소관) | 스캐폴딩과 구분. "구조를 검증했다"는 발화는 둘 중 무엇인지 명시해야 한다 |
| **머리** | 인지 관점 태그 — 문제 / 제작 / 쓰는 사람 / 패키징 / 심사자. 관점이 바뀌면 세션을 나눈다 | 세션과 1:1 강제 아님(v2.14에서 "시퀀스=한 세션" 은퇴). 작업량 분배 단위도 아니다 |
| **세션** | 컨텍스트 경계 단위. 머리 전환 시 분리(오염 방지), 경계마다 handoff, resume 금지 | Phase와 다름 — 한 세션이 여러 Phase를 다룰 수 있다(예: 문제 머리 세션이 Phase 2~3 수행) |
| **게이트** | 완료 조건이 명시된 실행 체크포인트(골격은 §2-E) — 한 대회 인스턴스에서 머리 안에 3~6개 | Phase와 다름 — Phase는 방법론의 순서(범용), 게이트는 이번 대회에 인스턴스화된 체크포인트(`A0.5` 같은 ID는 게이트) |
| **슬롯** | 한 대회 안의 독립 제출 단위(부문/기업/트랙). 슬롯 루프(Phase 2~8)의 반복 단위 | 대회 ≠ 슬롯 — 멀티 슬롯 대회는 Phase 1까지 공유, Phase 2부터 슬롯별 독립 |
| **킥오프 (계약)** | 슬롯 첫 작업 세션을 여는 계약 프롬프트 — 게이트 순서·산출물 skeleton·평가 축·승인 규칙 전부 탑재 (Phase 4-1) | 단순 "시작 메시지"가 아니라 계약 — 계약 없는 킥오프는 즉흥 재작업을 유발했다(실측) |
| **역주입** | Phase 9 회고 결과를 이 플레이북에 반영해 버전을 올리는 것 — 추가만이 아니라 강등·삭제(가지치기) 포함 양방향 | 회고 문서 작성 ≠ 역주입 — 플레이북 본문이 바뀌고 버전이 올라야 역주입 |
| **가지치기** | 역주입의 감량 방향 — 죽은 규칙·비대 절을 강등·삭제 판정(Phase 9-4 규율, 삭제 branch는 3개+ 대회 후 발동) | "반복"은 대상 아님 — 맥락이 다른 spaced reinforcement는 유지(v2.19 유지판정) |

## 4. 회고 → 플레이북 역주입 (Phase 9 산출물 형식 · 대회마다 갱신)

> 이 절은 **템플릿**이다. 매 대회 Phase 9에서 회고를 이 형식으로 남기고, 나온 gap을 이 플레이북 해당 단계에 역주입한 뒤 버전을 올린다(사전 자산화 루프의 완성). 아래 "최초 정비 회고"는 이 플레이북이 유래한 1개 대회 관측을 익명화한 실제 기입 예시 — 패턴을 보여주려 그대로 둔다. 그 뒤의 **"발표 후 갱신"**은 같은 대회의 결과가 나온 뒤 그 회고를 대조한 델타표 예시다. 두 블록을 나란히 두는 이유가 곧 이 절의 요점 — **발표 전 회고는 가설이고, 결과가 나오기 전까지 그 회고의 처방을 규칙으로 굳히면 안 된다**(실제로 발표 전 회고의 처방 하나는 결과 확인 후 부분 철회됐다).

**회고 기입 틀** (대회마다 새로 채움):

| 항목 | 내용 |
|---|---|
| 부족했던 것 (gap) | 무엇이 감점·재작업·사후발견을 만들었나 |
| 근본 원인 | 증상(점수)이 아니라 원인(루브릭 설계·표본 편향·검증불가 처리)까지 |
| 역주입 위치 | 이 플레이북의 어느 단계·게이트로 반영했나 (→ Pn) |
| 유지·자산화 | 잘 작동해서 다음 대회로 넘길 것 |

**최초 정비 회고 (1개 대회 관측, 익명화 예시)** — 부족했던 것 → 역주입:

1. **심사 루브릭이 Phase 1에 없었다** (→ Phase 1-2 신설 + 게이트별 미니 채점). 심사 시뮬레이션을 제출 후에 해서 발견이 수정 비용 최대 시점에 도착했다.
2. **전략·스캐폴딩이 로그 밖 외부 세션에 있었다** (→ Phase 4 3층 로그 전략). 가장 공들인 사고가 심사에 안 보였다.
3. **클린 환경 재현 검증이 게이트가 아니었다** (→ Phase 7-4). 재현 결함이 제출 후에 발견됐다.
4. **git 미활용** (→ 체크리스트). 작업 이력이 로그에만 있고 커밋 그래프에 없었다.
5. **9~10점 축 3개를 몰랐다** (→ Phase 8-2). 삼각측량·k회 통계 통제·독립 라벨은 전부 마감 전 실행 가능했던 기법.
6. **동일 방법론 3연발 리스크** — 멀티 슬롯 제출 시 방법론 다양화(하나는 좁은 완결형, 하나는 다른 구조) 검토. 심사가 어느 방향을 칠지 모를 때의 헤지. **[가설 — 결과로 corroborate됨, 근거 41]**: 실제로 전 슬롯이 같이 떨어졌다(상관 실패). N 슬롯 병렬 제출은 다양화가 아니다 — 방법론·문제 선택 스타일·도구 구성을 공유하면 공통 인자 하나가 전부를 같이 떨어뜨려 사실상 **1번의 베팅**이 된다. 다만 관측 3건 + 피드백 0줄이라 **가설 등급 유지**(차단 게이트로 승격 금지). 이 항목의 진짜 교훈은 헤지 내용이 아니라 **리스크를 발표 전에 적어놓고 제출 구성은 안 바꿨다**는 것 — 회고에 적는 것과 계획을 바꾸는 것은 다르다.
7. **유지·자산화한 것**: 킥오프 계약(슬롯 간 진화로 실증), 적대 검증(→ §2-A로 프로토콜화), claim graph, 좁은 문제 완결, 재실행 생존형 검증(필드 4/9가 여기서 죽었다), 업그레이드 로그 수집, 사람 dogfood.

**같은 대회 — 발표 후 갱신 (델타표 실제 기입 예시, Phase 9-5 형식)**

위 1~7은 **발표 전**에 쓴 회고다. 결과(전 슬롯 미진출, 1개만 예비후보)가 나온 뒤 대조하니 항목별로 운명이 갈렸다 — 이 델타가 Phase 9-5가 요구하는 "무엇이 개선이고 무엇이 재발인가"의 실제 판정이다.

| 이전 회고 항목 | 발표 후 판정 | 처분 |
|---|---|---|
| #5 9~10점 축 3개 (→ Phase 8-2) | **부분 철회.** 3축은 전부 *"이 루브릭에서 8→10"* 의 답인데, 그 루브릭이 관측 가능한 유일한 순위를 역방향으로 맞혔다. 미검증 루브릭에서 8→10 투자는 잘못된 목적함수 최적화. | ①·②는 Phase 1-2 캘리브레이션 게이트 통과를 선행 조건으로. ③(독립 라벨)만 존치 — 근거가 "점수"에서 "독립성 요구"로 격상 |
| #6 동일 방법론 3연발 리스크 | **corroborate(가설 유지).** 전 슬롯이 같이 떨어졌다(상관 실패). 발표 전에 리스크를 **적어놓고 제출 구성은 안 바꾼 것**이 실제 손해였다. 단 관측 3건 — 확정 불가. | 가설 라벨 유지(§4-6·Phase 1-8④). 규칙·차단 게이트로 승격 **금지** |
| #7 재실행 생존형 검증 | **유지.** 필드 공통 고장 모드(주장한 검증이 재실행에서 깨짐)에서 우리 제출물은 여전히 생존했다 — 이 축은 결과와 무관하게 자산. | 존치 |
| (신규) **자기 반론을 자기가 심사** | **확정 결함.** 최대 미스의 직접 원인. 결과 없이 회고 문서 내부 대조만으로 성립. | → §2-A "이의·보정 심사(비대칭)" 신설 · Phase 8-2 · Phase 9-3 (근거 39) |
| (신규) **루브릭을 검증 없이 순위·확률에 사용** | **확정 결함.** 두 독립 자체 루브릭이 같은 슬롯을 같은 방향으로 오판 = 계통 오차. | → Phase 1-2 캘리브레이션 게이트 신설 · Phase 9-1 용어 교정·확률 게이트 (근거 40) |

**이 델타의 메타 교훈** — 발표 전 회고는 *"이 평가 구조로 제출 전 수정→재채점 루프를 돌렸으면 만점에 근접했을 것, 다음 대회의 기본 프로세스로"* 라고 결론냈다. 결과는 그 결론을 뒤집었다: **순위를 역방향으로 맞히는 평가 구조로 루프를 더 돌렸다면 더 빠르게 틀린 방향으로 갔을 것이다.** 루프를 기본 프로세스로 채택하기 전에 필요한 건 루프를 더 도는 게 아니라 **루프의 목적함수를 라벨된 데이터로 검증하는 것**이다. 회고가 "더 열심히 하자"로 끝나면 그 회고는 대개 아직 원인에 안 닿은 것이다.
