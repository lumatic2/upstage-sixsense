# HANDOFF — 가져다 쓰려는 사람에게

> 2026-07-20 기준. 각자 버전을 만드는 팀원이 **자기 코드에 옮겨 쓸 수 있게** 쓴 문서다.
> "무엇을 만들었나"는 [README](../README.md)에, 여기엔 **어디서 막혔고 어떻게 풀었나**를 적는다.
> 이 레포를 통째로 가져갈 필요는 없다. 필요한 조각만 떼어 가면 된다.

---

## 0. 제일 먼저 알아야 할 것

이 프로젝트에서 진짜 어려웠던 건 **Upstage API를 호출하는 일이 아니었다.** 그건 반나절이면 된다.
어려웠던 건 그 다음이다 — **파싱 결과를 믿을 수 있게 만드는 일.**

파싱 정확도가 76.4%라는 건, 넷 중 하나는 틀린 채로 서비스에 나간다는 뜻이다.
그래서 이 레포의 절반은 파이프라인이고 나머지 절반은 **"틀린 걸 걸러내는 장치"**다.
심사 배점도 거기 몰려 있다(파이프라인 20 + 깊이 15 = 35점).

가져갈 게 하나뿐이라면 이걸 가져가라: **파싱 결과를 자동으로 저장하지 마라.**

## 1. 단계별로 막혔던 것과 해법

### 1-1. Document Parse — 좌표를 안 받으면 이름과 가격이 짝이 안 맞는다

기본 호출로 받으면 텍스트는 나오는데 **어느 가격이 어느 메뉴 것인지 알 수 없다.**
메뉴판은 왼쪽에 이름, 오른쪽에 가격이 떨어져 있는 2열 레이아웃이 많아서다.

```js
// api/parse-menu.js
ocr: "force",          // 이미지 안 텍스트를 강제로 다시 읽음
coordinates: true,     // 각 요소의 좌표 → 같은 줄끼리 묶을 수 있다
```

`coordinates: true`로 받아 y좌표가 비슷한 것끼리 묶는 로직이 `api/_lib/extract-menu.js`에 있다.

**표면별 정확도가 극단적으로 다르다** (`experiments/parse-poc/accuracy.md`):

| 사진 종류 | 정확도 |
|---|---|
| 인쇄 책자를 정면에서 | 100% |
| 유리창 세로 시트 | 100% |
| 타일 카드 (`3.0` = 3,000원 표기) | 66.7% |
| 손글씨 화이트보드 | 57.1% |
| 저해상도 웹 이미지 | 20.0% |

→ **사진을 어떻게 찍느냐가 모델보다 중요하다.** 수집 가이드에 "정면·밝게·가격 함께"를 넣어라.

후처리로 0% → 62.5% → 76.4%까지 올렸다. 실제로 효과가 컸던 것:
- OCR alt 텍스트 승격 (이미지로 인식된 영역에서 텍스트 건지기)
- 천원 단위 표기 복원 (`3.0` → 3,000)
- 공백 낀 숫자 (`4 000` → 4000)
- 한 줄에 여러 항목이 들어간 경우 분리

### 1-2. Solar — 모델 선택이 실측으로 갈렸다

`solar-mini`가 싸고 빠르지만 **"둘이서 2만원" 같은 나눗셈에서 예산을 틀렸다**(1인 1만원으로 못 나눔).
그래서 질의 구조화는 `solar-pro2`를 쓴다. 실측 왕복 0.7~1.3초.

```js
// 구조화 출력을 반드시 쓴다 — 자유 텍스트로 받으면 파싱이 또 문제가 된다
response_format: { type: "json_schema", json_schema: { name: "...", strict: true, schema: {...} } }
```

**타임아웃 폴백은 반드시 넣어라.** 데모 중에 외부 API가 느려지면 검색 자체가 죽는다.
`api/parse-query.js`는 실패 시 정규식 파서로 떨어지고, `api/recommend.js`는 이유 생성이 실패하면
데이터 기반 템플릿 문장으로 떨어진다. **추천 루프는 어떤 경우에도 죽지 않는다.**

### 1-3. 근거 검증 — 생성한 모델에게 채점을 시키면 안 된다

전용 Groundedness Check 모델은 2026-07 현재 계정 모델 목록에 없다(retired).
같은 계약을 **모델 분리**로 구현했다:

- 이유 생성 = `solar-pro2`
- 그 이유가 실데이터에만 근거하는지 판정 = `solar-mini` (독립 판정자)
- 판정이 `notGrounded`면 그 문장을 버리고 데이터 사실로 교체

화면의 "근거 검증됨" 배지가 이 판정 결과다. **환각을 막았다는 주장이 아니라 막는 동작을 보여준다.**
구현은 `api/recommend.js` 하단.

> 함정: 타임아웃이 짧으면 판정이 조용히 실패하고 배지가 안 뜬다. 실측 5%에서 그랬다.
> 생성 5s / 판정 3.5s로 올려 해결했다.

### 1-4. 검수 게이트 — 이게 없으면 파이프라인이 주장으로만 남는다

제보로 들어온 데이터는 시트에 `검수=대기`로 쌓이고, 운영진이 `/review.html`에서
**원본 사진과 나란히 놓고** 행마다 확인/제외/수정을 찍는다. `확인`이 찍힌 행만 서비스로 나간다.

```js
// api/_lib/sheet-data.js — 이 한 줄이 게이트다
.filter((m) => m.name && m.price && m.review === "확인")
```

**게이트는 우회 경로가 하나만 있어도 무너진다.** 우리는 실제로 그걸 겪었다:
- `api/data.js`에 Supabase 우선 경로가 있었는데 거기엔 게이트가 없었다. env 하나가 켜지면
  미검수 데이터가 그대로 나가는 구조 → 읽기 경로를 시트 하나로 단일화해 분기를 삭제했다.
- 화면 4개 중 3개만 고치고 완료로 볼 뻔했다(§3 참조).

### 1-5. 구글 시트를 DB로 쓰기 — 팀 협업에는 이게 낫다

DB를 쓰면 팀원이 데이터를 못 본다. 시트는 **누구나 열어서 눈으로 확인하고 고칠 수 있다.**
검수라는 행위 자체가 사람이 하는 일이라 시트가 잘 맞았다.

- **읽기**: 공개 gviz CSV — 키 없이 서버에서 읽힌다. `api/_lib/sheet-data.js`
- **쓰기**: Apps Script 웹훅 — 권한이 필요해 서버 함수에서만. `api/_lib/sheet-write.js`, `scripts/apps-script/`

> 함정 ①: 시트 기록은 **항상 `USER_ENTERED`**로. `RAW`로 쓰면 gviz가 헤더를 흡수해버린다.
> 함정 ②: Apps Script cold start가 10초를 넘긴 적이 있다. 타임아웃 20초로 잡았다.
> 함정 ③: **행 삭제 기능을 안 만들었다.** 그래서 테스트 제보를 넣으면 영구히 남는다. 조심.

## 3. 우리가 실제로 저지른 실수 — 같은 데 빠지지 마라

**① 화면 하나만 고치고 다 고쳤다고 믿었다.**
곁들임(공기밥 1,000원)이 한 끼처럼 추천되던 버그를 `api/recommend.js`에서 고치고 완료로 봤다.
독립 검증을 돌렸더니 **홈 화면·검증 페이지·구 테스트 페이지가 각자 자기 필터를 갖고 있어서**
같은 예산에 서로 다른 답을 내고 있었다. 두 번의 검증에서 두 번 반증당한 뒤에야 닫혔다.
→ **판정 로직은 한 곳에만 두고 데이터에 플래그로 실어 보내라.** 화면마다 규칙을 다시 쓰면 반드시 어긋난다.

**② 배포 파일 목록을 손으로 관리했다.**
Vercel 배포는 "목록에 적힌 파일이 전부"다. 새로 만든 3개 페이지가 목록에 없어서,
그대로 한 번 더 배포했으면 **라이브에서 그 페이지들이 삭제될 상태**였다.
→ 목록을 손으로 들지 말고 디렉터리를 재귀 수집해라(`scripts/deploy-staging.mjs`).

**③ 유닛 테스트 통과를 "동작한다"로 착각했다.**
API 응답만 확인하면 통과처럼 보이는데, 실제 브라우저를 열면 목록과 지도가 안 고쳐져 있었다.
→ **배포본을 실브라우저로 열어봐라.** 모바일 폭(390px)도 반드시. 우리 헤더는 거기서 무너져 있었다.

**④ 발표 자료가 코드보다 먼저 낡는다.**
7/17에 쓴 덱이 7/20에 이미 틀렸다(URL·수치·없어진 계획). 더 나쁜 건 **그 사이에 만든 제일 좋은 기능
두 개가 덱에 아예 없었다는 것.** → 기능 하나 끝낼 때마다 덱을 한 줄이라도 고쳐라.
