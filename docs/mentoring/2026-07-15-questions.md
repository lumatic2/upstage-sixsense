# 멘토링 사전 질문 (7/16~17, 30분)

> 한입지도 — 성균관대 명륜캠 학생 대상, 상황(예산·거리·목적) 기반 맛집+학식 추천 서비스
> 현재 상태: 배포된 데모(hanipmap.vercel.app) + 식당 96곳 수집 + 제보/승인 흐름 동작.
> 다음 단계: 메뉴판 사진을 Upstage Document Parse로 파싱해 메뉴·가격 데이터를 채우고, Solar로 자연어 검색을 고도화하려는 시점.

## 질문 (우선순위 순)

**1. 데이터가 세상에 없을 때 어떻게 시작하나요? (데이터 콜드스타트)**
저희 핵심 데이터는 식당 메뉴·가격인데, 조사해보니 어떤 공개 API도 이걸 제공하지 않았습니다(네이버 지역검색에도 가격 필드가 없음). 그래서 ① 초기 수동 입력 ② 사용자 제보 ③ 메뉴판 사진을 Document Parse로 파싱하는 3가지로 모으려 합니다. 이 조합이 현실적인가요? 실무에서는 이런 콜드스타트 문제를 어떤 순서로 푸나요?

**2. AI 파싱 결과를 어디까지 믿어야 하나요?**
메뉴판 사진 파싱 결과가 틀릴 수 있습니다. 전부 사람이 검수하면 제보가 느려지고, 전부 자동 반영하면 데이터가 오염됩니다. 지금은 "파싱 결과를 폼에 미리 채워주고 사용자가 수정 후 제출 → 운영자 승인" 방식으로 잡았는데, 이런 자동/수동의 경계선을 잡는 기준이 실무에 있나요?

**3. 남은 9일, 무엇부터 버려야 하나요? (우선순위)**
데모데이(7/25)까지 계획: 파싱 PoC 검증 → 학식 크롤러 → Solar 검색 통합 → 발표 준비. 시간이 모자라면 어떤 걸 먼저 포기하는 게 맞나요? 반대로 "이건 무조건 지켜라" 하는 건 뭔가요?

**4. LLM을 서비스에 넣을 때 백업은 어떻게 두나요?**
자연어 검색("8천원 이하 혼밥")을 지금은 정규식으로 파싱하는데 Solar로 바꾸려 합니다. LLM 응답이 느리거나 이상하게 나올 때를 대비해 기존 정규식을 폴백으로 남겨두는 구조를 생각했는데, 이게 맞는 방법인가요? 타임아웃은 몇 초 정도가 적당한가요?

**5. 수동 운영은 어디까지 허용되나요?**
제보 승인을 지금 Supabase 관리 화면에서 손으로 합니다. 데모 단계에서는 이대로 가도 괜찮은가요, 아니면 심사 관점에서 간단한 관리자 화면이라도 있어야 하나요?

**6. 발표 시연: 라이브 vs 녹화?**
라이브 시연 중 외부 API(Upstage, 네이버)가 느려지거나 죽으면 어떻게 하나요? 실무자분들은 데모를 어떻게 준비하시는지 궁금합니다.

---

*참고 — 저희가 이미 확인한 것들 (질문 배경):*
- 성균관대 학식 페이지는 HTML 표라서 Document Parse 없이 크롤링으로 충분 → 학식은 크롤러, 사진은 파서로 역할을 나눔
- Document Parse는 $10 무료 크레딧으로 약 1,000장 파싱 가능 → 비용은 문제 아님
- 유사 서비스(거지맵 등)는 수동 크라우드소싱 — 저희 차별점을 "데이터 생산 비용을 낮추는 파이프라인"으로 잡음
