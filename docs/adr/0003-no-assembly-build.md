# ADR-0003 — 본선 조립본(hanipmap 이식)을 만들지 않는다

- 상태: 채택 (2026-07-20)
- 대체: DR3 (`plans/2026-07-19-dr3-assembly.md`) — 폐기
- 관련: ADR-0001(문제·해법 형태) · `docs/team/2026-07-18-decisions.md` §6

## 맥락

7/18 팀 미팅에서 방법론이 "각자 서비스"로 전환됐다. 이때의 해석은 **각자 만들되 대표 조립본(팀원 레포 hanipmap)에 이식해 그것을 제출한다**였고, 그 전제로 DR3(본선 조립 — API·스키마를 hanipmap 에 PR 이식, 계약 문서, 팀 온보딩)을 계획했다.

2026-07-20 사용자 확정으로 이 전제가 사라졌다. 실제 팀 운영은 다음과 같다:

- 팀원 각자가 자기 버전을 끝까지 만든다.
- **2026-07-23(목) 21:00 미팅**에서 각 버전을 비교하고, 무엇을 제출물로 삼을지 **합의로 정한다**.
- 즉 제출물은 사전에 정해져 있지 않다. 이식할 "대표 버전"이라는 대상이 존재하지 않는다.

## 결정

**hanipmap 으로의 이식 작업을 하지 않는다.** DR3 을 폐기하고 그 자리를 두 milestone 으로 대체한다.

- **DR6 — 내 버전 완결**: 미팅에서 선택될 수 있는 상태로 이 레포의 서비스를 닫는다.
- **DR7 — 팀 공유**: 데이터·코드·개발 지식을 팀원이 *자기 버전에* 가져다 쓸 수 있는 형태로 내놓는다. 공유 형식은 **GitHub 레포 + 웹사이트 링크** 두 가지(사용자 확정).

DR3 step-3(발표 자료 실물 정합)만 유효하므로 DR6 으로 승계한다.

## 근거

1. **이식할 대상이 없다.** 대표 버전이 미팅 후에 정해지므로, 미팅 전 이식은 어느 레포로 갈지 모르는 채 하는 작업이다. 잘못 고르면 통째로 버려진다.
2. **이식은 설득력을 만들지 않는다.** 미팅의 판단 재료는 "돌아가는 서비스"지 "남의 레포에 들어간 PR"이 아니다. 같은 시간을 내 버전 완성도에 쓰는 편이 제출물로 선택될 확률을 직접 올린다.
3. **공유 목적은 이식 없이 달성된다.** 팀원이 필요한 건 내 코드가 자기 레포에 병합되는 것이 아니라, 무엇을 어떻게 만들었는지 보고 **자기 버전에 반영할 수 있는 것**이다. 공개 레포와 라이브 사이트 링크로 충분하다.
4. **일정.** 미팅(7/23 21:00)이 빌드업 마감(7/24) 직전이므로 실질 마감은 미팅이다. D-3 에 남의 레포 PR·리뷰·배포 사이클을 도는 것은 사고 여유가 없다 — `plans/horizons/demoday-run.md` 프리모템 시나리오 3("조립·발표가 마지막 날로 밀림")이 예측한 위험이 실제로 임박한 상태였다.

## 결과

- horizon `demoday-run` 의 닫는 기준 3번(hanipmap 배포 URL E2E)은 **팀 공유 패키지 도달 가능성**으로 교체한다.
- 이 레포의 배포본(https://sixsense.askewly.com)과 공개 레포(https://github.com/lumatic2/upstage-sixsense)가 유일한 산출 표면이 된다.
- 제출물이 내 버전으로 합의되면 그때 팀 레포 결정이 다시 열린다. 그 판단은 미팅 후 별도 ADR 로 남긴다.

## 검토 후 채택하지 않은 대안

- **일단 이식해두고 미팅에서 고른다**: 대표 버전 미정 상태의 선투자. 버려질 확률이 절반 이상이고, 그만큼 내 버전 완성도가 깎인다.
- **팀원 레포에 PR 만 열어두고 머지는 미팅 후**: PR 유지 비용(리베이스·리뷰 응대)이 D-3 에 발생한다. 코드는 공개 레포에서 그대로 읽을 수 있으므로 PR 형태여야 할 이유가 없다.
