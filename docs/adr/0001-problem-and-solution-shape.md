# ADR-0001 — 문제 정의 유지 + 메뉴판 파싱 중심 솔루션

- 날짜: 2026-07-15
- 상태: 승인 (팀 합의 + 사용자 확정)

## 맥락

리서치 4갈래(`docs/research/2026-07-15-*.md`)의 충돌 게이트는 기존 문제 정의("예산 필터로 학식+주변 식당 탐색")에 **KILL 판정**을 냈다 — 거지맵(131만 이용자)이 예산 필터+학식+대학가 지도를 이미 제공. 대안으로 "예산 상태 기반 식사 계획"(C2)이 제안됐으나, **팀 합의 결과 기존 정의를 유지**하기로 결정했다. 팀원 데모 한입지도(github.com/ljb0138/hanipmap)가 이미 존재한다: 자연어 질의(정규식 파싱) → 예산·거리·태그 필터 → 지도 추천 + 수동 제보 폼(메뉴:가격 타이핑 → Supabase).

## 결정

1. **문제 정의는 팀 합의대로 유지**한다. 충돌 리스크(거지맵)는 아래 2로 방어한다.
2. **차별화 축 = 데이터 파이프라인**: 제보 폼의 수동 타이핑을 **"메뉴판 사진 업로드 → Upstage Document Parse → 메뉴·가격 자동 구조화"**로 교체한다. 근거: 주변 식당 가격은 구조화 데이터가 부재하고(네이버 Local API에 가격 필드 없음 — 실행 확인) 메뉴판 사진에 갇혀 있다. 거지맵의 크라우드소싱은 수동 입력이다 — 사진 한 장으로 낮춘 제보 비용이 차별점.
3. **학식은 크롤링**한다 (Document Parse 금지) — 성대 식단은 이미 HTML 표+가격 텍스트임을 실사로 확인. 억지 파싱은 심사에서 "목적에 적합한 제품 적용"(10점) 감점 사유가 된다.
4. **Solar**는 ① 자연어 질의 구조화(정규식 `parseBudget` 대체 — "과선배가 사는 날" 같은 상황 해석) ② 추천 이유 생성에 쓴다.
5. 영수증 파싱(잔여 예산 상태 축)은 **범위 제외** — 7/24까지 범위 초과 위험. Next Candidate로 보류.

## 결과

- Upstage 2종(Document Parse + Solar) 사용 — 실격 요건 해소 + "적합한 적용" 서사.
- 루브릭 35점 구간(파이프라인 20 + 깊이 15)의 실체: 사진→파싱→검증→구조화 저장→예산 제약 추천.
- 수용한 리스크: (a) 거지맵과의 표면 유사성 — 발표에서 "데이터 생산 방식의 차이"로 정면 대응 필요 (b) 메뉴판 사진 파싱 정확도 미검증 — 최우선 PoC로 검증 (c) 기각된 C2 대비 문제 정의의 독창성 약함.

## 기각한 대안

- **C2 예산 상태 기반 식사 계획**: 증거는 더 강했으나(클러스터 A) 팀 합의로 기각. 영수증 축만 Next Candidate로 보존.
- **학식 식단표를 Document Parse로 파싱**: 데이터가 이미 구조화돼 있어 억지 — 실사로 기각.
