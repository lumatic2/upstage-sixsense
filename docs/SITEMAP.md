# SITEMAP — 페이지 구조 설계도

> 작성 2026-07-20 · **전면 현행화 2026-07-22 (DR11 step-2)**
> 이 문서는 **무엇이 어느 화면에 들어가는가**의 SSOT다. 화면 구현 전 여기에 먼저 쓴다.
> 상위 정의: `docs/PRD.md` · 기술 스택: `docs/TRD.md` · 결정: `docs/adr/` · 배점: `CLAUDE.md` 루브릭 표

## ⚠ 이 문서가 낡으면 화면이 거짓말을 한다

2026-07-19~21 사이 코드를 여섯 번 크게 고치는 동안 이 문서를 한 번도 안 고쳤다. 그 사이
`docs/adr/0004-personalization.md` 가 순서 개인화를 폐기했는데 **라이브 `/about.html` 은 그 기능을
계속 광고**하고 있었고, 이 문서는 자기 안에서 모순됐다(§구 블록3 "편집 가능" vs §7-2 "제보자는
고치지 않는다"). 전수 감사 기록 → `research/2026-07-21-dr11-design-doc-audit.md`.

**규약**: ADR 로 결정을 뒤집으면 그것을 설명하던 이 문서와 화면 문구를 **같은 changeset 안에서**
고친다. 나중에 하면 반드시 안 한다.

## 0. 설계 원칙 (이 구조를 고른 이유)

1. **루브릭이 페이지를 정한다.** 파이프라인 20 + 깊이 15 = 35점이 최대 배점이므로, 그 35점의
   실체(사진→파싱→검수)가 심사위원이 보는 화면 안에 있어야 한다.
2. **페이지 = 심사위원의 질문 1개.** 겹치면 합치고, 답이 없으면 페이지를 만들지 않는다.
3. **공개 5페이지 상한.** 운영진 전용 화면은 이 상한에 세지 않는다(사용자 여정에 없으므로).

## 1. 사이트맵 (현행)

```
공개 (사용자 여정 5)
/                    랜딩            "이게 뭐고 왜 필요한가"
├── /app.html        메뉴 찾기        "그래서 뭘 해주는데"
├── /contribute.html 사진 제보        "데이터는 어디서 오는가(직접)"
├── /about.html      데이터 파이프라인 "왜 ChatGPT가 아니라 이건가"
│     └── #upstage   Upstage AI      "어디에 무엇을 왜 썼나"   ← nav 5번째 항목의 목적지
└── /verify.html     파이프라인 검증   "그 주장이 지금 진짜인가"

비공개 (운영진 전용 · 어디에도 링크하지 않음)
    /review.html     제보 검수        "AI가 뽑은 걸 사람이 승인한다"
    /test.html       개발용 원본      verify.html 승격 실패 시 원복 경로 — 배포에서 제외
```

- **헤더 nav = 5개** — 홈 · 메뉴 찾기 · 사진 제보 · 데이터 파이프라인 · Upstage AI.
  마지막은 페이지가 아니라 `/about.html#upstage` 앵커다. 별도 페이지를 만들지 않은 이유:
  Upstage 활용은 파이프라인 서사와 분리되면 "제품 목록"이 되고, 붙어 있어야 "왜 여기에 이걸 썼나"가 읽힌다.
- `/verify.html` 은 푸터에서만 노출(심사위원·발표용 뒷문). `/review.html` 은 어디에도 링크하지 않는다.

## 2. 페이지별 Spec

### `/` — 랜딩

| 항목 | 내용 |
|---|---|
| 답하는 질문 | "이게 뭐고 왜 나한테 필요한가" |
| 루브릭 기여 | Service Impact 10 · 차별성 5 (간접) |
| 유입 | 발표 슬라이드 QR · 도메인 직접 |
| 유출 | 주 CTA → `/app.html` · 보조 → `/contribute.html` · `/about.html` |
| 부르는 API | `GET /api/config` · `GET /api/data` |

블록 (구현 순서):
1. **히어로** — `오늘 [금액]으로 뭐 먹지?` 금액 타이핑 루프 + 실동작 데모(지도 + 대화 3시나리오)
2. **증거 카운터** — 식당 / 메뉴 / 학식당 (`/api/data` 실측치 — 라벨 규약은 §5)
3. **마키** — 수집된 메뉴명·가격 흐름
4. `band-warm` ① — "한입지도는 메뉴를 알아요" (범용 챗봇 대비 카드 2장)
5. `band-ink` — "사진 한 장이 가격 데이터가 되기까지" (파이프라인 3스텝)
6. `band-warm` ② — "빠진 가게가 있나요?" → `/contribute.html` CTA
7. **클로징** — "지금 예산을 넣어보세요" + 예산 칩 → `/app.html`

**랜딩 데모 지도의 제약**: 마커 클릭 카드는 카카오 `CustomOverlay` 라 `overflow:hidden` 인
`.demo-body` 밖으로 나가면 잘린다. 가장자리 마커에서 카드가 경계를 넘으면 지도를 밀어
넣는 보정이 필수다(`fitCardIntoView` — `/app.html` 과 같은 경로). 보정 없이 마커를 추가하지 마라.

### `/app.html` — 메뉴 찾기 (핵심 기능)

| 항목 | 내용 |
|---|---|
| 답하는 질문 | "예산을 넣으면 진짜로 정해주나" |
| 루브릭 기여 | Service Impact 10+5 · Solution Depth 15 · Upstage 20 |
| 유입 | 랜딩 CTA · nav |
| 유출 | 결과 없음 → `/contribute.html` · 근거 궁금 → `/about.html` |
| 부르는 API | `/api/config` · `/api/data` · `/api/chat` · `/api/recommend` · `/api/parse-query`(폴백) |

블록:
1. **대화 콘솔** — 지도 + 대화 사이드바 좌우 분할(데스크톱) / 세로 스택(≤860px).
   자연어 입력 → `POST /api/chat`(solar-pro2, 구조화 출력 1회 왕복) → 답변 + **제안 칩**.
2. **오늘의 추천** — `POST /api/recommend` Top3. 카드마다 메뉴·가격 조합 / 도보 n분 /
   추천 이유 / **근거 검증 배지** / 카카오맵 길찾기 링크.
3. **오늘의 학식** — 학식당 4곳을 항상 같은 순서로. 데이터 없는 곳은 `휴무`, 단 **적재가 오늘까지
   닿지 않았으면 `정보 없음`**(`cafeteriaFeed.fresh` — §5-6). 휴무와 크롤 정지는 다른 사실이다.
4. `band-warm` — "찾던 가게가 없었나요?" → `/contribute.html`

**전체 식당 그리드는 두지 않는다** (2026-07-21 사용자 확정). 24곳을 아래에 나열하면 "골라줬다"가
"목록을 줬다"로 읽혀 서비스의 주장이 약해진다.

필수 상태 처리 (전부 구현됨):
- 빈 결과 → "조건에 맞는 곳이 아직 없습니다" + 예산 상향 제안 + 제보 CTA (빈 화면 금지)
- `/api/chat` 실패 → `/api/parse-query` 정규식 폴백으로 결과 유지
- 지도 SDK 실패 → 리스트만으로 동작

**제안 칩**: 모델이 후보를 쓰고 **서버가 실데이터로 검증**한 것만 내보낸다(`api/_lib/chips.js`).
고정 예산 칩 줄은 두지 않는다 — 첫 질문 이후 맥락과 무관해지기 때문(2026-07-21 사용자 지적).

### `/contribute.html` — 사진 제보

| 항목 | 내용 |
|---|---|
| 답하는 질문 | "이 데이터는 어디서 오는가 — 내가 직접 넣어볼 수 있나" |
| 루브릭 기여 | **Data Architecture 20 + Solution Depth 15 + Upstage 20 구간의 유일한 인터랙티브 실체** |
| 유입 | nav · 랜딩 §6 · app 빈 결과 |
| 유출 | 제출 완료 → `/app.html` |
| 부르는 API | `POST /api/parse-menu`(Document Parse) · `POST /api/contribute`(시트 append) |

블록 (단일 세로 흐름):
1. **업로드** — 메뉴판 사진. 샘플 사진 제공(심사위원이 사진 없이도 시연 가능해야 한다)
2. **파싱 중** — 무슨 일이 일어나는지 문장으로 노출("Document Parse가 표 구조를 복원하는 중")
3. **결과 확인** — 추출된 `메뉴 / 가격` 표를 **읽기 전용**으로 보여준다
4. **제출** — `검수="대기"` 로 저장 → "검수 후 반영됩니다"

**제보자는 고치지 않는다.** 검수 권한은 운영진에게만 있다(§6-2). 다만 파싱 결과를 **보여주기는
한다** — Document Parse 결과가 심사위원 눈에 보이는 유일한 순간이라, 감추면 하드 요건 제품이
화면에서 사라진다. 편집만 없고 노출은 유지.

**정직성 규칙**: 파싱 정확도를 100%인 척하지 않는다. "AI 가 채우고 사람이 검수한다"를 실측
정확도와 함께 화면에 밝힌다(정본 수치 = `experiments/parse-poc/accuracy.md`).

필수 상태 처리: 파일 크기·형식 거부 / **파싱 0행·실패 시 수기 입력 폴백**(빈 행 1개를 주고 흐름을 잇는다).

## 3. 공통 요소

- **헤더**: 로고 좌 / nav 중앙(5개) / 로그인 우. 로그인은 "아직 없습니다" 다이얼로그
  — 인증은 anti-scope이고, 개인화가 로그인 없이 동작한다는 사실을 여기서 설명한다.
  **이 다이얼로그는 4개 페이지에 복제돼 있다** — 문구를 고치면 전부 고쳐야 한다(DR11 실측).
- **푸터**: 브랜드 / 서비스(홈·메뉴 찾기·사진 제보) / 이야기(데이터 파이프라인·Upstage AI·파이프라인 검증).
- `theme.css` 단일 스타일시트. light mode 고정(2026-07-20 확정). **신규 토큰 발명 금지.**

## 5. 화면 문구·라벨 규약 ★신설 (2026-07-22)

계기: 랜딩 카운터가 `학식 9` 를 보여주고 있었다. 학식당 개수도 오늘 메뉴 수도 아닌 **전 날짜 누적
행 수**였고, cron 이 매일 도는 동안 그 숫자는 계속 불어나고 있었다.

1. **숫자 라벨은 무엇을 세는지 라벨 자신이 밝힌다.** "학식"처럼 세는 대상이 여러 가지로 읽히는
   이름을 쓰지 않는다.
2. **세는 대상이 바뀌면 라벨도 같은 커밋에서 바꾼다.** 값만 바꾸고 라벨을 두면 그 자리에서 거짓이 된다.
3. **화면에 쓰는 수치는 정본 파일·API 필드에서 읽는다.** 기억이나 과거 문서의 숫자를 옮겨 적지 않는다
   (정확도 = `experiments/parse-poc/accuracy.md`, 학식당 = `/api/data` 의 `cafeteriaBoard`, 적재 신선도 = 같은 응답의 `cafeteriaFeed`).
4. **기능을 지우면 그 기능을 설명하던 문구를 같은 changeset 에서 지운다.** 화면 문구는 코드의
   주석이 아니라 사용자와의 약속이다.
5. **없는 것을 감추지 않는다.** 정확도·한계·미구현은 화면에 적는다 — 이 서비스의 주장이
   "데이터를 지어내지 않는다"이기 때문에, 문구가 그 주장을 어기면 기능이 무의미해진다.
6. **모르는 것을 아는 것처럼 말하지 않는다** (2026-07-22 추가). 빈 데이터에 사실을 뜻하는 말을
   붙이지 않는다 — 학식 행이 0일 때 "휴무"라고 말하면 **크롤러가 멈춰도 화면은 정상으로 보인다**
   (실제로 2026-07-21 에 적재 5일 지연이 "4곳 전부 휴무"로 보였다). 판정할 근거가 있을 때만
   단정하고(`/api/data` 의 `cafeteriaFeed.fresh`), 없으면 **"정보 없음"** 으로 말한다.

## 6. 제보 파이프라인 — 제출 → 파싱 → 대기 → 검수 → 승인 → 노출

### 6-1. 저장소 = 구글 시트 (Supabase 미채택)

시트가 이미 팀 데이터 정본이고, **검수 모델도 시트 안에 있다** — 즉 시트가 곧 어드민 패널이다.
별도 관리자 화면 없이 검수·승인이 되고 편집 권한·변경 이력·동시 편집이 따라온다.

| 탭 | 검수 관련 열 | 값 |
|---|---|---|
| `[식당]` | `상태` | `입력완료` |
| `[메뉴]` | `출처` · `검수` · `비고` | `사진파싱`/`제보` · `대기`/`확인` · `사이드`/`한끼` |

| Supabase RLS 로 하려던 것 | 시트 대체 |
|---|---|
| 익명 읽기 = approved 만 | `loadSheetData()` 가 `검수="확인"` 만 통과 |
| 익명 쓰기 = pending 강제 | 서버 함수만 append, `검수="대기"` 하드코딩 |
| 승인 = 소유자만 UPDATE | 시트 편집 권한 보유자만 셀 변경 |

### 6-2. 흐름 (검수 주체 = 운영진)

```
① /contribute.html      메뉴판 사진 + 식당명·위치 입력
② POST /api/parse-menu  Upstage Document Parse → {메뉴,가격}[] 후보
③ 브라우저              파싱 결과를 읽기 전용으로 확인 → "검수 후 반영됩니다"
④ POST /api/contribute  시트 append · 검수="대기" · 출처="제보"
⑤ /review.html          운영진이 사진 원본과 대조해 행별 확인/제외   ← 사람
⑥ loadSheetData()       검수="확인" 행만 서비스 노출
```

⑤ 는 데모데이 시연의 결정타다: *"지금 올린 게 대기로 들어왔고, 저희가 사진과 대조해 확인하면
이렇게 서비스에 뜹니다."*

### 6-3. 쓰기 수단 — Apps Script 웹앱 (실행 완료)

standalone 스크립트 + `SpreadsheetApp.openById()`. "실행 주체: 나"로 배포해 **본인 편집 권한**으로
쓴다 — 시트 소유자가 아니어도 성립. Vercel env 2개: `SHEET_WEBHOOK_URL`·`SHEET_WEBHOOK_TOKEN`.
append 대상은 별도 `[제보]` 탭이 아니라 `[식당]`·`[메뉴]` 정본이고 `검수="대기"` 로 격리한다
(승인 비용이 셀 1개로 끝나기 때문). `출처="제보"` 표식으로 롤백 가능.

### 6-4. 승인 게이트 — 실행 결과 (2026-07-20 종결)

게이트를 켰을 때 시트 `[메뉴]` 189행 중 `확인` 은 1행뿐이었다.

- **기각된 안: 기존 행 일괄 `확인` 처리.** 사람이 안 본 걸 확인으로 찍는 것은 검수 단계를 만들면서
  동시에 무력화하는 자기모순이고, 심사에서 가장 위험한 종류의 허위 주장이다.
- **따라서 실제로 사람이 검수했다.** 이건 회피할 작업이 아니라 파이프라인 20점의 증거 그 자체다.
  현재 노출 식당 24곳 · 검수 통과 메뉴 342행.

### 6-5. 실패·남용 처리

사진 크기·형식 거부(클라이언트+서버) · Parse 실패 시 수기 입력 폴백 · 0행 추출 시 빈 행 1개 ·
Apps Script 장애 시 파싱 결과는 화면에 남기고 "지금은 접수가 안 됩니다"(파싱 성과를 날리지 않는다).
레이트리밋은 데모 규모에 과설계로 판단해 제외 — 토큰 없는 요청 거부 + 1회 제출 행 수 상한만.

⚠ **검증 중에도 시트에 테스트 제보를 남기지 마라** — 행 삭제 수단이 없어 영구 잔존한다.
배선 확인은 `/api/config` 의 `ready` 플래그로.

## 7. 재방문 개인화 — 정본은 ADR-0004

> **개인화 = 지난 방문에 물었던 관심을 이어받아 답하는 것. 추천 순서는 개인화하지 않는다.**

이 절은 포인터다. 정의·경계·기억 계약은 `docs/adr/0004-personalization.md` 가 정본이고,
화면 설명은 `/about.html#s5` 다. **여기에 정의를 다시 쓰지 마라** — 두 곳에 쓰면 갈라진다
(이 문서의 구 §8 이 순서 가점 설계를 담고 있었고, 그것이 폐기된 뒤에도 남아 있었다).

화면 요구사항만 남긴다:
- 첫 방문(빈 프로필)에 개인화 없이 정상 동작해야 한다.
- 기억은 **보인다 · 그 자리에서 끌 수 있다 · 끄면 화면이 실제로 바뀐다.**
- 로그인 다이얼로그는 "이력은 이 브라우저에만 저장됩니다"를 설명한다.
