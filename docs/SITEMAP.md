# SITEMAP — 페이지 구조 설계도

> 작성 2026-07-20 · DR2 step-5 dogfood 중 페이지 구조 재설계
> 상위 정의: `docs/PRD.md` · 배점 기준: `CLAUDE.md` 루브릭 표 · 아키텍처: `docs/ARCHITECTURE.md`
> 이 문서는 **무엇이 어느 페이지에 들어가는가**의 SSOT다. 화면 구현 전 여기에 먼저 쓴다.

## 0. 설계 원칙 (이 구조를 고른 이유)

1. **루브릭이 페이지를 정한다.** 파이프라인 20 + 깊이 15 = 35점이 최대 배점인데, 지금 그 35점의 실체(사진→파싱)는 `public/test.html` 이라는 **스타일 없는 개발용 페이지 안에만** 있다. 심사위원이 보는 화면에 이게 없으면 35점 구간은 발표 슬라이드로만 주장된다.
2. **페이지 = 심사위원의 질문 1개.** 페이지마다 답하는 질문이 하나씩만 있어야 한다. 겹치면 합치고, 답이 없으면 페이지를 만들지 않는다.
3. **5페이지 상한.** DR2 plan §스캐폴딩("≤5 화면") 유지. 남은 빌드업은 7/24까지.

### `/contribute.html` — 사진 제보 ★신규 · 최우선

| 항목 | 내용 |
|---|---|
| 답하는 질문 | "이 데이터는 어디서 오는가 — 내가 직접 넣어볼 수 있나" |
| 루브릭 기여 | **Data Architecture 20 + Solution Depth 15 + Upstage 20 = 55점 구간의 유일한 인터랙티브 실체** |
| 유입 | nav · 랜딩 §5 · app 빈 결과 |
| 유출 | 제출 완료 → `/app.html` |

왜 신규인가: PRD 핵심기능 #2(★차별화 축)이자 "거지맵과 뭐가 다른가"의 답인데, 현재 `test.html §①` 의 개발용 폼으로만 존재한다. 심사위원이 **직접 사진을 올려 파싱되는 걸 보는 30초**가 발표 슬라이드 10장보다 강하다.

블록 (단일 세로 흐름 — 스텝 3개):
1. **업로드** — 메뉴판 사진 드래그&드롭 / 카메라 촬영. 샘플 사진 1장 제공(심사위원이 사진 없이도 시연 가능해야 함)
2. **파싱 중** — `POST /api/parse-menu` (Document Parse, `ocr:"force"`·`coordinates:true`). 대기 동안 무슨 일이 일어나는지 문장으로 노출("Document Parse가 표 구조를 복원하는 중")
3. **검수·수정** — 추출된 `메뉴 / 가격` 표를 편집 가능한 행으로 프리필. 정확도 76.4%(M2 실측) 전제이므로 **"AI가 채우고 사람이 고친다"를 UI가 명시**한다. 식당명·위치 입력
4. **제출** — `pending` 상태로 저장 → "검수 후 반영됩니다" 안내

정직성 규칙: 파싱 정확도를 100%인 척하지 않는다. 수정 UI 자체가 파이프라인 설계의 일부라는 서사(= `about` §데이터가 틀리면 과 연결).

필수 상태 처리: 파일 크기·형식 거부 / Parse 실패 시 수기 입력 폴백 / 0행 추출 시 빈 행 1개 제공.

## 7. 제보 파이프라인 — 제출 → 파싱 → 대기 → 검수 → 승인 → 노출

### 7-1. 저장소 결정: 구글 시트 (Supabase 대체)

시트를 이미 팀 데이터 정본으로 쓰고 있고, **검수 모델도 이미 시트 안에 있다.**

| 탭 | 검수 관련 컬럼 | 현재 값 |
|---|---|---|
| `[식당]` | `상태` (K열) | `입력완료` |
| `[메뉴]` | `출처` (E열) · `검수` (F열) · `비고` (G열) | `사진파싱` · `대기`/`확인` |

즉 **구글 시트가 곧 어드민 패널**이다. 별도 관리자 화면을 0줄 짜도 검수·승인이 되고, 편집 권한·변경 이력·동시 편집이 공짜로 따라온다. TRD §69-70 이 Supabase RLS 로 하려던 "익명은 pending 만 insert, 승인은 소유자만" 을 시트 권한이 그대로 대체한다:

| TRD 의 Supabase RLS 설계 | 시트 대체 |
|---|---|
| 익명 읽기 = approved 만 | `loadSheetData()` 가 `검수=확인` 만 통과 (**§7-4 — 지금 없음**) |
| 익명 쓰기 = `status='pending'` 강제 | 서버 함수만 append, `검수="대기"` 하드코딩 (클라이언트는 시트에 직접 못 씀) |
| 승인 = 소유자만 UPDATE | 시트 편집 권한 보유자만 셀 변경 |

### 7-2. 흐름 (검수 주체 = 운영진 — 2026-07-20 사용자 확정)

```
① 브라우저 /contribute.html      메뉴판 사진 + 식당명·위치 입력
② POST /api/parse-menu (기존)    Upstage Document Parse → {메뉴,가격}[] 후보
③ 브라우저                       파싱 결과를 읽기 전용으로 확인 → "검수 후 반영됩니다"
④ POST /api/contribute (신규)    시트 append · 검수="대기" · 출처="제보"
⑤ 운영진 검수                    사진 원본과 대조하며 행별 확인/제외  ← 사람(§7-4)
⑥ loadSheetData() 게이트         검수="확인" 행만 서비스 노출
```

**제보자는 고치지 않는다.** 검수 권한은 운영진에게만 있다. 다만 ③ 에서 파싱 결과를 **보여주기는 한다** — Document Parse 결과가 심사위원 눈에 보이는 유일한 순간이라, 감추면 하드 요건 제품이 화면에서 사라진다. 편집만 없고 노출은 유지.

사진만으로는 식당명·위치를 알 수 없으므로 ① 에서 함께 받는다.

⑤ 는 데모데이 시연의 결정타다: *"지금 올린 게 대기로 들어왔고, 저희가 사진과 대조해 확인하면 이렇게 서비스에 뜹니다."*

### 7-3. 쓰기 수단 — Apps Script 웹앱 (권장)

- **standalone 스크립트**(사용자 드라이브 소유) + `SpreadsheetApp.openById()` 로 팀 시트에 접근. 팀 시트에 스크립트를 심지 않아 소유권 다툼·팀 노출이 없다.
- 웹앱을 "실행 주체: 나"로 배포하면 **본인 편집 권한**으로 시트에 쓴다 → **시트 소유자가 아니어도 된다**(편집 권한만 있으면 성립. 2026-07-20 확인: 사용자는 편집 가능, 소유는 팀원).
- Vercel env 2개: `SHEET_WEBHOOK_URL` · `SHEET_WEBHOOK_TOKEN`. GCP 콘솔·서비스 계정 JSON 불요.
- 기존 로컬 쓰기 경로(`scripts/*` 의 `gws` CLI 로컬 OAuth)는 서버리스에서 못 쓴다 — 그래서 별도 경로가 필요하다.

**append 대상**: 별도 `[제보]` 탭이 아니라 `[식당]`·`[메뉴]` 정본에 바로 넣고 `검수="대기"` 로 격리한다. 승인 비용이 셀 1개로 끝나기 때문(별도 탭이면 승인 = 수작업 복사). 정본 오염 위험은 `출처="제보"` 표식으로 롤백 가능하게 만든다.

### 7-4. 선결 과제 — 승인 게이트가 없고, 189행이 미검수다 ⚠

`api/_lib/sheet-data.js:62` 는 `.filter((m) => m.name && m.price)` 뿐이고 `검수` 값을 필터링하지 않는다. 노이즈 행을 손으로 삭제해서 버텨 온 상태(2026-07-20 정리 주석). 제보 페이지를 열기 전에 `검수="확인"` 통과 조건이 반드시 있어야 한다.

**시트 실측 (2026-07-20):** `[메뉴]` 189행 중 `대기` 189 · `확인` 1. 게이트를 켜면 서비스 데이터가 1행이 된다.

- **기각된 안: 기존 행 일괄 `확인` 처리.** 사람이 안 본 걸 확인으로 찍는 것은 검수 단계를 만들면서 동시에 무력화하는 자기모순이고, 심사에서 가장 위험한 종류의 허위 주장이다. (2026-07-20 사용자 지적으로 철회)
- 따라서 **189행은 실제로 사람이 검수한다.** 이건 회피할 작업이 아니라 루브릭 파이프라인 20점의 증거 그 자체다. 어디서 검수하느냐는 §7-5.

### 7-5. 검수를 어디서 하나 — (b) `/review.html` 확정 (2026-07-20 사용자)

| 안 | 방식 | 비용 | 시연 |
|---|---|---|---|
| (a) 시트에서 손 검수 | 구글 시트 `검수` 열을 직접 편집 | 개발 0 · 노동 189행 | 시연 때 시트 화면 공유 |
| (b) `/review.html` 검수 화면 신설 | 사진 원본과 파싱 결과를 나란히 놓고 행별 확인/제외 | 개발 ~반나절 · 노동 대폭 단축 | 파이프라인 전 구간이 제품 안에서 보임 |

(b) 권장 근거: 검수 단계가 "구글 시트를 손으로 고칩니다"로 남으면 심사에서 설계가 아니라 임시방편으로 읽힌다. 검수 UI 가 있으면 수집→AI 추출→사람 검수→서비스가 전부 제품 안에서 관측된다(파이프라인 20 + 깊이 15 직결). 189행 검수를 어차피 해야 하므로 그 노동이 도구 개발로 상쇄된다.

(b) 채택 시 제약: 페이지 6개가 되어 §0 원칙 3(5페이지 상한)을 깬다 — 단 `/review.html` 은 운영진 전용이라 공개 nav 에 넣지 않으므로 사용자 여정은 5페이지 그대로다. 인증이 없으므로 URL 비공개 + 쓰기 요청에 토큰 필요로 최소 방어.

### 7-5. 실패·남용 처리

- 사진 크기·형식 거부(클라이언트 + 서버 양쪽) · Parse 실패 시 수기 입력 폴백 · 0행 추출 시 빈 행 1개
- Apps Script 장애 시: 파싱 결과는 화면에 남기고 "지금은 접수가 안 됩니다" 안내(파싱 성과를 날리지 않는다)
- 남용 방지는 데모 규모에 맞춰 최소로 — 토큰 없는 요청 거부 + 1회 제출 행 수 상한. 레이트리밋은 과설계로 판단해 제외.

## 8. 재방문 개인화 — 로그인 없이

### 8-1. 목적 분해

요구는 "재방문자에게 기존 사용 데이터 기반 추천". 여기서 실제로 필요한 건 **"이 사람이 누구인가"(인증)가 아니라 "이전에 뭘 봤는가"(이력)** 다. 후자는 로그인 없이 된다.

- 채택: **localStorage 익명 프로필**. 서버 0 · 계정 0 · 쿼터 0. TRD §44 가 이미 `localStorage`(최근 검색어·기본 조건)를 클라이언트 상태 저장소로 지정하고 있어 결이 맞는다.
- 인증 인프라는 루브릭 배점 **0점**이고 DR2 plan anti-scope 다. 남은 일정(빌드업 7/24)을 여기 쓰지 않는다.

### 8-2. 저장 항목 (`localStorage: hanip.profile`)

| 키 | 내용 | 쓰임 |
|---|---|---|
| `budgets` | 최근 예산 3건 | 재방문 시 입력칸 프리필 + "지난번엔 8천원이었어요" |
| `picked` | 클릭·선택한 식당 id | rerank 가점 |
| `dismissed` | "다시 안 볼래" 표시한 id | 추천에서 강등 |
| `tags` | 파싱된 태그 누적 빈도(혼밥·매운 것 등) | 태그 선호 rerank |
| `visits` | 방문 횟수·최종 방문일 | 첫 방문/재방문 UI 분기 |

### 8-3. 추천에 어떻게 쓰나

`POST /api/recommend` 요청 바디에 `profile` 을 함께 보내 서버가 rerank 한다. **서버는 사용자 데이터를 저장하지 않는다(무상태)** — 이건 구현 편의일 뿐 아니라 "개인화하면서 개인정보를 서버에 안 쌓는다"는 발표 서사가 된다.

- 첫 방문(빈 프로필): 개인화 없이 정상 동작해야 한다. 프로필은 **가점 신호일 뿐 필터가 아니다** — 프로필 때문에 결과가 0건이 되는 경로를 만들지 않는다.
- 재방문 UI: 히어로/검색 상단에 "지난번 조건으로 다시 볼까요?" 한 줄 + 개인화가 적용됐음을 카드에 표시("전에 고른 곳과 비슷해요") — 개인화가 **눈에 보여야** 점수가 된다.
- 프로필 초기화 버튼 필수(개인화 끄기).

### 8-4. 정직한 한계 (about 에 명시)

- 기기·브라우저가 바뀌면 이력이 사라진다. 시크릿 모드에서는 동작하지 않는다.
- 진짜 계정이 필요해지는 시점: 기기 간 동기화 · 제보 귀속 · 사용자 간 협업 필터링. 전부 데모데이 이후 과제로 남긴다(ADR 로 보존).
- 헤더의 "로그인" 버튼: 개인화가 로그인 없이 동작하므로 **"로그인은 아직 없습니다" 다이얼로그 유지**하되 문안을 "이력은 이 브라우저에만 저장됩니다"로 교체한다.
